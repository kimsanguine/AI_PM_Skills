# 도메인 컨텍스트 — PM Framework (암묵지 체계화)

## 1) Domain Scope

pm-framework는 **PM의 암묵적 판단 기준을 명시적 TK-NNN 구조로 변환하여 에이전트와 팀이 학습하고 재현 가능하게 하는 추출 및 구조화 시스템**입니다.

| 포함 | 제외 |
|-----|------|
| PM 경험에서 패턴 포착 | 이미 표준화된 Best Practice |
| 암묵지 → TK 구조화 | 일반 LLM 지식 (누구나 아는 것) |
| TK 유형 분류 (5가지) | 기술 기술서 작성 |
| 활성화/비활성화 조건 정의 | 도구 사용법 학습 |
| TK 간 연관성 매핑 | 프로젝트 관리 프로세스 |
| TK 검증 및 정제 | Instruction 직접 작성 |

**핵심**: pm-framework는 "경험을 TK로 번역하는 변환 장치"입니다.

---

## 2) Primary Users

1. **PM (Product Manager)**
   - 자신의 경험에서 패턴을 찾아 명시적 TK로 정리
   - "나는 왜 이런 판단을 했나?"의 근거를 기록
   - 축적된 TK로 팀의 의사결정 품질 향상

2. **에이전트 (Agent)**
   - 새로 추출된 TK를 학습하여 다음 판단에 반영
   - Instruction에 반영된 TK를 따라 의사결정 자동화
   - TK 기반 지식 그래프를 통해 맥락 인식 향상

3. **팀 (Team)**
   - PM의 사고방식을 TK로 공유받아 빠르게 학습
   - 새로운 팀원이 "왜 우리는 이렇게 판단하나"를 이해
   - 의사결정 문화 정규화

4. **조직 (Organization)**
   - PM 개인의 암묵지가 조직 자산으로 전환
   - TK 축적으로 "Domain Knowledge 해자" 형성
   - 리더 교체 시에도 경험이 유지됨

## 3) Required Inputs

### 필수 입력 1: PM의 경험/상황 설명
```
"오늘 회의에서 MVP를 배포할 때,
 기능을 하나만 넣고 시작했더니 피드백이 빨리 들어왔어.
 전에는 모든 기능을 다 넣으려고 하다가 3배 오래 걸렸는데."
```

또는:
```
"우리 팀은 처음엔 CEO가 요청한 것을 우선으로 했는데,
 그러면서 팀의 진짜 중요한 일들이 밀렸어.
 이제는 CEO 요청도 '비즈니스 임팩트'로만 판단해."
```

### 선택 입력: 명확한 판단 기준 제시
```
"그래서 지금은 항상 '이 작업이 사용자에게 어떤 가치인가'를 먼저 묻고,
 그 다음에 실행할지 말지 결정해."
```

### 선택 입력: 실패 사례 또는 대조
```
"전에 반대로 했을 때와 지금의 결과가 뭐가 달라?"
```

## 5) Guardrails

### Guardrail 1: 개인 선호도 vs 검증된 패턴 구분
**규칙**: "내가 좋아한다"와 "여러 번 검증됐다"는 다름
```
❌ "나는 항상 A 방식을 선호해" → TK가 아님 (개인 취향)
✅ "우리가 A 방식으로 3번 했을 때 B 방식보다 2배 빨랐어" → TK (검증됨)
```

### Guardrail 2: 일반적인 지식은 TK가 아님
**규칙**: 누구나 알만한 일반적 조언은 TK로 추출하지 말 것
```
❌ "우선순위가 높으면 높은 것부터 한다" → 상식
✅ "우선순위를 CEO가 요청한 것 vs 임팩트 기반으로 결정했을 때,
    임팩트 기반이 팀 생산성 30% 향상" → 경험 기반 TK
```

### Guardrail 3: 활성화 조건이 없으면 검색에서 안 나옴
**규칙**: "언제 써야 하나"가 명확하지 않으면 Contextual Retrieval 패턴이 작동 안 함
```
❌ "TK가 있는데 왜 검색이 안 되지?"
   원인: 활성화 조건이 애매해서 CR 엔진이 일치 못 함

✅ "신규 에이전트 설계 초안을 잡을 때" 명확히 → 검색 정확도 95%+
```

### Guardrail 4: 극도로 특수한 상황의 TK는 일반화 필요
**규칙**: "우리 회사 특수 상황"은 TK가 아니라 프로세스
```
예: "매주 목요일 3시에 경영진 회의가 있어서
    수요일까지 보고서를 제출해야 한다"
→ 이것은 TK가 아니라 조직 프로세스

TK화: "마감 48시간 전 최종 검토를 해야 한다" (일반화)
```

### Guardrail 5: TK의 유효 기한 명시
**규칙**: 시간이 지나면서 시장/기술 변화로 TK가 부실화될 수 있음
```
기록:
- TK-123: [내용]
- 작성 날짜: 2026-03-01
- 마지막 검증: 2026-03-07
- 재검증 필요: 2026-06-01 (분기별)
```

### 8-1) Polanyi-Nonaka 암묵지 이론 적용: SECI 모델의 4단계를 TK 추출 프로세스에 매핑

**이론 배경**

Michael Polanyi는 "우리는 말할 수 있는 것보다 더 많은 것을 안다"("We can know more than we can tell")는 명제로 암묵지의 중요성을 정립했습니다. 이를 조직 차원에서 체계화한 것이 Ikujiro Nonaka의 SECI 모델(지식 창조 나선)입니다.

**SECI 4단계와 TK 추출의 매핑**

| 단계 | SECI 모델 | PM TK 추출 프로세스 | 예시 |
|-----|---------|-------------------|------|
| 1 | **Socialization (암묵지→암묘지)** | **대화 중 추출**: PM과 팀의 상호작용에서 패턴 발굴 | "이건 어떻게 판단하셨어요?"라는 질문을 통해 명시되지 않은 판단 기준 포착 |
| 2 | **Externalization (암묘지→명시지)** | **TK 구조화**: 포착된 패턴을 TK-NNN 형식으로 문서화 | 패턴/활성화/비활성화/Why를 명시적으로 작성하여 지식을 외부화 |
| 3 | **Combination (명시지→명시지)** | **TK 연관성 매핑**: 여러 TK를 조합하여 더 큰 의사결정 프레임워크 형성 | TK-001(긴급 판단) + TK-007(우선순위)을 연결하여 "스코프 관리" 가이드라인 구성 |
| 4 | **Internalization (명시지→암묘지)** | **에이전트/팀의 학습**: TK 기반 Instruction을 실행하며 새로운 암묘지로 내재화 | 에이전트가 TK-012(prototype-first)를 따르면서 "왜 프로토타입을 먼저 하는지"를 체화적으로 이해 |

**실전 적용 지침**

- **S 단계 확장**: 팀 회의, 레트로스펙티브, 1:1 미팅에서 "판단 근거"를 의도적으로 질문하여 암묘지 발굴 기회 확대
- **E 단계 품질**: 단순 정보 나열이 아니라 "왜 이 판단이 중요한가"를 포함하여 외부화. 미완성 TK는 "가설(H)" 태그로 마킹
- **C 단계 연결**: 월 1회 TK 리뷰 회의에서 새로운 TK들을 기존 TK와 연관시켜 지식 그래프 강화
- **I 단계 검증**: 에이전트가 TK를 적용한 결과(성공/실패)를 기록하여 재검증 데이터로 활용

### 8-2) Type 2 (Failure Pattern) vs Type 4 (Anti-Pattern) 명확한 구분 기준

**문제 배경**

B-검증 피드백에서 Type 2와 Type 4의 경계가 모호하다는 지적이 있었습니다. 실무에서는 두 가지가 혼용되기 쉬운데, 이론적 근거로 명확히 구분할 필요가 있습니다.

**정의 비교**

| 구분점 | Type 2 (Failure Pattern) | Type 4 (Anti-Pattern) |
|-------|----------------------|---------------------|
| **명제 구조** | "이렇게 하면 망한다" | "이것만큼은 하지 마라" |
| **근거** | 과거 경험에서 추출한 인과관계 | 원칙이나 규범, 보편적 금지 |
| **시간축** | 있음 (시점, 상황에 따라 변할 가능성) | 없음 (항상 적용되는 절대 원칙) |
| **발생 조건** | 특정 상황, 맥락에 한정 | 언제 어디서든 해당 (예외 거의 없음) |
| **검증 방식** | 데이터/경험으로 인과관계 입증 | 논리/원칙으로 정당화 |
| **활성화 조건** | 구체적 상황 명시 | 보편적 (조건 거의 필요 없음) |
| **비활성화 조건** | 예외 상황 존재 | 예외가 거의 없음 |

**리트머스 테스트 (Litmus Test)**

이 판단이 **"특정 상황의 인과적 결과인가"** vs **"보편적 원칙인가"**를 구분하려면:

```
질문 1: "이 판단이 다시는 틀릴 일이 없는가?"
  → Yes: Type 4 (Anti-Pattern) — 예: "도구 탓하기 금지" (항상 나쁨)
  → No: Type 2 (Failure Pattern) — 예: "스펙 완성 후 개발" (상황에 따라 필요할 수도)

질문 2: "이것을 어기는 경우가 정당화될 수 있는가?"
  → No: Type 4 (Anti-Pattern) — 규제/컴플라이언스 같은 절대 규칙
  → Yes: Type 2 (Failure Pattern) — 특수 상황 (규제 요구)에서는 스펙 완성 후 개발도 필요

질문 3: "미래의 기술/시장 변화로 이 판단이 역전될 가능성이 있는가?"
  → Yes: Type 2 (Failure Pattern) — 예: "지금은 프로토타입부터 하지만, AI 모델이 완벽해지면 달라질 수도"
  → No: Type 4 (Anti-Pattern) — 예: "도구 탓하기는 언제나 나쁠 것"
```

**실전 적용 예시 (참고 사례)**

```
상황: "초기에 모든 기능을 다 정의하고 개발을 시작했더니 3배 오래 걸렸다"

분석:
1. "이것이 특정 상황의 결과인가?" → Yes, 신규 제품 개발 초기 단계에서만 해당
2. "예외가 있는가?" → Yes, 규제 때문에 사전 스펙이 필수인 경우
3. 결론: **Type 2 (Failure Pattern)**

TK-XXX: 스펙 완성 후 개발의 함정
📌 패턴: 새 기능 개발 시, 모든 스펙을 완성하고 개발을 시작하면 마감 시점에 이미 가정이 틀렸을 확률이 높다.
🟢 활성화 조건: 새로운 기능 개발 시작 전, 불확실성이 높은 요구사항
🔴 비활성화 조건: 규제/컴플라이언스로 사전 문서화 의무인 경우
```

```
상황: "AI 에이전트가 틀린 결과를 냈을 때, 팀원들이 '도구가 못해서'라고 결론 내리는데, 대부분은 프롬프트 문제"

분석:
1. "이것이 특정 상황의 결과인가?" → 상황에 관계없이 항상 설계 문제
2. "예외가 있는가?" → 매우 드묾 (모델의 알려진 한계 제외)
3. 시간이 지나도 바뀔까? → 10년 뒤에도 "도구 탓"은 나쁠 것
4. 결론: **Type 4 (Anti-Pattern)**

TK-YYY: 도구 탓하기 금지
📌 패턴: AI 에이전트 품질 문제 발생 시, "모델이 못해서"라고 결론 내리기 전에, 프롬프트/컨텍스트/데이터 문제를 먼저 확인한다.
🟢 활성화 조건: AI 에이전트 품질 문제 발생 시 (항상)
🔴 비활성화 조건: 모델 자체의 알려진 한계 (예: 한국어 수학 계산)
💡 Why: 대부분의 "AI 품질 문제"는 실제로는 설계/프롬프트 문제. 도구 탓을 하면 개선 기회를 놓침.
```

### 8-3) Contextual Retrieval (CR) 패턴 구현 가이드

**개념**

Contextual Retrieval은 단순 키워드 매칭이 아니라, 각 TK의 **활성화 조건**을 기반으로 "지금 이 상황에서 필요한 TK"만 동적으로 로드하는 패턴입니다. 임베딩 기반과 규칙 기반 방식을 병행하여 정확도를 높입니다.

**CR 필드 설계**

모든 TK에 다음 메타데이터를 추가:

```
TK-NNN: [제목]
📌 패턴: ...
🟢 활성화 조건: [구체적 상황, 가능하면 임베딩 가능한 키워드]
   - 신호 1: 신규 에이전트 설계 → 임베딩 키: "agent design, MVP, new system"
   - 신호 2: 개발 일정 밀림 → 임베딩 키: "deadline, schedule slip"
🔴 비활성화 조건: [예외 상황]
💡 Why: ...
🔗 연관 TK: ...
📊 CR Score (예시값): 활성화 신호 매칭도 (0.0~1.0)
```

**임베딩 기반 vs 규칙 기반 비교**

| 방식 | 임베딩 기반 (Semantic) | 규칙 기반 (Lexical) |
|-----|-------------------|------------------|
| **작동 원리** | 쿼리와 활성화 조건의 의미적 유사도 계산 | 정확한 키워드 매칭 또는 불린 로직 |
| **장점** | 의도를 파악하여 유사 상황도 발굴 가능 | 높은 정확도, false positive 적음 |
| **단점** | 컨텍스트 부족 시 오매칭 가능 | 표현이 조금 달라도 미스 |
| **사용 예시** | "스코프 늘어나고 일정이 밀렸어" → TK-001 발견 | 키워드 "긴급 요청"이 정확히 포함된 TK만 검색 |
| **추천 조합** | **Hybrid 방식**: 규칙으로 후보 필터링 → 임베딩으로 순서 정렬 |

**구현 체크리스트 (참고 사례)**

- [ ] 모든 TK에 활성화 조건 + 임베딩 키워드 포함
- [ ] CR 엔진이 쿼리 받으면: (1) 키워드 필터 → (2) 임베딩 정렬 → (3) Top-3 TK 반환
- [ ] 활성화 조건과 실제 상황의 매칭도를 0.0~1.0 점수로 표시
- [ ] 월 1회 "검색 미스" 분석: "TK가 있었는데 안 나왔다"는 사례 수집 → 활성화 조건 개선
- [ ] 재검증 주기: 분기별 (기술/시장 변화로 활성화 조건 진부화 확인)

### 8-4) TK 품질 평가 매트릭스 (예시값)

**평가 틀**

모든 TK는 4개 차원에서 1~5점으로 평가됩니다.

```
TK-NNN: [제목]

품질 매트릭스 (1=낮음, 5=높음):
┌─────────────────────────────────────────────┐
│ 1) 일반성 (Generalizability): 3점           │
│    → 이 TK가 얼마나 넓은 상황에 적용 가능한가?  │
│    설명: 에이전트 설계 초기에만 적용         │
│            (좁은 맥락, 범위 제한)          │
│                                          │
│ 2) 검증도 (Validation): 5점                 │
│    → 실제 데이터/경험으로 얼마나 입증되었는가?  │
│    설명: 10개 에이전트 프로젝트에서          │
│            일관되게 확인됨                 │
│                                          │
│ 3) 실행가능성 (Actionability): 4점         │
│    → 에이전트/팀이 따를 수 있을 정도로       │
│        구체적인가?                         │
│    설명: "MVP부터 시작하세요"는 명확하지만  │
│            "임팩트 기반 판단"은 애매함      │
│                                          │
│ 4) 연결성 (Connectivity): 2점              │
│    → 다른 TK와 얼마나 잘 연결되어 있는가?    │
│    설명: 연관 TK가 2개뿐, 더 연결 필요      │
└─────────────────────────────────────────────┘

종합 평가: 14점 / 20점 (예시값)
평가자: AI PM Skills Contributors, 2026-03-08
다음 재평가: 2026-06-08 (분기별)
```

**등급 기준 (참고 사례)**

| 등급 | 합계 점수 | 의미 | 조치 |
|------|---------|------|------|
| A | 18-20 | 매우 높은 품질, 에이전트 적극 적용 | 팀 전체에 공유 |
| B | 15-17 | 좋은 품질, 에이전트 적용 가능 | 활성화 조건 재정의 검토 |
| C | 12-14 | 중간 품질, 보완 필요 | 연관 TK 추가, 검증도 강화 |
| D | 10-11 | 낮은 품질, 재구조화 권장 | 분해하여 더 구체화하거나 기존 TK와 병합 |
| F | <10 | 매우 낮음, 삭제 또는 대폐기 검토 | "Deprecated" 마킹, 아카이빙 |

### 8-5) TK 라이프사이클 관리: 생성→검증→활성→폐기 단계별 기준

**단계별 프로세스**

```
[Stage 1: 생성 (Creation)]
├─ 언제: PM의 경험/패턴 추출
├─ 상태: "Draft" 또는 "Hypothesis (H)"
├─ 저장: PM-ENGINE-MEMORY.md 또는 임시 노트
├─ CR 활성: 비활성 (검증 전이므로)
├─ 품질 점수: 미평가
└─ 기간: 1일 ~ 1주

[Stage 2: 검증 (Validation)]
├─ 언제: 추출 후 1주 이내
├─ 방법:
│  ├─ 데이터 검증: 실제 프로젝트 데이터로 인과관계 확인
│  ├─ 동료 리뷰: 팀의 합의 확보
│  ├─ Quality Gate 5개 통과 확인
│  └─ 중복 TK 검사
├─ 상태: "Validated"
├─ 품질 점수: 평가 완료 (1~20점)
└─ 기간: 3일 ~ 2주

[Stage 3: 활성 (Active)]
├─ 언제: 검증 완료 후
├─ 조건: 품질 점수 C 이상 (12점 이상)
├─ CR 활성: 활성화 조건 기반으로 동적 로드
├─ 에이전트: Instruction에 반영
├─ 추적: 월 1회 적용 사례 로깅
└─ 기간: 무기한 (또는 재검증까지)

[Stage 4: 재검증 (Re-validation)]
├─ 언제: 분기별 (3개월마다)
├─ 확인 사항:
│  ├─ 시장/기술 변화로 인한 부실화 여부
│  ├─ 최근 3개월 적용 사례 검토
│  ├─ 활성화 조건의 여전한 타당성
│  └─ 새로운 연관 TK 발굴
├─ 결과:
│  ├─ 검증 통과 → 다시 Stage 3로
│  ├─ 부실화 감지 → Stage 5로
│  └─ 예외 조건 추가 → 조건 업데이트 후 Stage 3로
└─ 기간: 2주

[Stage 5: 폐기/아카이빙 (Deprecation)]
├─ 언제: TK가 더 이상 유효하지 않을 때
├─ 예시:
│  ├─ 시장 조건 변화 (AI 모델 정확도 급상승)
│  ├─ 조직 변화 (프로세스 완전 변경)
│  ├─ 기술 진화 (도구 자체가 불필요해짐)
│  └─ 비용 대비 효과 부족
├─ 조치:
│  ├─ "Deprecated" 표시 (삭제 아님)
│  ├─ "폐기 사유" 문서화
│  ├─ 아카이브 폴더로 이동
│  └─ 참고용으로 남김
└─ 예시: "TK-051: 구 모델 API 호출 방식" → Deprecated 2026-03
```

**라이프사이클 추적 메타데이터**

```
TK-NNN: [제목]
📌 패턴: ...
🟢 활성화 조건: ...
...

📅 라이프사이클:
- 생성일: 2026-03-01
- 검증일: 2026-03-08
- 현재 상태: Active
- 품질 점수: 16점 (B등급)
- 마지막 재검증: 2026-03-08
- 다음 재검증: 2026-06-08
- 적용 횟수: 7회 (최근 3개월)
- 개선율: +25% (의사결정 품질 향상도)
```

### 8-6) 실전 적용 사례 3선 (참고 사례)

#### 사례 1: 초기 스코프 결정 (에이전트 프로젝트)

**상황**

신규 AI 에이전트 프로젝트 시작. PM이 "모든 기능을 다 정의하고 개발 시작"하려고 했는데, 과거 유사 프로젝트에서 3배 오래 걸린 경험 있음.

**추출된 TK (참고 사례)**

```
TK-045: Minimum Viable Agent (MVA)

📌 패턴:
에이전트 설계 초안을 잡을 때, 이 에이전트가 해야 할
단 하나의 핵심 기능만 정의하고, 그 핵심만으로 작동하는
최소 버전을 먼저 배포한 후, 실사용 데이터로 확장 방향을 결정한다.

🟢 활성화 조건:
- 신규 에이전트 설계 초안 수립
- MVP 스코프 정의
- 배포 계획 수립
(키워드: agent design, new feature, MVP, scope)

🔴 비활성화 조건:
- 규제/컴플라이언스로 사전 기능이 모두 필수인 경우
- 기존 에이전트의 점진적 기능 추가 (이미 Core MVP는 배포됨)

💡 Why:
초기에 기능이 많을수록:
→ 복잡도 급증 (feature interaction 증가)
→ 실패 원인 파악 어려움
→ 피드백 루프 길어짐

MVP로 시작하면:
→ 피드백이 명확 (한 가지 기능에 집중)
→ 확장도 빠름 (이미 검증된 기능에 추가)
→ 비용 효율 높음 (불필요한 기능 개발 방지)

🔗 연관 TK:
- TK-015: Prototype-first Development
- TK-010: Growth Over Vanity Metrics (리텐션 우선)
- TK-062: Feature Complexity Kills Adoption

📊 CR Score: 0.92 (높은 활성화 신호)
📅 라이프사이클: Active | 품질 18점 (A등급) | 적용 5회
```

**실제 효과 (참고 수치, 예시값)**

- 첫 배포까지 기간: 12주 → 4주 (3배 단축)
- 초기 피드백 수집: 2주 후 → 배포 1주일 후
- 개발 재작업: 45% → 15% (추정)

#### 사례 2: 긴급 요청 판단 (우선순위 관리)

**상황**

CEO가 "긴급"이라고 요청한 작업과 실제 비즈니스 임팩트가 다르다. 신호 구분 기준이 필요.

**추출된 TK (참고 사례)**

```
TK-001: 긴급 요청 우선순위 판단

📌 패턴:
긴급해 보이는 요청이 와도, 먼저 "이것이 Why-Urgent인가"를 확인한다.
실제 마감(deadline)이 없는 '인식된 긴급'은 우선순위에서 제외.
우선순위 판단: 발신자 신분이 아니라 "비즈니스 임팩트" 기준.

🟢 활성화 조건:
- 여러 이해관계자에게 동시 요청 수신
- CEO, 임원진에게서 "긴급" 호출 받음
- 팀이 현재 태스크와 요청 사이에서 갈등
(키워드: urgent, priority, stakeholder, CEO request)

🔴 비활성화 조건:
- 실제 비즈니스 SLA 있는 장애 (예: 서버 다운)
- 고객 계약상 마감 명확함

💡 Why:
긴급 요청 80%는 발신자의 불안에서 비롯됨.
발신자 신분으로 우선순위 정하면:
→ 긴급마다 방향 바뀜
→ 핵심 작업 중단
→ 팀 모멘텀 손실

비즈니스 임팩트로 판단하면:
→ 핵심 작업 보호
→ 팀 생산성 30% 향상 (내부 데이터)
→ CEO도 "지금 당신의 요청이 #1이 아니네"를 이해 가능

🔗 연관 TK:
- TK-003: Stakeholder Energy Management
- TK-007: Impact-Based Prioritization Framework
- TK-089: Context Window for Decision-Making (PM의 시간/에너지 한계)

📊 CR Score: 0.88
📅 라이프사이클: Active | 품질 17점 (B등급) | 적용 12회
```

**실제 효과 (참고 사례)**

- 긴급 요청으로 인한 컨텍스트 스위칭: 월 15회 → 3회
- 팀의 "본래 계획" 완료율: 65% → 92%
- 스트레스 레벨 (자가 평가): 7/10 → 4/10

#### 사례 3: AI 모델 품질 문제 진단 (도구 탓 금지)

**상황**

에이전트가 틀린 결과를 낼 때마다 "Claude가 못해서"라고 결론. 실제로는 프롬프트, 컨텍스트, 데이터 문제.

**추출된 TK (참고 사례)**

```
TK-019: 도구 탓하기 금지 (Tool-Blaming Antipattern)

📌 패턴:
AI 에이전트가 틀린 결과를 냈을 때,
"모델이 못해서"라고 결론 내리기 전에
프롬프트 → 컨텍스트 → 데이터를 먼저 확인한다.

실패 원인 체크리스트:
```
