# Troubleshooting — pm-framework 스킬

## 1) 추출한 TK가 너무 일반적: 누구나 아는 것

**증상:**
- 추출한 TK: "우리는 우선순위가 높은 것부터 한다"
- "이건 상식이 아닌가?"
- TK를 에이전트에 줘도 별 효과 없음

**확인:**
- 이것이 정말 "당신만의" 암묵지인가?
- 누군가 다르게 판단할 여지가 있는가?
- "우선순위"를 정하는 기준이 구체적인가?

**조치:**
1. **특수화하기**: "우선순위가 높은 것" → "비즈니스 임팩트 + 기술 실행 용이성 + 고객 대기 시간" 조합으로 점수화
2. **구체적 상황으로**: "항상 그렇다" → "이런 조건에서는 이렇게"로 맥락화
3. **반례 찾기**: "이 경우는 예외" → 활성화/비활성화 조건 명시
4. **데이터 기반**: "우리 팀은 지난 1년간 이 기준으로 결정해서 성공률 85% 달성"
5. **폐기 고려**: 정말 일반적이면 TK가 아니라 "Best Practice" 레퍼런스로 처리

## 2) TK 중복: 이미 있는 것과 같은 내용

**증상:**
- 새 TK-090: "긴급 요청 우선순위 판단"
- 기존 TK-003: "긴급 요청 우선순위 판단"
- "어? 이건 중복인데?"

**확인:**
- 두 TK의 활성화 조건: 정확히 같은가?
- 미묘한 차이가 있는가? (시간 범위, 대상 고객 등)
- 유사도: 0.85 이상이면 중복 기준

**조치:**
1. **중복 확인**: 유사도 0.85+ → 명확한 중복
2. **병합**: 두 TK를 하나로 통합
   - 더 정확한 버전으로 선택
   - 두 버전의 장점 결합
3. **기존 TK 처리**: 통합되지 않은 기존 TK는 "Deprecated" 플래그 추가 (삭제 금지)
4. **버전 정보**: "v2 (2026-03-08 병합됨: TK-090과 통합)"라고 명시
5. **다시 검토**: 새로운 TK 작성 전 기존 0.7 이상 유사도 검사 (자동화 권장)

## 3) TK 활성화 조건이 모호: CR (Contextual Retrieval) 미충족

**증상:**
- 새 TK: "우리는 이런 상황에서는 이렇게 한다"
- "그런데 이런 상황이 정확히 뭐지?"
- 에이전트가 언제 이 TK를 로드해야 할지 불명확

**확인:**
- 활성화 키워드: 구체적인가?
  - 비구체적: "의사결정", "중요"
  - 구체적: "긴급+마감일 없음", "기술팀+예산 변경"
- CR Score 임계값: 정의되었는가? (권장 0.7 이상)
- 검색 테스트: "이 TK를 찾으려면 뭐라고 검색해야 하나?" → 그게 키워드

**조치:**
1. **키워드 구체화**: "의사결정" → "우선순위 판단", "요청 거절", "타이밍 결정" 등 세분화
2. **CR 메타데이터 추가**:
   ```
   활성화 키워드: ["긴급요청", "예상치못한변경", "우선순위판단"]
   CR Score 임계값: 0.72
   비활성화 키워드: ["규제의무", "법률제약"]
   ```
3. **테스트**: "긴급 요청이 왔다" 검색 → TK 나타나는가?
4. **반복 개선**: 실제 사용 후 "자주 놓치는 경우"가 있으면 키워드 추가

## 4) TK가 시간이 지나면서 틀렸다는 걸 깨달음

**증상:**
- 6개월 전 TK: "MVP는 항상 1개 기능만"
- 지금: 시장이 빠르게 변해서 MVP도 3가지 기능 필요
- "이 TK는 유효하지 않다"

**확인:**
- TK 자체가 틀렸나? (보편적 원칙 위반)
- 아니면 맥락이 변했나? (시장/기술 변화)
- 과거에는 왜 유효했는가? (근거는 남아있는가?)

**조치:**
1. **삭제 금지**: TK를 지우지 말 것 (역사와 학습 기록 필요)
2. **활성화 조건 축소**: "MVP는 1개 기능" → "초기 스타트업은 1개 기능"으로 범위 제한
3. **시간 범위 명시**: "2023-2024년 시장 상황에서는 유효했으나, 2025년 이후 재검토 필요"
4. **반대 패턴 TK 작성**: "Expanded MVP" → "성숙 시장에서는 MVP도 3가지 기능 필요"
5. **근본 원인 기록**: "왜 시장이 변했나? 앞으로 어떤 신호를 보면 다시 변할까?" 기록

## 5) TK 유형 분류 애매: Type 2 (Failure Pattern) vs Type 4 (Anti-Pattern)

**증상:**
- TK: "우리는 도구 탓하지 말고, 먼저 프롬프트 문제를 본다"
- 이게 "Failure Pattern" (실패 사례)인가 "Anti-Pattern" (금지 원칙)인가?

**확인:**
- 리트머스 테스트 3가지 질문:
  1. "다시는 틀릴 일이 없는가?"
     - YES → Type 4 (Anti-Pattern, 강한 금지)
     - NO → Type 2 (Failure Pattern, 경고)
  2. "어기는 경우가 정당화될 수 있는가?"
     - YES → Type 2 (예외 가능)
     - NO → Type 4 (절대 금지)
  3. "미래 변화로 역전될 가능성이 있는가?"
     - YES → Type 2 (시간에 따라 달라짐)
     - NO → Type 4 (항상 유효)

**조치:**
1. **리트머스 테스트 적용**:
   - "도구 탓하기"
   - Q1: "다시는 AI가 도구 탓하지 않을까?" → YES (항상 프롬프트 확인 먼저)
   - Q2: "어기는 경우가 정당화될까?" → NO (모델 한계 있어도 프롬프트부터)
   - Q3: "미래에 역전될까?" → NO (기본 원칙은 유지)
   - **결론: Type 4 (Anti-Pattern) — 강한 금지**

2. **분류 결정**: 위 결과로 Type 4가 명확하면 Anti-Pattern으로 분류
3. **지문 명시**: "우리는 이것만큼은 하지 말자" 톤으로 작성
4. **근거 강화**: "이걸 어겼을 때 뭐가 됐나?" 사례 제시
