# Bad Example — pm-framework 오용

## 사용자 요청
"우리는 보통 새 기능을 추가할 때 우선순위를 높은 것부터 한다.
이걸 TK로 만들어줄래?"

## 거절 이유
- 이것은 TK가 아니라 상식: "우선순위가 높으면 높은 것부터 한다"는 누구나 아는 것
- pm-framework의 기준:
  - 반복적으로 검증된 판단인가? → "당연히 그렇다"는 건 검증이 아님
  - 실제 경험에서 배운 뉘앙스가 있는가? → 없음 (너무 일반적)
  - 활성화 조건이 명확한가? → 명확하지 않음 (언제나 적용하는가?)

## 올바른 라우팅
- 먼저 구체적 경험을 찾기: "우선순위를 어떻게 결정해?"
  - "발신자가 CEO면 올라간다?" → Anti-Pattern TK 가능
  - "비즈니스 임팩트로만 결정한다" → Decision Pattern TK 가능
  - "처음 3명의 요청은 무시했는데 10명 넘어야 패턴이 보인다" → Heuristic TK 가능

## 수정 방향
사용자가 더 구체적으로: "우리는 처음엔 '많이 요청하는 사람' 우선으로 했는데,
팀이 분산되고 생산성이 30% 떨어졌어.
그래서 이제 '비즈니스 임팩트'로만 판단하기로 했는데,
이걸 TK로 구조화하고 싶어."

→ 그러면 pm-framework가 적절:
  ```
  TK-NNN: Stakeholder Energy Management

  Type 1 (Decision Pattern) + Type 2 (Failure Pattern)

  패턴: 발신자의 직급이나 압박감이 아니라,
        비즈니스 임팩트로만 우선순위를 결정한다.

  활성화: 여러 이해관계자의 동시 요청이 올 때
  비활성화: 없음

  Why: 발신자 중심 우선순위 → 팀 분산, 집중력 저하 (실제 경험)
       임팩트 중심 우선순위 → 생산성 향상 (검증됨)
  ```
