# PM-ENGINE-MEMORY Starter Kit

> 이 파일은 pm-engine 스킬의 **빠른 시작 템플릿**입니다.
> 아래 5개의 Seed TK를 복사한 뒤, 자신의 경험으로 수정/추가하세요.
> 실제 사용 시 이 파일을 `PM-ENGINE-MEMORY.md`로 저장합니다.

---

## 사용법

1. 이 파일을 프로젝트 루트에 `PM-ENGINE-MEMORY.md`로 복사
2. Seed TK의 `[내용]` 부분을 자신의 경험으로 교체
3. 매일 1개씩 TK를 추가 — `/pm-tacit-extract` 명령 활용
4. pm-engine 스킬이 자동으로 이 파일을 참조합니다

---

## Seed TK (5개)

### TK-001: 긴급 요청 우선순위 판단

- **핵심 원칙**: 긴급(Urgent)과 중요(Important)를 분리한다. "CEO가 말했으니까"는 긴급이지 중요가 아닐 수 있다.
- **판단 기준**: ①사용자 영향 범위 ②매출 직결 여부 ③되돌릴 수 없는 정도(Irreversibility)
- **🟢 활성화 조건**: 이해관계자가 "급하다"고 말할 때
- **🔗 연관 TK**: TK-004
- **출처**: [프로덕트 매니저로서 3년간 200건+ 긴급 요청 처리 경험]

### TK-002: AI 네이티브 사고 필터

- **핵심 원칙**: 새 기능을 기획할 때 "AI가 이 작업의 80%를 할 수 있는가?"를 첫 번째로 묻는다.
- **판단 기준**: ①반복성 ②패턴 존재 여부 ③오류 허용 범위
- **🟢 활성화 조건**: 새 기능 또는 워크플로우를 설계할 때
- **🔗 연관 TK**: TK-003, TK-005
- **출처**: [AI SaaS 제품 3개 런칭 경험에서 도출]

### TK-003: 에이전트 비용 10배 법칙

- **핵심 원칙**: AI 에이전트의 운영 비용은 초기 예상의 10배가 된다. API 비용뿐 아니라 모니터링, 오류 복구, 엣지 케이스 핸들링을 반드시 포함한다.
- **판단 기준**: ①API 토큰 비용 ②재시도/재처리 비율 ③인간 개입 빈도
- **🟢 활성화 조건**: 에이전트 ROI를 계산하거나 비용 시뮬레이션할 때
- **🔗 연관 TK**: TK-002
- **출처**: [AI Dubbing 서비스 운영 1년차 비용 분석]

### TK-004: "2주 법칙" — MVP 기능 스코핑

- **핵심 원칙**: MVP 기능 하나가 2주를 넘으면 스코프가 잘못 잡힌 것이다. 2주 안에 끝나는 단위로 쪼개라.
- **판단 기준**: ①기능 단위가 독립적으로 릴리즈 가능한가 ②A/B 테스트 가능한가 ③유저가 가치를 느끼는가
- **🟢 활성화 조건**: 스프린트 플래닝 또는 기능 스코핑 시
- **🔗 연관 TK**: TK-001
- **출처**: [Agile PM 5년, 120+ 스프린트 운영]

### TK-005: 데이터 없는 가설은 가설이 아니다

- **핵심 원칙**: "유저가 원할 것 같다"는 가설이 아니라 희망이다. 최소한 정성 데이터(인터뷰 3건) 또는 정량 데이터(이벤트 로그 1주)가 있어야 가설이다.
- **판단 기준**: ①근거 유형(정성/정량/없음) ②샘플 수 ③반복 확인 가능성
- **🟢 활성화 조건**: PRD 작성 시 "문제 정의" 단계
- **🔗 연관 TK**: TK-002
- **출처**: [Data-Driven PM 방법론 + 유저 리서치 50건+ 경험]

## v0.7.0 Seed TK (운영 노하우 4종)

> hplan을 실제 운영 중인 PM의 암묵지에서 추출. 4영역(실행 통합 / PRD 검증 깊이 / 에이전트 생태계 / PPTX 생산성).

### TK-006: 독립 태스크 ≥2면 worktree + 4명+ 팀

- **핵심 원칙**: 파일 충돌 없는 독립 태스크가 2개 이상이면 `.worktrees/<task>` 격리 + Lead/Builder×2/Reviewer/Verifier 4명+ 팀으로 병렬 진행한다.
- **판단 기준**: 작업 쌍이 수정 파일을 공유하는가 (0건이면 병렬 OK, 1건이라도 있으면 직렬화)
- **🟢 활성화 조건**: 새 빌드 시작 시 + 사용자가 "ralph loop / 자율 모드" 명시
- **🔗 연관 TK**: TK-007 (PRD 검증 깊이)
- **출처**: [hplan 운영 중 worktree 미사용 시 머지 충돌 다수 발생 경험]

### TK-007: PRD는 workflow + userflow 두 다이어그램 정합성으로 검증

- **핵심 원칙**: PRD에 mermaid workflow(시스템 관점) + userflow(유저 관점) 두 다이어그램을 모두 두고, requirements와의 매핑을 결정론 스크립트로 검증한다.
- **판단 기준**: orphan(한쪽에만 있는 노드) ≤ 임계치 + missing requirement = 0
- **🟢 활성화 조건**: PRD 작성 또는 handoff 직전
- **🔗 연관 TK**: TK-006, TK-008
- **출처**: [PRD에 "맞춰봤다"는 LLM 자기보고를 신뢰하다 구현 단계 누락 발견 경험]

### TK-008: 5+ 에이전트면 단일 KPI가 아니라 5축 가중 ScoreCard로 비교

- **핵심 원칙**: 운영 에이전트가 5개를 넘으면 개별 KPI 뷰는 신호가 사라진다. Accuracy/Reliability/Cost/Velocity/Satisfaction 5축 가중 단일 점수로 비교 가능하게 만든다.
- **판단 기준**: 가중치 합 = 100 / 사업 컨텍스트에 따라 그룹별 재배분 / 단일 점수 < 60은 sunset 또는 즉시 개선
- **🟢 활성화 조건**: 포트폴리오 ≥5 에이전트 + 주간/월간 운영 회의
- **🔗 연관 TK**: TK-009
- **출처**: [다수 에이전트 다팀 운영 시, 단일 KPI 뷰로는 우선순위 분배가 불가능했던 경험]

### TK-009: PPTX는 단일 흐름이 아니라 4엔진 라우팅 결정으로 시작

- **핵심 원칙**: 슬라이드 생성을 단일 도구로 처리하면 청중·규모·스타일이 맞지 않는 비효율이 발생. 입력 신호(장수/스타일/이미지 필요/영상 입력)로 4엔진 중 하나에 라우팅한다.
- **판단 기준**: 영상 입력 → video / 장수 ≥30 → mckinsey / 장수 ≤10 + 이미지 → hifidelity / 그 외 → html-qa
- **🟢 활성화 조건**: PPTX 요청 진입 시 Step 0 엔진 선택
- **🔗 연관 TK**: TK-008 (운영 비용 가시화)
- **출처**: [4엔진 라우팅 운영, 디폴트 단일 엔진 대비 처리 시간·디자인 일관성 모두 개선 검증]
