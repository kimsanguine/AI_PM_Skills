## Step 1~3 분석

- **상황**: 10개+ 프로젝트에서 반복 관찰된 스코프 크리프 패턴
- **핵심 판단**: 거절 타이밍과 언어 선택 — "안 된다" vs "다음 스프린트에"
- **TK 유형**: **Type 1 — Decision Pattern** (작은 요청을 받는 순간의 판단 기준)

## Step 4 — TK 구조화

```
TK-041: 스코프 크리프 리다이렉션 타이밍

📌 패턴:
"이것만 하나 더"라는 요청이 오면, 거절하지 않는다.
대신 즉시 "이 요청의 크기와 현재 스프린트 여유"를 계산하고,
여유가 없으면 "다음 스프린트에 넣겠습니다"로 전환한다.
거절이 아니라 일정 내 수용 — 이것이 관계를 보호하면서 스코프를 지키는 방법이다.

🟢 활성화 조건:
현재 스프린트가 60% 이상 채워진 상태에서 추가 요청이 올 때.
요청자가 "작은 거"라고 프레이밍할 때 — 이 표현 자체가 경보 신호.

🔴 비활성화 조건:
추가 요청이 현재 스프린트의 핵심 목표와 직접 연결되어 있고,
제외 시 deliverable이 불완전해지는 경우.

💡 Why:
스코프 크리프의 위험은 단일 요청이 아니라 누적이다.
10개 프로젝트 경험에서 일정 지연의 원인을 역추적하면
항상 "그때 그 작은 요청"들이 쌓여 있었다.
"안 된다"는 이해관계자와의 마찰을 만들지만,
"다음 스프린트에"는 수용의 언어이면서 현재 스코프를 지킨다.
거절의 내용이 아니라 타이밍과 언어가 팀의 신뢰를 결정한다.

🔗 연관 TK: TK-001 (긴급 요청 우선순위 판단), TK-008 (3회 반복 → 자동화 원칙)
```

## Step 5 — 추가 포착된 하위 패턴

이 경험에서 **2개의 추가 TK**를 더 뽑을 수 있습니다:

```
TK-042: "작은 거"라는 표현 = 스코프 리스크 신호

📌 패턴:
요청자가 "이건 작은 거예요", "금방 되지 않나요?"라고 말하는 순간,
그 요청의 실제 크기를 반드시 추정한다.
발신자의 크기 인식은 항상 구현 복잡도를 과소평가한다.

🟢 활성화 조건: "금방", "작은 거", "간단한 수정" 표현 감지 시
🔴 비활성화 조건: 기술적으로 검증된 단순 텍스트/디자인 수정
💡 Why: 표현의 단순함이 요청자의 기대치를 낮추고, PM의 방어를 낮춘다.

🔗 연관 TK: TK-041
```

```
TK-043: 스코프 크리프 누적 모니터링

📌 패턴:
스프린트 중반(Day 3~4)에 "추가된 요청 목록"을 리뷰한다.
원래 스코프 대비 10% 이상 추가됐으면 반드시 조정한다.

🟢 활성화 조건: 스프린트 중반 체크포인트
🔴 비활성화 조건: 추가 요청이 삭제된 항목으로 상쇄된 경우
💡 Why: 크리프는 단일 사건이 아닌 누적 현상. 개별 모니터링으로는 못 잡는다.

🔗 연관 TK: TK-041, TK-042
```
