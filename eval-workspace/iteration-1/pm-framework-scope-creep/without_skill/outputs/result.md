# TK 유닛: 스코프 크리프 차단 - "다음 스프린트" 타이밍 기술

---

## 1. 핵심 인사이트 (1-sentence)
> **스코프 크리프는 거절의 문제가 아니라 타이밍의 문제다.**

---

## 2. 문제 구조

```
작은 요청 1건
  → "이건 금방 되지 않나요?"
    → 합의 없이 흡수
      → 작은 요청 2건
        → ...
          → 일정 지연 → 원인 불명확 → 팀 소진
```

핵심 패턴: 각 요청은 **작아 보이지만**, 의사결정 비용·맥락 전환·QA 범위가 따라온다.

---

## 3. 원칙

| 원칙 | 내용 |
|---|---|
| **가시화** | 추가 요청은 반드시 백로그에 등재한다. 머릿속 합의는 없다. |
| **분리** | "할 수 있냐"와 "지금 하냐"를 명확히 구분해서 말한다. |
| **대안 제시** | 거절이 아니라 배치(placement)로 답한다. |

---

## 4. 실행 프레임워크 - "3-초 판단 트리"

```
새 요청 들어옴
       │
       ▼
현 스프린트 목표에 직접적 영향?
  ├─ YES → 팀과 트레이드오프 논의 (무엇을 빼낼지)
  └─ NO  → "좋은 아이디어입니다. 백로그에 넣고 다음 스프린트에 평가합시다."
                │
                ▼
           즉시 백로그 등재 (그 자리에서)
           → 요청자에게 링크/티켓 공유
           → 우선순위 논의 일정 잡기
```

---

## 5. 타이밍별 스크립트

**요청이 들어오는 순간 (가장 중요)**
> "지금 스프린트에 넣으면 [X]가 밀립니다. 다음 스프린트 첫 번째로 올리면 어떨까요?"

**이미 여러 건이 쌓였을 때**
> "지금까지 추가된 항목이 [N]개인데, 한번 같이 우선순위를 보겠습니다."

**'금방 되는 거잖아요' 압박 시**
> "작업 자체는 2시간이지만, 지금 넣으면 QA와 배포 범위가 바뀝니다."

---

## 6. 흔한 함정

- **백로그 등재를 나중으로 미루는 것** → 구두 합의는 기억이 엇갈린다
- **"안 된다"고 말하는 것** → 관계 비용이 크고, 실제로 안 되는 게 아님
- **트레이드오프 없이 흡수하는 것** → "다 할 수 있다"는 신호를 계속 보내게 됨

---

## 7. 전달 가능한 핵심 (팀에게 가르칠 때)

1. 추가 요청 = **좋은 아이디어** + **비용 존재** → 두 개를 분리해서 다뤄라
2. "다음 스프린트에"는 **거절이 아니라 예약**이다
3. 백로그 가시화가 곧 **스코프 방어선**이다
