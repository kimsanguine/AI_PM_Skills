# Bad Example — 엔진 라우팅 실패 패턴

> 라우팅 결정을 생략하거나 잘못 내려서 시간·비용·품질이 깨지는 케이스.

---

## Anti-pattern 1 — 모든 요청을 디폴트로 처리

**잘못된 결정:**
> 모든 요청을 html-qa로 보냄.

**문제:**
- 40장 강의자료를 html-qa로 만들면 슬라이드별 개별 렌더링이 폭주 → 처리 시간 10배
- 디자인 일관성도 깨지기 쉬움

**올바른 결정:**
- 장수 ≥ 30 → mckinsey (PptxGenJS 결정론 디자인 + 속도)

---

## Anti-pattern 2 — 영상 입력을 직접 처리

**잘못된 결정:**
> YouTube URL을 html-qa나 mckinsey로 바로 입력.

**문제:**
- 두 엔진 모두 영상을 받지 못함
- 사용자가 직접 전사하고 정리해서 다시 입력해야 함 (수작업 폭증)

**올바른 결정:**
- 영상 입력은 무조건 video 엔진 전처리 → 후속 엔진 체이닝

---

## Anti-pattern 3 — 5장 피치덱을 mckinsey로

**잘못된 결정:**
> 투자 피치 5장을 mckinsey 엔진으로 생산.

**문제:**
- McKinsey 톤은 5장 임팩트 피치에 과중함
- 비주얼 임팩트 부족 (이미지 자동 생성 부재)

**올바른 결정:**
- 5장 + 피치 + 이미지 필요 → hifidelity

---

## Anti-pattern 4 — Step 0 생략

**잘못된 결정:**
> 엔진 선택 근거 없이 바로 슬라이드 작성 시작.

**문제:**
- 도중에 엔진 변경 필요 인지 → 재작업
- 비용 예측 불가

**올바른 결정:**
- Step 0에서 엔진 + 근거 + 체인 + 비용 경고를 명시적으로 출력
