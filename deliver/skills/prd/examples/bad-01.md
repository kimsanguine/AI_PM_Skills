# Bad Example — Unified PRD가 빠지기 쉬운 함정

> 1인 빌더가 자주 만드는 추상·솔루션 어조 PRD

---

## 잘못된 PRD 예시

```markdown
# PRD — 변호사 AI 도구

목표: 변호사들이 AI를 편하게 쓸 수 있게 합니다.

타깃: 모든 한국 변호사 (4만명)

기능:
- AI 채팅
- 검색 기능
- 자동화

수익화: 월 구독 (TBD)

성공 지표: 사용자 만족
```

---

## 문제점

| 함정 | 왜 문제인가 | 어떻게 고치나 |
|------|------------|------------|
| **ICP가 4만명 전체** | beachhead 5-criteria 통과 못 함 — 너무 넓어 메시지 만들 수 없음 | "1-5인 변호사 사무소 / 7년차 이상" 같이 좁힘 |
| **JTBD 없음 — "편하게 쓸 수 있게"** | Job이 없고 Feature만 있음 | "유사 사건을 5분 안에 정리하고 싶다" 같이 Job 작성 |
| **솔루션 어조 — "AI 채팅·검색·자동화"** | 사용자가 그것을 원하는 게 아님 | 워크플로우 어조로 — "본 제품으로 일하는 방식이 어떻게 바뀌나" |
| **제외사항 없음** | 모든 걸 만들겠다는 신호 → 60일 안에 못 끝남 | "절대 안 하는 것 5개" 강제 |
| **MVP·Full 분리 없음** | "기능: AI 채팅·검색·자동화" 셋 다 60일에 못 만듦 | Now/Next/Later 3-tier 강제 분할 |
| **결정 옵션 매트릭스 없음** | 어떤 RAG 인프라? 결제? Orchestration? | 최소 5개 결정 + 트레이드오프 명시 |
| **Anti-Goals 없음** | 변호사 책임 영역에서 hallucination 위험 | 최소 3개 — 도메인·데이터·법적 책임 |
| **Tools 호출 제한 없음** | API 폭주 시 cogs 즉시 망함 | 모든 도구에 호출 제한 명시 |
| **3-tier Memory 없음** | 컨텍스트 윈도우 초과 위험 | Working / Long-term / Procedural 분리 |
| **Trigger 모호** | 언제 작동하는지 모름 | Cron/Event/Manual/Pipeline 중 명시적 선택 |
| **Output 예시 없음** | 결과 품질 검증 불가 | 실제 출력 1개 작성 강제 |
| **성공 지표 추상 — "사용자 만족"** | 측정 불가 | Dual-axis (Business + Operational) + Anti-Metric |
| **검증 가능 가설 없음** | 60일 후 가설 검증 못 함 | Top-3 가설 + 2-day experiment |
| **실패 모드 없음** | Hallucination·결제 실패·데이터 유출 시 대응? | Section 14 매트릭스 + HITL 트리거 |

---

## 1줄 진단

> 14개 섹션 중 14개 모두 비어 있거나 추상. 5명 사랑 인터뷰에 들고 갈 수 없는 PRD.
