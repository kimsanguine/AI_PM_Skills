# Troubleshooting — okr 스킬

## 1) 비즈니스 임팩트 KR이 측정 불가능할 때

**증상:**
- "우리 OKR의 KR1이 '사용자 만족도 향상'인데 어떻게 측정하죠?"
- "이 KR을 수치로 어떻게 표현해야 할지 모르겠어요."
- "정성적 목표인데 정량화할 방법이 없어요."

**확인:**
- KR 현재 표현이 추상적인가? ("향상", "개선", "높이기" 등)
- 측정 가능한 지표로 전환할 대안이 있는가?
- 도메인 특화 프록시 지표를 정의할 수 있는가?

**조치:**
1. 추상적 KR → 측정 가능한 프록시 지표로 변환:
   ```
   // Bad
   KR: 사용자 만족도 향상

   // Good (프록시 지표 3가지 예시)
   KR: 에이전트 재사용률 80% 이상
   KR: 주간 호출 건수 500회 이상
   KR: 고객 피드백 4.5/5.0 점 이상 (월간 설문)
   ```
2. 프록시 지표 선택 기준:
   - 정량화 가능한가?
   - 에이전트의 실제 가치를 대표하는가?
   - 측정 가능한 도구/방법이 있는가?
3. 팀과 합의: 선택한 프록시 지표가 진짜 목표를 반영하는지 검증

## 2) 월간 리뷰 시 데이터 부재로 KR 달성률 계산 불가

**증상:**
- "지난달 KR 성과를 계산하려고 하는데 로그가 없어요."
- "매주 또는 매월 기록을 안 해서 추정만 할 수 있어요."
- "정확한 성과를 모르기 때문에 다음 분기 조정이 어렵습니다."

**확인:**
- KR별 자동 로깅 설정 여부: 각 KR마다 자동으로 기록되는 시스템이 있는가?
- 수동 로깅: 매주/매월 누군가 기록하는 프로세스가 있는가?
- 계산 방법: 마지막 주 데이터만인가 아니면 누적 데이터인가?

**조치:**
1. KR별 자동 로깅 설정 (우선):
   ```python
   # 예: API 호출 성공률 자동 기록
   def log_kr_metric(kr_name, value, timestamp):
       db.insert({
           'kr': kr_name,
           'value': value,
           'timestamp': timestamp,
           'date': datetime.now().date()
       })

   # 에이전트 실행 후 매번 호출
   success_rate = (successful_calls / total_calls) * 100
   log_kr_metric("KR3: 성공률", success_rate, now())
   ```
2. 수동 로깅 프로세스 (자동화 불가능한 KR):
   - 매주 금요일: PM이 정성적 KR 상태 기록 (Notion / 스프레드시트)
   - 형식: `[날짜] KR_NAME: 현재값, 진행 사항`
3. 지난달 리뷰: 로그 없으면 추정값 사용
   ```
   KR1: "약 75% 달성했을 것으로 추정" (근거: 주간 체감)
   ```
4. 다음 달부터: 자동 로깅으로 정확한 데이터 수집

## 3) KR이 너무 높게 설정되어 분기 중반부터 달성 불가능 확실할 때

**증상:**
- "처음에 설정한 KR이 너무 높아서 이미 달성 불가능해 보여요."
- "주간 진행률이 목표의 30%인데, 이대로면 달성 불가합니다."
- "KR을 낮춰도 되나요? 아니면 그냥 놔둬요?"

**확인:**
- 현재 달성률: 목표 대비 얼마나 진행했는가? (예: 50점 목표 중 10점만 달성)
- 달성 불가능성 확실도: "거의 불가능"인가 아니면 "노력하면 가능"인가?
- KR 설정 오류 원인: 아키텍처 문제? 스코프 미포함? 외부 의존성?

**조치:**
1. 즉시 KR 재협상: 낮게 재설정 (고민하지 말고)
   ```
   // 원래 KR
   KR1: 에러율 0% (현재 15% → 달성 거의 불가능)

   // 재설정
   KR1: 에러율 10% 이하로 감소 (더 현실적)
   ```
2. 재설정 이유 문서화:
   - "초기 추정 실패: 아키텍처 개선이 예상보다 복잡"
   - "스코프 외 의존성: 외부 API 지연"
   - "데이터 부재: 초기 데이터 수집 부족"
3. 다음 분기 교훈: 이번 오차를 기반으로 다음 KR 설정 조정
4. 팀 소통: 재협상 이유를 팀과 공유 (신뢰성 유지)

## 4) 비용 KR을 초과했을 때

**증상:**
- "월간 비용이 $50으로 설정했는데 $150 나왔어요."
- "어디서 비용이 이렇게 많이 들었을까요?"
- "다음 달은 어떻게 조정할까요?"

**확인:**
- 비용 대폭 증가의 원인: 모델 티어 변경? 요청 빈도 증가? 토큰 사용량 증가?
- 예산 문제인가 아니면 추정 오류인가?
- 비용 최적화 가능성: 어디를 줄일 수 있는가?

**조치:**
1. 원인 분석:
   ```
   예산: $50 (월)
   실제: $150

   비용 항목별 분석:
   - Haiku 호출: $30 → $50 (API 호출 2배 증가)
   - Sonnet 호출: $20 → $80 (모델 티어 상향)
   - 저장소: 무료 → $20 (데이터 축적)
   ```
2. 우선순위 최적화 (비용 감소 순서):
   - 1순위: 모델 다운그레이드 (Sonnet → Haiku, 가능한가?)
   - 2순위: API 호출 빈도 감소 (배치 처리, 캐싱)
   - 3순위: 토큰 사용 최소화 (프롬프트 최적화)
   - 4순위: 저장소 정리 (오래된 데이터 삭제)
3. 다음 달 목표 재설정:
   - 현실적 비용 상한선 설정: $50 → $80 (버퍼 추가)
   - 비용 KR 모니터링 강화: 주간 단위로 추적

## 5) 여러 에이전트의 KR이 충돌할 때

**증상:**
- "고객 지원 에이전트는 비용을 늘려야 하는데, 예산 한도 때문에 다른 에이전트를 줄여야 해요."
- "A 에이전트와 B 에이전트의 비용이 모두 증가하면 전체 예산을 초과합니다."
- "어떤 에이전트를 우선순위로 할지 결정해야 합니다."

**확인:**
- 포트폴리오 전체 비용 한도: 얼마나 정해졌는가?
- 각 에이전트의 비즈니스 임팩트: KR1 달성도가 높은가?
- 현재 배분: 어느 에이전트에 리소스가 집중되어 있는가?

**조치:**
1. 포트폴리오 리뷰: 각 에이전트의 KR 달성률 + 비용 효율성 매트릭스
   ```
   | 에이전트 | 월 비용 | 임팩트 KR 달성률 | 효율 (임팩트/$) |
   |---------|--------|-----------------|-----------------|
   | 고객 지원 | $100 | 80% | 0.8 |
   | 뉴스 수집 | $30 | 60% | 2.0 |
   | 마케팅 | $50 | 40% | 0.8 |
   ```
2. 우선순위 재정렬:
   - High Impact + High Efficiency: 리소스 증가
   - Low Impact + Low Efficiency: 일시 중지 또는 감축
3. 비용 재할당:
   - 총 예산: $200
   - 고객 지원: $120 (우선순위 1)
   - 뉴스 수집: $60 (우선순위 2)
   - 마케팅: $20 (일시 중지 또는 축소)
4. 팀 커뮤니케이션: 재정렬 이유와 일정 공유
