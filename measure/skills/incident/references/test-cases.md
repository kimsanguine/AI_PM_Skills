# Test Cases — incident 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "정확도가 95%에서 45%로 갑자기 떨어졌어. 뭔가 심각하게 잘못된 것 같다"
   - 이유: 급락(> 10%)은 명확한 장애 신호 → 즉시 대응 필요

2. "API 호출 비용이 어제는 $100이었는데 오늘 $1,000이 나왔다"
   - 이유: 비용 폭증(2배 이상) → incident 분류

3. "고객이 우리 에이전트가 민감한 고객 정보를 외부에 노출시켰다고 신고했다"
   - 이유: 데이터 유출 의심 → SEV-1 긴급 대응

4. "어제부터 에이전트가 계속 틀린 정보를 주고 있어"
   - 이유: 환각/오정보 반복 → SEV-2 급격한 품질 저하

5. "내부 QA에서 에러율이 2%에서 18%로 급증했다고 보고했다"
   - 이유: 신뢰성 급락 → incident 분류

### Should NOT Trigger (5)

1. "1주일 정확도가 92%에서 90%로 내려갔는데 점진적인 건가 급락인가?"
   - 올바른 라우팅: `cohort` (시간 추이 분석, 점진적 저하 추적)

2. "A/B 테스트로 어느 프롬프트가 더 좋은지 통계적으로 검증해 줄 수 있나?"
   - 올바른 라우팅: `agent-ab-test` (성능 비교 검증)

3. "정확도 급락의 원인을 5 Whys로 찾아야 하는데 어떻게 시작할까?"
   - 올바른 라우팅: `premortem` 또는 `incident` (원인 분석 후, 재발 방지)

4. "이번 달 비용이 많이 드니까 최적화 전략이 필요하다"
   - 올바른 라우팅: `burn-rate` (비용 추적 및 최적화)

5. "KPI 대시보드에서 여러 메트릭이 악화되고 있다"
   - 올바른 라우팅: `kpi` (메트릭 정의 및 모니터링) 또는 `incident` (급락만 해당)

## 2) Edge Cases

### 경계 사례 (5)

1. **심각도 분류 착오**
   - 입력: "SEV-3로 분류했는데 실제로는 5,000명이 영향받았다"
   - 예상 행동: 심각도 즉시 SEV-2로 상향, 대응 인력 증원, 에스컬레이션
   - 근거: 심각도 분류 오류는 대응 지연으로 이어짐

2. **원인 분석 불완전**
   - 입력: "API timeout이 발생했다는 걸 알았는데, 왜 타임아웃이 났는지는 모르겠다"
   - 예상 행동: 5 Whys 최소 3단계 이상 진행, 시스템 레벨 원인까지 파악
   - 근거: 직접 원인만 찾으면 같은 실패가 반복됨

3. **블래스트 반경 과소 평가**
   - 입력: "100명이 영향받았다고 보고했는데 나중에 1,000명이었다"
   - 예상 행동: 영향도 재산정, 추가 공지, 보상 재계산, TK 추출 내용 확대
   - 근거: 저평가하면 사후 대응 부족

4. **롤백 불가능 상황**
   - 입력: "이전 버전이 없거나 복구 경로를 모른다"
   - 예상 행동: 즉시 hotfix 대안 검토, 임시 우회(workaround) 시행, 향후 버전 관리 강화
   - 근거: 롤백 불가능하면 복구 시간 지연

5. **커뮤니케이션 지연**
   - 입력: "발견 후 12시간 후에 고객에게 공지했다"
   - 예상 행동: 1시간 이내 1차 공지 원칙, 4시간 주기 업데이트, 신뢰도 재확보 계획
   - 근거: 지연된 공지는 신뢰도 추락, 소셜 미디어 부정적 언급 증가
