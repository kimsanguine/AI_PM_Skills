# Test Cases — kpi 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "새로운 고객 지원 에이전트를 배포했는데 어떤 지표를 모니터링해야 할까?"
   - 이유: 에이전트 KPI 정의가 필수 → 명확한 트리거

2. "Accuracy, Latency, Cost 등 여러 지표가 있는데 뭐가 가장 중요한가?"
   - 이유: KPI 우선순위 정렬과 대시보드 설계 필요

3. "SLA를 정의해야 하는데 realistic한 목표 값이 뭘까?"
   - 이유: KPI와 SLA 연결 (alert threshold 설정)

4. "팀마다 Accuracy를 다르게 계산하고 있다. 기준을 통일하려면?"
   - 이유: KPI 정의 표준화

5. "월간 비용이 계속 증가하고 있는데 일일/주간 모니터링이 필요하다"
   - 이유: 실시간/주기적 KPI 추적 구축

### Should NOT Trigger (5)

1. "지난 분기 North Star 메트릭 추이를 보고 싶다"
   - 올바른 라우팅: `north-star` (단일 핵심 지표)

2. "여러 버전의 Accuracy 추이를 비교하고 싶다"
   - 올바른 라우팅: `cohort` (버전별 장기 추적)

3. "정확도가 95%에서 45%로 급락했다"
   - 올바른 라우팅: `incident` (장애 대응)

4. "A/B 테스트에서 정확도 개선을 통계적으로 검증해야 한다"
   - 올바른 라우팅: `agent-ab-test` (성능 비교)

5. "비용을 최적화하고 싶다. 최적화 전략이 뭘까?"
   - 올바른 라우팅: `burn-rate` (비용 추적 및 최적화)

## 2) Edge Cases

### 경계 사례 (4)

1. **KPI 정의 불명확**
   - 입력: "Accuracy를 추적하는데 정답 판단 기준이 다르다"
   - 예상 행동: 정의 표준화 (formula, data source, collection method 명시), 자동화 계산으로 전환
   - 근거: 정의 불명확하면 팀 간 혼동, 의사결정 근거 약화

2. **임계값 설정 오류**
   - 입력: "Accuracy > 95% 목표인데 기준선이 92%라서 달성이 어렵다"
   - 예상 행동: 비교값 기반 현실적 재설정 (현재 92% → 3개월에 94% 같이 단계적), 달성 가능성 검증
   - 근거: 달성 불가능한 목표는 팀 사기 저하

3. **Guardrail 부재**
   - 입력: "CPE 최적화에만 집중했더니 Accuracy 85%로 급락했다"
   - 예상 행동: Anti-metric 명시 (CPE < $0.1이어도 Accuracy > 90% 유지), 트레이드오프 관리
   - 근거: 주 지표 최적화가 다른 지표를 훼손할 수 있음

4. **데이터 수집 지연**
   - 입력: "일일 리포트가 2-3일 후 생성되어 의사결정이 늦다"
   - 예상 행동: 자동화 대시보드 구축, 실시간/일일 메트릭 분리, 주기 단축
   - 근거: 지연된 데이터는 의사결정 타이밍 상실
