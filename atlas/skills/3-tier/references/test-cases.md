# Test Cases — 3-tier 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "3개의 에이전트(전략, 조율, 실행)가 협력해서 복잡한 작업을 해야 한다"
   - 이유: 3-tier Prometheus-Atlas-Worker 패턴 설계 필수

2. "전략팀이 목표를 정하면 실행팀이 그걸 어떻게 분해해서 작업을 나눌지 명확하지 않다"
   - 이유: Atlas 조율 계층 설계 필요

3. "여러 Workers가 병렬로 작업하는데 결과를 어떻게 통합할 것인가?"
   - 이유: 멀티 에이전트 오케스트레이션 패턴 설계

4. "PM 워크플로우에서 전략 수립(Prometheus) → 계획 수립(Atlas) → 실행(Workers)을 체계화하고 싶다"
   - 이유: 3-tier 구조로 복잡한 멀티 에이전트 시스템 설계

5. "에이전트 간 데이터 전달, 에러 처리, 품질 관리를 명확히 정의하고 싶다"
   - 이유: 3-tier 통신 프로토콜 정의

### Should NOT Trigger (5)

1. "두 개의 에이전트만 있으면 되는데 어떻게 연결할까?"
   - 올바른 라우팅: `orchestration` (2개 에이전트 간 기본 패턴)

2. "어떤 에이전트로 어떤 작업을 할지 라우팅해야 한다"
   - 올바른 라우팅: `router` (작업 분류 및 에이전트 선택)

3. "단일 에이전트의 성능을 개선하고 싶다"
   - 올바른 라우팅: 다른 argus 스킬들 (agent-ab-test, kpi 등)

4. "멀티 에이전트 시스템의 경쟁 우위를 분석하고 싶다"
   - 올바른 라우팅: `moat` (해자 분석)

5. "복잡한 워크플로우를 자동화하는 최적 구조를 알고 싶다"
   - 올바른 라우팅: `orchestration` (일반 오케스트레이션 패턴)

## 2) Edge Cases

### 경계 사례 (4)

1. **Worker 응답 타임아웃**
   - 입력: "Worker가 지정된 시간 내 응답하지 않는다"
   - 예상 행동: Atlas가 자동으로 재시도, 대체 Worker로 전환, 또는 Prometheus에 지연 보고
   - 근거: 분산 시스템에서 타임아웃은 흔함, 자동 복구 필수

2. **Worker 출력 형식 불일치**
   - 입력: "Worker의 결과 형식이 예상과 다르다"
   - 예상 행동: Atlas가 결과 파싱 실패 감지, Worker에 재요청, 또는 예비 Worker 호출, 최후의 수단으로 수동 개입
   - 근거: 형식 불일치는 자동화 파이프라인을 중단

3. **Prometheus의 모순된 목표**
   - 입력: "목표가 '빨리 완료하면서 동시에 완벽한 품질'로 상충한다"
   - 예상 행동: Atlas가 명확화 요청을 Prometheus로 전송, 재정의된 목표 대기
   - 근거: 모순된 제약을 받으면 실행 불가능

4. **최종 결과 품질 미달**
   - 입력: "결과가 완성되었는데 Prometheus 기준을 충족하지 않는다"
   - 예상 행동: Atlas가 재작업 사이클 시작, 최대 N회 반복 후 실패 보고 및 원인 분석
   - 근거: 품질 게이트 통과 없이는 결과 반환 불가
