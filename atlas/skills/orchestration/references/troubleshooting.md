# Troubleshooting — orchestration 스킬

## 1) 선택한 패턴이 실제 요구사항과 안 맞음

**증상:**
- 배포 후 예상 지연: 2초, 실제 지연: 6초
- 또는 에러율이 <1% 목표인데 실제 5%
- "이 패턴이 맞지 않는 것 같은데?"

**확인:**
- 패턴 선택 시 정의한 요구사항 확인:
  - 지연시간 허용치: 정확했는가?
  - 에러 허용치: 정확했는가?
  - 태스크 의존성: 모두 정확히 파악했는가?
- 배포 후 실제 수행 시간 측정:
  - Sequential: sum(모든 agent) = ?
  - Parallel: max(모든 agent) = ?
  - 네트워크/IO 오버헤드: 예상했는가?

**조치:**
1. **요구사항 재분석**: "정말 2초여야 하나? 3초로 늘려도 될까?"
2. **패턴 전환**:
   - 지연 초과 → Sequential → Parallel로 변경
   - 또는 Parallel에서 느린 agent 분해 (1개를 2개 병렬로)
3. **하이브리드 패턴**: Sequential + Parallel 결합 (Research와 Marketing은 병렬, 그 후 Analysis 순차)
4. **Timeout 조정**: 완벽한 응답 대신 부분 응답 (예: top-3만 aggregation)
5. **3개월 재측정**: 변경 후 실제 개선 효과 검증

## 2) 병렬 패턴에서 느린 agent 병목

**증상:**
- Agent A: 1초, Agent B: 0.8초, Agent C: 3초
- 전체 지연: 3초 (Agent C 대기)
- "느린 agent 때문에 30% 지연 증가?"

**확인:**
- 느린 agent의 이유:
  - 모델이 느린가? (T3 vs T1)
  - API 레이턴시 문제?
  - 자체 로직이 복잡한가?
- 빠를 수 없는 작업인가? (근본적 복잡도)
- Fan-in aggregation 시간: 몇 초?

**조치:**
1. **Timeout 설정**: Fan-in 최대 대기 2초, 그 후 부분 결과로 aggregation
2. **느린 agent 분해**:
   - 1개의 복잡한 task → 2~3개의 독립 task로 쪼개기
   - 예: "전체 분석" → "감정 분석" + "카테고리 분석" (병렬 실행)
3. **빠른 버전 추가**: T3 agent → T2 fallback (품질 손실, 속도 gain)
4. **캐싱 도입**: 같은 입력이 반복되면 이전 결과 재사용
5. **비동기 업데이트**: Agent C가 느리면 → A, B 결과 먼저 반환, C는 비동기 후 전달

## 3) 라우터 분류 오류: 모호한 입력

**증상:**
- 사용자: "가격을 어떻게 책정해요? 우리 예산이 부족한데..."
- 라우터: 영업 에이전트로 라우팅 (실제로는 기술 + 가격 복합)
- 영업 에이전트만으로는 부족, 기술 에이전트 필요

**확인:**
- 라우터 학습 데이터: 모호한 케이스를 충분히 포함했는가?
- 프롬프트 명확성: "다음 중 가장 가까운 카테고리: A/B/C" 형태로 충분한가?
- 신뢰도: 라우터가 자신감 점수를 제공하는가? (0~1)

**조치:**
1. **라우터 프롬프트 개선**:
   - 명확한 예제 추가 (모호한 케이스 포함)
   - "이것이 영업/기술 중 어느 것에 더 가까운지?" → "어느 것이 먼저 필요한지?" 질문 변경
2. **신뢰도 기반 라우팅**:
   - 신뢰도 > 0.8 → 단일 라우팅
   - 신뢰도 < 0.6 → 재분류 또는 양쪽 모두 시도
3. **Fallback 라우팅**: 1차 라우팅 후, "더 도움이 되려면 OO 에이전트도 필요한가?" 자가 판단
4. **사용자 피드백 루프**: 잘못 라우팅된 경우 수집 → 학습 데이터 강화
5. **Hybrid 라우팅**: 한 번에 여러 에이전트 호출하되, 우선순위 다르게 (1차: 영업, 2차: 기술)

## 4) 계층 오케스트레이션 오버헤드: 너무 복잡해짐

**증상:**
- 설계: "Hierarchical 패턴으로 모든 complexity 처리"
- 실제: "Orchestrator가 많은 통신 함 → 지연 50% 이상"
- 또는: "코드가 너무 복잡해서 버그 많음"

**확인:**
- 현재 계층 수: 몇 개?
  - 1개 layer: orchestrator → workers (괜찮음)
  - 2개 layer: orchestrator → sub-orchestrator → workers (복잡해짐)
  - 3개+ layer: 거의 실패 (오버헤드 > 이득)
- 통신 지연: 각 계층 간 통신 시간이 측정되었는가?
- 단순한 패턴으로는 불가능한가? (Sequential 4개 agent = Hierarchical?)

**조치:**
1. **계층 수 감소**: 3-tier → 2-tier 또는 flat (orchestrator + workers)
2. **간단한 패턴으로 롤백**:
   - Hierarchical → Sequential (task A → B → C → D)
   - 또는 → Parallel (A|B, 이후 C|D)
3. **Sub-orchestrator 단순화**: 복잡한 로직 제거, 선택지 줄이기
4. **병렬화**: Sub-orchestrator들이 독립적이면 모두 병렬 실행
5. **"단순함이 가장 강력"**: 복잡할수록 debugging, 유지보수 비용 ↑
