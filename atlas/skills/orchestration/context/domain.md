# 도메인 컨텍스트 — Orchestration

## 1) Domain Scope
에이전트 시스템에서 **2개 이상의 에이전트가 협력할 때** 어떤 패턴으로 조율할지 선택하고 설계하는 영역.
- 포함: Sequential, Parallel, Router, Hierarchical, Event-Driven 패턴
- 제외: 단일 에이전트 프롬프팅, 개별 에이전트 성능 최적화, 모델 선택 (router 스킬)

## 2) Primary Users
- **제품 설계자**: 멀티 에이전트 워크플로우 구조 결정
- **PM**: 복잡한 업무 프로세스를 에이전트 시스템으로 매핑
- **엔지니어**: 실제 구현 전 패턴 검증

## 3) Required Inputs
- 해결할 비즈니스 문제 또는 워크플로우 시나리오
- 각 작업/에이전트의 특성:
  - 작업 수, 의존성 관계
  - 지연시간 요구사항
  - 에러 허용도
  - 비용 제약

## 4) Output Contract
- **패턴 선택**: Sequential / Parallel / Router / Hierarchical 중 하나 + 선택 근거
- **구체적 설계**:
  - 각 에이전트의 역할과 책임
  - 에이전트 간 데이터 흐름 (입출력 형식)
  - 에러 처리 전략 (재시도, 폴백, 중단)
  - 기대 성능지표 (지연시간, 비용)
- **복잡도 검증**: "이것이 필요한 가장 간단한 패턴인가?"

## 5) Guardrails
- 너무 복잡한 패턴 선택 금지: "가장 간단한 패턴부터 시작" 원칙
- 패턴 선택 전에 요구사항 명확화 필수
- 에러 처리 전략 없이 패턴 확정 금지
- 단일 에이전트로 충분한 문제에 오케스트레이션 강요 금지

## 6) Working Facts (TO BE UPDATED by reviewer)
- [ ] 현재 실무에서 가장 많이 사용되는 패턴: Sequential Chain (PM 업무의 약 60-80%, 프로젝트마다 다름)
- [ ] Router 패턴 도입 시 분류 정확도 <90%이면 폴백 전략 필수
- [ ] Hierarchical은 3개 이상 레이어 시 관리 복잡도 급증 (실무 권장: 최대 2단계)
- [ ] Event-Driven은 오버엔지니어링 위험 높음 (초기 사용률 <10%)

> 아래는 특정 프로덕션 환경에서의 사례입니다. 조직과 도메인에 따라 다르게 설계할 수 있습니다.

### 주요 오케스트레이션 패턴 비교

프로덕션에서 사용 가능한 주요 패턴들의 비교:

| 패턴 | 특징 | 사용 시나리오 | 장점 | 고려사항 |
|------|------|-----------|------|---------|
| **Sequential** | 단계별 순차 실행 (데이터 수집 → 분석 → 보고서 생성) | 이전 결과가 다음 입력이 될 때 | 가장 간단하고 예측 가능 | 느린 작업에서 지연시간 증가 |
| **Parallel** | 독립 작업 병렬 실행 후 결과 병합 | 여러 데이터 소스 동시 조회, 다양한 관점 분석 | 지연시간 단축 (예시: 4개 작업 병렬화 시 2-3배 속도 향상) | 작업 간 간섭 가능성, 결과 병합 복잡도 증가 |
| **Router Pattern** | 입력 조건에 따라 다른 에이전트/경로로 분기 | 요청 유형별로 다른 전문 에이전트 필요 | 전문성 증대 (도메인별 최적화) | 정확도 기준(90% 이상 권장) 필수, 폴백 전략 필요 |
| **Hierarchical** | 상위 리더 → 하위 서브에이전트로 위임 | 복잡한 작업을 역할별로 분산 | 책임 분리, 확장성 우수 | 3개 이상 레이어 시 관리 복잡도 급증 (권장: 최대 2단계) |

**선택 기준**:
- "가장 간단한 패턴부터 시작" 원칙 권장
- 요구사항 (의존성, 지연시간, 비용)에 따라 점진적으로 복잡화
- Hierarchical은 확장성이 필요할 때만 도입 (예: 에이전트 10개 이상)

**아래는 Hierarchical 패턴의 구체적 설계 사례입니다.**

### 리더 에이전트 → 서브에이전트 위임 패턴

프로덕션 LangGraph 시스템은 멀티 에이전트 구조로, **단일 리더 에이전트가 하위 서브에이전트로 위임**하는 Hierarchical 패턴을 채택합니다.

```
리더 에이전트 (의사결정)
    ↓ 위임 (명확한 작업 지시)
┌─────────────────────────────────────┐
├─ 일반 목적 서브에이전트 (General)   │
├─ 셸 스크립트 서브에이전트 (Bash)    │
├─ 커스텀 서브에이전트 (확장 가능)    │
└─────────────────────────────────────┘
    ↑ 결과 수집 & 통합
리더 에이전트 (최종 응답)
```

**핵심: 리더는 직접 실행하지 않음.** 모든 작업은 등록된 서브에이전트에 위임되며, 이를 통해 책임 분리와 관심사의 명확한 구분을 달성합니다.

### 미들웨어 파이프라인 설계 (핵심 원칙)

프로덕션 시스템의 요청/응답 파이프라인 설계 원칙:

**핵심 미들웨어 (필수)**:
- **Clarification**: 사용자 요청의 모호성 해소
- **Memory**: 메모리 검색 및 주입 (컨텍스트 강화)
- **Thread Data**: 스레드별 상태 격리 및 관리

**선택 미들웨어 (필요시)**:
- **Subagent Limit**: 서브에이전트 호출 제한 (무한 루프 방지)
- **Sandbox**: 격리된 환경에서 코드 실행 (보안)

**설계 원칙**:
- 최소 설정: 위 필수 3개만으로도 기본 Super Agent 구축 가능
- 필요에 따라 추가 미들웨어 (Uploads, Summarization 등) 활성화
- 실행 순서는 변경 불가능 (데이터 변환의 일관성 보장)

> **참고**: 위 미들웨어 파이프라인은 LangGraph 구현 예시입니다. Sequential, Parallel, Router, Hierarchical 등 다른 오케스트레이션 프레임워크에서도 유사한 미들웨어 개념(요청 전처리, 메모리 관리, 상태 격리)이 적용됩니다.

### Registry 기반 서브에이전트 동적 발견

프로덕션 시스템은 서브에이전트를 **Registry 패턴** (LangGraph 구현 예시 — 다른 프레임워크에서도 유사한 config-driven 레지스트리 패턴 적용 가능)으로 관리합니다:

```yaml
# config.yaml의 subagents 섹션
subagents:
  general:
    class: "agents.GeneralPurposeAgent"
    description: "일반 목적 작업 (분석, 글쓰기, 계획)"
    timeout: 30  # 초 단위, 기본값 override 가능
    model:
      name: "gpt-4-turbo"
      fallback: "gpt-3.5-turbo"

  bash:
    class: "agents.BashAgent"  # 내장 builtin
    description: "셸 명령어 실행 (파일, 스크립트)"
    timeout: 60
    model: null  # 미사용 (도구 기반)

  custom_analytics:
    class: "skills.custom.AnalyticsAgent"  # gitignored custom/
    description: "고객 분석 전문 에이전트"
    timeout: 120
    model:
      name: "gpt-4"
      fallback: null  # fallback 없음 (전문 작업)
```

**적용 패턴**:
1. **런타임 로드**: 리더 에이전트는 registry를 쿼리하여 사용 가능한 서브에이전트 목록 확인
2. **동적 선택**: 작업 특성에 맞는 서브에이전트 자동 선택 및 인스턴스화
3. **Timeout Override**: 작업별로 timeout 설정 가능 (config에서 명시적 정의)
4. **Model Resolution**: Requested 모델 → Default 모델 → None 순서로 폴백

### 적용 교훈: 오케스트레이션 설계 시 분리해야 할 관심사 3가지

| 관심사 | 설계 원칙 | 프로덕션 구현 (Hierarchical 패턴) |
|------|---------|-------------|
| **위임 관계** | 리더는 위임만, 실행은 금지 | Lead Agent → Subagent 명확 분리 |
| **실행 파이프라인** | 각 단계의 책임을 미들웨어로 명시화 | 10개 미들웨어 체인 (불변 순서) |
| **에이전트 라이프사이클** | Registry로 중앙 관리, 동적 발견 | Config-driven subagent registry |

**참고**: 위 설계 원칙은 Hierarchical 오케스트레이션 패턴에 해당합니다. Sequential, Parallel, Router 패턴에서는 다른 구조를 사용합니다:

**실무 적용 사례**:
- "우리 시스템도 이렇게 복잡하지 않은데?" → 처음부터 최소 구조로 시작하세요. 모든 미들웨어가 필수는 아닙니다. Clarification + Memory + Thread Data만으로도 기본 Super Agent를 만들 수 있습니다.
- "그런데 우리는 에이전트가 5개 이상인데?" → Registry로 관리하면 **2개에서 100개로 확장해도 코드 변경 없음**. Config.yaml만 수정하면 됩니다.
- "타임아웃은 어떻게 정해야 할까?" → 파일 I/O 작업은 60초 이상, 일반 API 호출은 30초, 복잡한 분석은 120초 이상 등 **작업 유형별로 명시적으로 설정**하세요.

아래 MCP 및 CrewAI 사례들은 **패턴 선택 이후** 플랫폼 수준에서 고려하는 아키텍처입니다. 본 스킬의 핵심은 Sequential/Parallel/Router/Hierarchical 중 **어떤 패턴을 선택할 것인가**이며, 이후 선택한 패턴을 어떻게 구현할 것인가는 별도의 고려 사항입니다.

## MCP 기반 오케스트레이션: 프로덕션 실전 사례

### Function Calling → MCP 전환 경험

프로덕션 AI 제품 운영 사례에서 얻은 MCP 도입 실전 경험:

#### 이전 (Function Calling 방식)
```
Claude API 챗봇 초기 설계
    ↓
필요: Notion DB 직접 접근
    ↓
tools=[{
  "name": "notion_query",
  "description": "Notion 쿼리",
  "parameters": {...}
}, {
  "name": "notion_update",
  "description": "Notion 업데이트",
  ...
}, {
  "name": "slack_send",
  "description": "Slack 메시지",
  ...
}]
    ↓
새로운 도구 추가마다 (Figma, GitHub, Jira 등)
    └─ API 클라이언트 코드 직접 수정 필요
    └─ 매번 프롬프트 업데이트
    └─ 팀 규모: 1명 → 3명 → 10명으로 커질수록 관리 난장
```

**문제점:**
- "Notion 연동이 필요하면 → 엔지니어 호출"
- "GitHub도 추가? → 또 코드 수정"
- "Figma는? → 또 코드 수정"
- **각 서비스마다 별도 인증 처리**, 에러 핸들링, 타입 정의... 반복

#### 이후 (MCP 방식)
```
MCP Server 표준 인터페이스
    ├─ Notion MCP 서버 (Anthropic 공식)
    ├─ GitHub MCP 서버 (Anthropic 공식)
    ├─ Figma MCP 서버 (커뮤니티)
    └─ 커스텀 MCP 서버 (프로덕션 환경)

Claude API
    │
    └─ 단일 인터페이스로 모든 도구 접근
       (각 MCP 서버가 상세한 구현 담당)

결과:
- 코드 수정: 0줄
- 프롬프트 업데이트: 거의 없음
- 새 서비스 추가: MCP 서버만 설정 (config.json 수정)
```

**전환의 효과:**
- **개발 속도 10배** (새 서비스 추가 시)
- **팀 규모 확장 용이** (각자 MCP 서버 담당 가능)
- **표준화** (모든 팀이 MCP 프로토콜 이해)

### "AI의 USB": MCP의 본질과 의미

#### USB의 역사 (아날로지)
```
1990년대 이전:
- 프린터 연결 → 병렬 포트 케이블 (프린터마다 다름)
- 스캐너 연결 → SCSI 인터페이스 (또 다름)
- 외장 하드 → IDE, SATA... 각각 다름
결과: 컴퓨터 = "연결 악몽"

USB 표준화 이후 (1996~):
- 모든 주변기기 → USB 단일 인터페이스
- 플러그앤플레이
- 에코시스템 폭발
```

#### MCP의 역사 (AI 관점)
```
2023년 이전 (Function Calling):
- Claude API에 도구 추가 → tools=[] 직접 정의
- OpenAI API → 또 다른 정의
- Anthropic이 아닌 다른 LLM → 또 다른 정의
결과: "각 LLM마다 도구 연결 방식이 다름" = USB 이전의 혼돈

MCP 표준화 (2024~):
- 모든 AI, 모든 도구 → 단일 MCP 인터페이스
- Notion, GitHub, Figma 등 주요 기업이 MCP 서버 공식 개발
- 개발자는 "어떤 LLM인지" 신경 쓸 필요 없음
```

**MCP = "AI의 USB"인 이유:**
1. **표준화**: 모든 AI 모델이 동일 인터페이스 이해
2. **상호 운용성**: LLM 교체해도 도구 재연결 불필요
3. **에코시스템 효과**: Notion, GitHub 같은 거대 기업이 MCP 서버 제공 → 개발자는 직접 API 호출 불필요
4. **진입 장벽 감소**: "MCP만 이해하면" 어떤 AI와도 통합 가능

```
│  └─ 목표: "요구사항 명확히, 일정 평가"
│     ├─ "Feature Set: 로그인, 프로필, 검색 (3개)"
│     └─ "예상 기간: 3주 (충분하다)"
│
├─ Dev 에이전트 (역할: Engineer)
│  ├─ 도구: GitHub 쿼리, 기술 스택 검색, 코드 복잡도 분석
│  └─ 목표: "기술 가능성 평가"
│     ├─ "React 기존 코드 재사용 가능? Yes"
│     └─ "새로운 라이브러리 필요? No"
│
└─ QA 에이전트 (역할: QA Lead)
   ├─ 도구: 테스트 케이스 생성, 회귀 테스트 분석
   └─ 목표: "테스트 범위 및 위험 평가"
       ├─ "테스트 케이스: 50개"
       └─ "위험 지점: API 통합 (주의 필요)"
    ↓
[최종 의사결정]
PM: "고스톱. 3주로 충분하지만 QA에서 API 위험 지적"
Dev: "동의. 그래도 2.5주면 코드 완성 가능"
QA: "추천: 1주 버퍼 추가 → 4주 계획 유지"
    ↓
결론: "4주 내 배포 가능, API 테스트 강화 필수"
```

**CrewAI의 특징:**
- **역할 기반**: "PM은 이 도구를 쓸 수 있고, Dev는 저 도구를 써"
- **협업**: 각 에이전트의 결과를 후속 에이전트가 input으로 사용
- **일관성**: 각 역할이 자신의 관점에서만 판단 (편향 방지)

**MCP vs CrewAI:**
| 항목 | MCP | CrewAI |
|-----|-----|---------|
| **역할** | "도구 표준화" | "에이전트 역할 표준화" |
| **사용처** | 단일 에이전트가 다양한 도구 접근 | 다양한 역할의 에이전트 협력 |
| **개발 난이도** | MCP 서버만 구현하면 됨 | 각 에이전트의 instruction 설계 필요 |
| **확장성** | 도구 추가 용이 | 역할 추가 용이 |

### 적용 교훈: "오케스트레이션에서 MCP가 바꾸는 3가지"

#### 1. 통합 복잡도의 급감
```
이전 (Function Calling):
- Notion 도구 추가 → Notion API 이해 필수
- GitHub 도구 추가 → GitHub API 이해 필수
- Slack 도구 추가 → Slack API 이해 필수
총 3개 도구 = 3개 API 학습 곡선

MCP 이후:
- Notion MCP 서버 설정 → Notion MCP 프로토콜만 알면 됨
- GitHub MCP 서버 설정 → GitHub MCP 프로토콜만 알면 됨
- Slack MCP 서버 설정 → Slack MCP 프로토콜만 알면 됨
총 3개 도구 = 1개 MCP 프로토콜 (반복 학습 불필요)
```

**효과:** 새 엔지니어 온보딩 시간 50% 단축

#### 2. 멀티 에이전트 관리의 단순화
```
이전:
에이전트 A (도구: Notion, GitHub)
에이전트 B (도구: Slack, Figma)
각 에이전트마다 도구 연결 로직 다름
→ 버그 확산 위험 높음

MCP 이후:
모든 에이전트 → 동일 MCP 인터페이스로 도구 접근
→ 버그 1개 고치면 모든 에이전트 이득
```

**효과:** 버그 수정 효율성 3배 향상

#### 3. 에코시스템 효과
```
프로덕션 환경에서만:
- 직접 구현: Notion, GitHub MCP 서버

Anthropic 공식 + 커뮤니티:
- 무료로 제공: Figma, Slack, Google Drive, GitHub 등 20+개
- "우리가 만들 필요 없네" → 개발 리소스 절약

결과: 프로덕션 환경에서 필요한 도구 중 80%가 이미 MCP로 제공됨
```

**실무 교훈:**
> "MCP를 도입하려면, 단순히 '도구 표준화'라고 생각하지 말고, '에코시스템에 편승하는 기회'로 봐야 한다."
