# 나쁜 예제 — Memory Architecture

## 사용자 요청
"우리 고객 상담 에이전트가 기억해야 할 것이 너무 많으니, Vector DB에 모든 상담 기록을 저장하고 매번 유사도 검색해서 주입하면 되지 않을까?"

## 거절 이유
- 메모리 유형을 구분하지 않은 모든-Vector-DB 접근
- Episodic 메모리는 **시간순 검색**이 중요하지만, Vector DB는 유사도만 제공
- 결과적으로 관련성 없는 오래된 기록을 자주 반환 → 검색 효율성 저하
- 기존 메모리와 새 정보의 모순 감지 불가능

## 올바른 라우팅
- **Episodic 메모리**: PostgreSQL (user_id, timestamp 인덱싱) + 최신순 정렬
- **Semantic 메모리**: Vector DB (유사도 검색 필요한 경우만)
- **Procedural**: Markdown 파일 (버전 관리, 사람이 이해 가능)

## 수정 방향
"모든 메모리를 한 곳에 저장하되 검색 방식을 분화하려면?" → 하이브리드 아키텍처 설계
또는
"상담사가 지난 상담을 빨리 찾아야 한다면?" → Episodic 메모리를 PostgreSQL로 재설계
