# Troubleshooting — growth-loop 스킬

## 1) Cold Start 문제: 초기 데이터 부족

**증상:**
- 출시 3개월 후에도 사용량 < 100건
- 데이터 포인트가 너무 적어서 패턴을 찾을 수 없음
- "이 데이터로 개선이 가능할까?"라는 의심

**확인:**
- 월 사용량 추이 확인 (지난 3개월 변화 없는가?)
- 사용자 세그먼트: 활동적 사용자 vs 비활동 사용자 비율
- 데이터 수집 방법: 명시적(explicit) 피드백이 제대로 작동하는가?

**조치:**
1. Seed data 전략 도입: 수동으로 초기 데이터 100개 이상 주입 (전문가 또는 템플릿)
2. Transfer learning 활용: 유사 도메인에서 학습된 모델 또는 데이터 세트 활용
3. TK injection: PM 또는 도메인 전문가가 초기 품질 확보를 위해 프롬프트에 암묵지 주입
4. 사용자 인센티브: 초기 피드백 제공자에게 보상 (discount, early access 등)
5. Cohort 분석: 특정 사용자 그룹(예: Beta user)의 데이터부터 시작해 빠른 루프 형성

## 2) Anti-Loop 발생: 데이터 수집 후 개선이 안 됨

**증상:**
- 3개월 동안 데이터는 쌓였는데 정확도가 같다
- 에이전트 성능이 정체 상태 (improvement plateau)
- 사용자가 "별로 나아진 게 없는데?"라고 느낌

**확인:**
- 데이터 품질: 수집된 데이터에 노이즈가 많진 않은가? (레이블 오류, 중복)
- 개선 방법: 프롬프트 최적화인가 파인튜닝인가? 올바른 방법을 썼는가?
- 발생 원인 진단: 다음 중 어느 것인가?
  - Data Decay: 시간이 지나면서 데이터가 outdated 되었나? (예: 계절성, 트렌드 변화)
  - Privacy Barrier: 고객이 실제로는 데이터 공유를 꺼렸나?
  - Cost Escalation: 데이터 처리 비용이 예상치를 초과했나?
  - Quality Ceiling: 특정 수준 이상 개선이 물리적으로 불가능한가?

**조치:**
1. **Data Quality 강화**: 데이터 정제 파이프라인 추가 (noise 제거, 중복 제거)
2. **개선 방법 변경**: 프롬프트 → 파인튜닝 또는 RAG 업데이트로 전환
3. **필요 데이터 재정의**: "무엇이 문제를 푸는 데 필수인가?" 다시 질문 (예: 더 많은 예제 필요? 메타데이터 필요?)
4. **사이클 가속화**: 주 1회 → 주 2~3회로 업데이트 빈도 증가 (단, 비용 대비)
5. **사용자에게 투명성 제공**: "지난달 대비 정확도 ↑ N%"라는 피드백으로 심리적 개선감 제공

## 3) Privacy 장벽: 고객이 데이터 수집 거부

**증상:**
- 고객이 명시적으로 "우리 데이터를 써서 학습하지 마"라고 말함
- 혹은 계약에 "고객 데이터는 우리만 사용 가능" 조항 포함
- 데이터 모아 전략을 완전히 차단당함

**확인:**
- 계약서 검토: 데이터 사용 권리가 명시되어 있는가?
- 고객 우려 파악: "왜 우리 데이터를 쓰면 안 된다고 생각하는가?" (보안/경쟁 이유 등)
- 현재 상태: Seed data만으로는 충분한가? (고객 특화 데이터 없이 generic만 가능)

**조치:**
1. **명시적 동의 UI 추가**: "이 피드백을 제품 개선에 사용할까요? (개인 식별 정보 제거)"라는 opt-in 추가
2. **익명화 정책 강화**: 모든 고객 데이터를 익명화 후 저장 → 고객이 안심
3. **차등 가격 모델**: "데이터 공유 고객 = 할인" 인센티브 제공
4. **고객 갭 수용 및 보상**: 이 고객은 generic moat만 가능하다고 받아들이고, 다른 moat(workflow lock-in, proprietary knowledge)로 보상
5. **계약 재협상**: "학습 목적 데이터는 공유하되, 다른 고객에게 판매하지 않는다"는 제한 조항으로 협상

## 4) 플라이휠 주기가 너무 김: 사용자가 개선 체감 못함

**증상:**
- 데이터 수집 → 프롬프트 업데이트까지 4주 이상 소요
- 사용자가 "아, 더 좋아졌다"는 느낌이 없음
- 피드백 제공 동기 없음 → 반복 사용 저하

**확인:**
- 현재 사이클: 데이터 수집(?) → 분석(?) → 업데이트(?) 각 단계 기간 측정
- 병목 지점: 어디서 가장 오래 걸리는가?
  - 데이터 수집? (사용량 낮음)
  - 데이터 처리? (배치 시스템 느림)
  - 모델 평가? (검증 절차 복잡)
  - 배포? (CI/CD 느림)

**조치:**
1. **배치 처리 속도 개선**: 주 1회 → 주 2~3회 또는 일일 업데이트로 가속
2. **자동화된 평가 지표 도입**: 수동 평가 대신 자동 메트릭(정확도, 만족도 스코어) 사용
3. **사용자에게 개선 피드백 보여주기**: 매주 업데이트 내역을 푸시 또는 인앱 알림 ("이번주 개선사항 3가지")
4. **병렬 처리 고도화**: 각 단계를 순차 → 병렬로 변경 (예: A/B 테스트와 수집 동시 진행)
5. **Shadow mode 활용**: 사용자 선택 전에 개선 사항을 "제안만" 하고 채택 결정권을 사용자에게

## 5) Loop Type 선택 실수: 수익 목표와 맞지 않는 루프 설계

**증상:**
- Type B (Content Loop)인데 Type A (Data Quality)로 설계함
- 콘텐츠 생성이 핵심인데, 데이터 개선에만 초점
- 실제 성장 목표(트래픽 증가 또는 고객 확보)에 도움이 안 됨

**확인:**
- 가장 큰 성장 드라이버가 무엇인가?
  - 더 나은 데이터/모델 성능? → Type A
  - 생성 콘텐츠가 새 사용자 유입을 만드는가? → Type B
  - 멀티 에이전트 협업으로 가치 증가? → Type C
  - PM 경험 축적이 에이전트 판단 개선? → Type D
- 현재 병목은 무엇인가? (성능 부족인가, 사용자 유입인가, 복잡도인가?)

**조치:**
1. **Primary Loop 재정의**: 수익 임팩트 크기 순으로 Loop Type 우선순위 재정렬
2. **Secondary Loop 추가**: Type A가 약하면 Type B 또는 Type C로 보완 (단일 moat보다 다중 moat)
3. **리소스 재배분**: Type B가 더 중요하다면, 데이터 개선팀보다 콘텐츠 생성팀에 인력 집중
4. **성공 지표 변경**: "정확도"에서 "신규 사용자 수" 또는 "사용 시간"으로 KPI 전환
5. **3개월 재검토**: 선택한 Type으로 3개월 운영 후, 실제 impact를 측정하여 Type 변경 재검토
