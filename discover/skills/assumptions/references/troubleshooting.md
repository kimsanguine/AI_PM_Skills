# Troubleshooting — assumptions

## 1) 모든 가정이 "위험도 1~2점"으로 평가됨

**증상:**
- 팀이 에이전트를 과도하게 긍정적으로 평가
- "우리 기술 스택이 이미 검증되었으니까 리스크가 낮다"는 태도
- 점수가 대부분 1~2점 범위에 집중

**확인:**
- 점수 분포가 1~5 범위에 고르게 분포하는지 검증
- "이 가정이 틀렸을 가능성"을 적극적으로 물어보기
- 부정적 관점의 가정도 발굴했는지 확인

**조치:**
- 악의적 질문 추가: "이 에이전트가 완전히 망할 확률은 얼마나 될까?"
- 반대편 입장에서 가정 재검토: "경쟁팀이 이 에이전트에 문제점을 지적한다면?"
- 과거 유사 프로젝트의 실패 사례 검토

---

## 2) Top 3 가정이 모두 Feasibility인데 Value/Ethics가 없음

**증상:**
- "API가 있는가?", "권한이 있는가?", "응답 속도는?" 등 기술 검증만
- "정말 필요한가?", "고객이 신뢰할까?" 같은 비즈니스/윤리 질문 누락
- Value 축 가정 개수가 3개 미만

**확인:**
- Value 축 가정이 제대로 발굴되었는지 확인
- "누가 이 에이전트를 쓸 것인가?", "실제로 시간이 단축되는가?" 같은 질문 여부
- Ethics 축 가정이 최소 3개 있는지 확인

**조치:**
- Value 축 재검토 강제: 최소 3개의 비즈니스 가정 필수
- 사용자 인터뷰 또는 5일 수동 실행 실험 추가
- Ethics 축 가정도 최소 3개 발굴 (특히 자동화 영향이 큰 경우)

---

## 3) 검증 실험 결과 Top 1 가정이 틀렸음

**증상:**
- 실험 실행 후 "우리 예상이 틀렸다" 판명
- 예: "API 권한이 없다", "데이터가 없다", "정확도가 50% 수준"
- 이미 1~2주 검증 시간 소비

**확인:**
- 실패 가정이 정말 "Top 1" 리스크였는지 확인 (점수 재평가)
- 다른 Top 2, 3 가정들도 실제로 검증되었는지 확인

**조치:**
- 즉시 피벗 옵션 검토: "다른 데이터소스를 쓸 수 있을까?", "다른 API가 있을까?"
- 해당 가정 기반 재설계 고려
- 근본적으로 부정할 가능성이 높으면 opp-tree로 돌아가 기회 재탐색

---

## 4) Ethics 위험을 감지했는데 팀이 "괜찮을 거야" 무시

**증상:**
- assumptions 스킬에서 "위험도 4~5점" Ethics 항목 발견
- 팀: "이정도면 괜찮지 않을까? 나중에 처리하자"
- 검증 미루고 구현 진행

**확인:**
- "위험도 4점 이상 가정"이 정말 식별되었는지 재확인
- 팀의 위험 회피 태도 또는 이해 부족 여부

**조치:**
- "위험도 4점 이상 Ethics 항목은 hitl 스킬 검토 강제" 규칙 적용
- 회의에서 위험도 재평가 및 서명 기록 남기기
- 구현 이전에 hitl 스킬로 Human-in-the-Loop 설계 완료 필수
