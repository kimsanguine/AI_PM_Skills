# Domain Context — assumptions

## 1) Domain Scope

Agent Assumption Analysis — 에이전트 구현 전 숨겨진 가정을 발굴하고 위험도를 수치화하는 도메인

**적용 범위:**
- 신규 에이전트 아이디어 (구현 결정 직후)
- 다중 에이전트 오케스트레이션 (개별 에이전트의 가정 검증)
- 제3자(고객) 관점의 에이전트 도입 평가

**제외 범위:**
- 이미 배포된 에이전트의 성능 개선 (별도 스킬)
- 기술 스택 선택 (build-or-buy 스킬)
- 비즈니스 기회 발굴 (opp-tree 스킬)

---

## 2) Primary Users

- **PM / Product Manager**: 에이전트 기회를 선택 후 구현 리스크 평가
- **Tech Lead / Engineer**: 기술 가능성과 실제 운영 가능성의 갭 파악
- **비즈니스 팀**: "정말 ROI가 있을까?"에 대한 데이터 기반 판단
- **Security / Compliance**: Ethics 축 리스크 평가 (고객 영향, 법적 책임)

---

## 3) Required Inputs

- **에이전트 아이디어 / 기회명**: "뉴스 브리핑 에이전트", "고객 문의 자동 분류" 등
- **Automation Outcome**: 이 에이전트가 달성하려는 목표 (예: "PM 뉴스 수집 시간 주 8시간 → 0.5시간")
- **기술 스택 (선택사항)**: 사용할 모델, 외부 API, 데이터 소스
- **팀의 현재 우려사항**: "정말 정확할까?", "고객이 받아들일까?" 등

---

## 4) Output Contract

**Step-by-Step Deliverables:**

1. **4축 Assumption Map** (최소 12개 가정: 각 축 3개 이상)
   - Value axis (3~5개)
   - Feasibility axis (3~5개)
   - Reliability axis (3~5개)
   - Ethics axis (3~5개)

2. **Priority Matrix** (위험도 × 검증 난이도)
   - 상위 5개 가정 점수 계산 및 정렬

3. **Top 3 심층 분석** (각 100~150 words)
   - 가정이 틀렸을 때 시나리오
   - 피벗 옵션 (3가지 이상)
   - 검증 실험 설계 (2일 이내 가능)

4. **Ethics Risk Assessment**
   - 위험도 4점 이상 항목 강조
   - Human-in-the-Loop 필요 여부 판단

5. **다음 단계 권장**
   - 검증 실험 통과 → agent-instruction-design
   - Ethics 위험 높음 → hitl 스킬
   - 검증 실패 → opp-tree로 돌아가 기회 재탐색

---

## 5) Guardrails

**필수 체크사항:**

- ✓ 4축 각각에 구체적인 가정이 3개 이상 있는가?
  - 일반적인 가정("에이전트가 작동할 것이다") → 거절
  - 구체적 가정("LLM Haiku 모델이 도메인 용어를 70% 이상 정확히 인식할 것이다") → 승인

- ✓ 위험도와 검증 난이도 점수가 합리적인가?
  - 모든 가정이 점수 5점 → 점수 재평가 필요 (팀의 낙관 바이어스 의심)
  - 점수 분포가 1~5 범위에 고르게 분포 → 양호

- ✓ 검증 실험이 2일 이내에 실행 가능한가?
  - "시장에 출시해서 피드백을 받자" → 거절 (너무 느림)
  - "5명의 사용자 인터뷰 + API 테스트 10회" → 승인

- ✓ Ethics 위험도가 4점 이상인 항목이 발견되었는가?
  - 발견됨 → 반드시 hitl 스킬 검토 강제
  - 미발견 → "정말 놓친 게 없는가?" 재확인

## 6) Working Facts

**에이전트 가정 분석의 일반적 특성:**

| 항목 | 설명 | 참고 |
|------|------|------|
| **평균 가정 개수** | 에이전트당 12~20개 | 단순 에이전트: 8~12개 / 복잡한 에이전트: 15~25개 |
| **우선순위 상위** | 점수 상위 5개만 심층 분석 | 하위 항목은 "모니터링" 상태로 유지 |
| **검증 리드타임** | 평균 2~5일 | 실험 복잡도에 따라 가변 |
| **Value 축** | 가장 자주 무시됨 | 기술팀이 Feasibility만 검토하고 Value 누락하는 경향 |
| **Ethics 축** | 배포 후 문제 발생 가능성 높음 | 사전 검토 필수 (사후 대응 비용 훨씬 높음) |

**TO BE UPDATED by reviewer:**
- 우리 조직의 일반적 가정 검증 기간 (현재: 2일 기준, 조직마다 다를 수 있음)
- 에이전트 정확도 기준 (도메인마다 상이: 뉴스 요약 90% vs 금융 99.9%)
- Ethics 위험 평가 체크리스트 (산업/규제 환경별 상이)

## 7) Fill-in Checklist

구현 팀이 assumptions 스킬 실행 후 다음을 확인:

- [ ] 4축 가정이 각각 3개 이상 구체적으로 정의되었는가?
- [ ] 점수 계산 근거(위험도, 난이도)가 명시되었는가?
- [ ] Top 3 가정의 "틀렸을 시나리오" + "피벗 옵션"이 서술되었는가?
- [ ] 검증 실험이 2일 이내에 가능한 수준인가?
- [ ] Ethics 위험도 4점 이상 항목이 발견되었고, hitl 스킬 검토가 예정되었는가?
- [ ] 다음 단계 (instruction-design / hitl / 기회 재탐색) 가 명확한가?
