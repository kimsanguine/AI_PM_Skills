# Troubleshooting — opp-tree 스킬

## 1) Opportunity Score가 모두 높을 때 (평가 오류 의심)

**증상:**
- "계산한 모든 기회의 점수가 120점 이상이에요."
- "기회별 점수 차이가 거의 없어요."
- "우선순위를 정할 수 없습니다."

**확인:**
- 반복빈도 점수: 모든 기회가 5점인가?
- 자동화가능성 점수: 모든 기회가 5점인가?
- 판단의존도 역수: 모든 기회가 5점(저판단)인가?

**조치:**
1. 점수 재검토: 각 항목을 더 엄격하게 평가
   ```
   // Before (너무 관대)
   기회 A: 반복빈도 5 × 자동화 5 × 판단역수 5 = 125

   // After (재평가)
   기회 A: 반복빈도 4 (주 5회가 아니라 월 2회) × 자동화 3 (API 제한) × 판단역수 3 (판단 필요) = 36
   ```
2. 특히 "판단 의존도" 재평가:
   - 판단이 필요하면 → 역수를 3~4로 낮춤
   - 판단이 최소 → 역수를 5로 유지
3. 팀 리뷰: "정말 자동화 가능한가?" 재논의
4. 점수 상위 3개만 선택

## 2) 선택한 Opportunity가 실제로 반복되지 않을 때

**증상:**
- "2주 실험 결과 선택한 작업은 일주일에 1번도 안 반복돼요."
- "반복빈도 점수를 너무 높게 잡았네요."
- "이 기회는 일단 제외하고 다른 걸로 진행할까요?"

**확인:**
- 실제 관찰 기간: 얼마나 측정했는가? (1주 vs 2주 vs 1달)
- 반복 패턴: 정말 반복되지 않는가 vs 계절성/주기성이 있는가?

**조치:**
1. 해당 Opportunity 제외: 점수 부정확 → 선택 제외
2. 다음 순위 Opportunity로 즉시 전환 (재점수 불필요)
3. 점수표 갱신: "반복빈도" 항목 수정
   ```
   기회 A: 반복빈도 2 (월 1회) → 점수 재계산
   ```
4. 배운 교훈: "반복 패턴 관찰 최소 2주 필요" 문서화

## 3) 필요한 API/도구가 없거나 비용이 과도할 때

**증상:**
- "API에 접근 권한이 없어요."
- "필요한 API 비용이 월 $5k 이상이라 경제성이 없어요."
- "기술적으로 구현이 불가능한 것 같아요."

**확인:**
- Feasibility 검증: Step 4 실험에서 API 접근성 테스트했는가?
- 비용 추정: 실제 구현 비용과 자동화 절감 비용을 비교했는가?

**조치:**
1. Feasibility 점수 재평가 (낮게):
   ```
   원래 Opportunity Score: 100점
   Feasibility 재평가: 1점 (불가능)
   → 해당 기회 제외
   ```
2. 대체 기술/API 탐색:
   - 더 저렴한 대안이 있는가?
   - 다른 데이터 소스로 대체 가능한가?
3. 비용 vs 절감 분석:
   - API 비용 $5k/월 > 절감액 $1k/월 = 부정적 ROI → 제외
4. 다음 순위 Opportunity로 전환

## 4) Agent Type(Trigger vs Pipeline vs Research) 선택이 모호할 때

**증상:**
- "이 기회를 Trigger Agent로 할지 Pipeline Agent로 할지 결정 못 했어요."
- "어떤 Agent Type이 가장 효율적일지 모르겠어요."
- "3가지 유형 모두 가능해 보이는데요."

**확인:**
- 워크플로우 특성: 단일 작업인가 vs 다단계인가?
- 입력 명확도: 입력이 정확히 정의되는가 vs 동적인가?
- 의존성: 외부 데이터 조회가 필요한가?

**조치:**
1. Agent Type 선택 가이드:
   ```
   Trigger Agent:
   - 명확한 이벤트 트리거 (이메일 수신, Slack 메시지)
   - 단일 작업 실행 (분류, 요약)
   - 판단 최소
   예: "이메일 분류" → Trigger Agent

   Pipeline Agent:
   - 정해진 다단계 워크플로우
   - 각 단계가 순차적
   예: "요청 수신 → 검색 → 응답 생성 → 저장" → Pipeline Agent

   Research Agent:
   - 탐색적 데이터 수집
   - 동적 범위의 정보 탐색
   예: "경쟁사 정보 수집" → Research Agent
   ```
2. Step 3에서 최소 2가지 Agent Type 제안 후 팀이 선택
3. 확실하지 않으면: 구현 난이도 낮은 것부터 시작

## 5) Ethics 위험을 실험 단계에서 미감지한 경우

**증상:**
- "자동화 기회가 좋아 보였는데 배포 후 윤리 이슈가 제기되었어요."
- "고객 데이터를 자동으로 삭제하는 기능인데 이게 자동화되면 안 된다고 해요."
- "어떻게 처음부터 감지할 수 있을까요?"

**확인:**
- Step 4 실험에서 Ethics 축을 평가했는가?
- 팀 리뷰(법무, 보안, 비즈니스)를 거쳤는가?
- 자동화 시 발생 가능한 피해를 충분히 고려했는가?

**조치:**
1. 즉시: hitl 스킬 연계 → HITL 재설계
   - 자동화 레벨 하향 (Level 5 → Level 2)
   - Approval Gate 또는 Escalation Chain 추가
2. 다음 AOT부터:
   - Step 4에 "Ethics 축 팀 리뷰 미팅" 추가
   - 법무/보안/PO 참석 필수
   - 체크리스트: "데이터 삭제?", "고객 대면?", "법적 영향?", "평판 위험?"
3. 배포 권장사항 문서화
