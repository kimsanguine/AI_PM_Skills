# Domain Context — hitl

## 1) Domain Scope

Human-in-the-Loop Design — 에이전트의 각 작업마다 자동화 레벨을 결정하고, 인간 개입 지점과 트리거 조건을 설계하는 도메인

**적용 범위:**
- 의사결정, 송금, 고객 대면, 데이터 삭제 등 중요한 작업의 HITL 설계
- 초기 배포 또는 새로운 에이전트의 신뢰 구축 전략
- 에이전트 오류의 영향도가 높은 경우
- Approval Gate, Confidence Threshold, Shadow Mode 등 패턴 선택

**제외 범위:**
- 이미 배포된 에이전트의 성능 개선 (별도 스킬)
- 기술적 구현 (instruction-design 스킬)
- 제품 신뢰 구축 전략만 필요한 경우 (agent-gtm 스킬)

---

## 2) Primary Users

- **PM / Product Manager**: 에이전트의 신뢰성 및 안전성 확보
- **Security / Compliance**: 법적/규제 리스크 관리
- **Tech Lead**: 에이전트 아키텍처에 HITL 패턴 구현
- **Customer Success**: 초기 고객 온보딩 시 신뢰 구축

---

## 3) Required Inputs

- **에이전트 워크플로우 또는 이름**: 구체적 에이전트 정의
- **수행하는 작업 목록**: 에이전트가 하는 모든 작업
- **각 작업의 중요도**: 오류 발생 시 영향 범위
- **운영 환경 제약**: 인간 개입 시간, 채널 등

---

## 4) Output Contract

1. **작업 목록** (5~10개)
2. **2축 평가** (가역성 × 오류 영향도)
3. **자동화 레벨** (각 작업마다 1~5)
4. **HITL 패턴** (Approval Gate 등)
5. **개입 트리거** (신뢰도, 에러 카운트 등)
6. **Shadow Mode 계획** (기간, 전환 기준)
7. **설계 체크리스트** (모든 항목 완료 확인)

---

## 5) Guardrails

- ✓ 모든 작업이 목록화되었고, 각각 자동화 레벨이 할당되었는가?
- ✓ Level 5 (완전 자동)는 오류 영향도가 극히 낮은 경우만인가?
- ✓ HITL 패턴이 구체적으로 선택되었는가? ("사람이 확인한다" X)
- ✓ 각 개입의 트리거 조건이 정량적인가? (신뢰도 80% 이상, 에러 > 3회/일 등)
- ✓ Shadow Mode 기간과 전환 기준이 명확한가?

---

## 6) Working Facts

**자동화 레벨 일반적 특성:**

| 작업 유형 | 일반적 레벨 | 이유 |
|----------|-----------|------|
| 정보 수집 | 4~5 | 오류 영향도 낮음 |
| 분류/라우팅 | 3~4 | 신뢰도 기반 threshold |
| 초안 생성 | 2~3 | Approval Gate 필수 |
| 송금/결제 | 1~2 | 인간 승인 필수 |
| 데이터 삭제 | 1 | 수동만 가능 |

## 8) 참고 사례: Linear Quality Gate의 HITL 패턴

> 아래는 특정 프로덕션 환경에서의 사례입니다. 조직과 도메인에 따라 다르게 설계할 수 있습니다.

### HITL 설계의 핵심: Quality Gate

**Quality Gate는 "에이전트의 산출물이 정말 실행 가능한가?"를 검증하는 자동 또는 인간 중심 게이트입니다.**

Linear 환경에서 Task 생성, Team 권한 변경, 상태 전이 등 중요한 작업을 에이전트가 수행할 때, 이 게이트를 통과해야만 실제 작업이 시작됩니다.

---

### Quality Gate 설계 원칙

#### 1) 승인 게이트: Requester ≠ Approver 원칙

```
에이전트 요청 → Preflight 검증 → 게이트 판정
  ├─ PASS: 자동 실행 (위험도 낮음)
  └─ BLOCK: 인간 승인 필요
      └─ 승인자 선택 규칙:
          • 요청자(에이전트 운영자) ≠ 승인자(독립적 검증자)
          • 승인자는 해당 작업의 비즈니스 영향도를 이해하는 자
          • 예: Task 생성 → PM 승인, 권한 변경 → Admin 승인
```

**왜 이 원칙?**
- 에이전트(또는 운영자) 자신이 승인하면, 사실상 게이트가 무의미
- 독립적 검증자의 "잠깐, 이게 맞나?" 질문이 오류 발견의 핵심

---

#### 2) Tier별 SLA 차등화

**작업의 중요도에 따라 승인 시간을 차등 적용:**

| Tier | 작업 유형 | 승인 SLA | 담당자 | 예시 |
|---|---|---|---|---|
| **A (Critical)** | 데이터 삭제, 권한 변경, 결제 승인 | 24시간 (예시값, 조직별 조정 가능) | C-Level / Tech Lead | Task 전체 삭제, Private Team 접근 제거 |
| **B (High)** | Task 중대 상태 변경, 자동화 규칙 수정 | 48시간 (예시값, 조직별 조정 가능) | PM / Team Lead | 완료된 Task를 다시 In Progress로 |
| **C (Medium)** | Task 생성, 일반 상태 변경, 필드 업데이트 | 72시간 (예시값, 조직별 조정 가능) | Stakeholder / 팀원 | 신규 Task 생성, Priority 변경 |

**SLA의 의미:**
- 승인 요청이 들어오면, 명시된 시간 내에 승인/거절 결정 필요
- 초과 시 → 자동 에스컬레이션 (다음 단계 참조)

---

#### 3) 자동 에스컬레이션: SLA 초과 시 상위 승인자 이관

```
승인 요청 (Tier A, 24시간 SLA)
  ├─ 24시간 내 승인 → ALLOW / BLOCK
  └─ 24시간 경과 && 응답 없음
      └─ 자동 에스컬레이션:
          • 원래 승인자: Team Lead
          • 상위 승인자로 이관: VP / 경영진
          • 알림: "24시간 경과, 상위 승인자에게 이관됨"
          • 패턴: 자동화를 통해 "절차 지연"을 방지
```

**효과:**
- 승인자가 응답하지 않아도, 프로세스가 멈추지 않음
- 중요도 높은 작업일수록, 더 빠르게 결정자에게 올라감
- 역할 추적: "누가 최종 승인했나?" 기록됨

---

#### 4) Bypass 요청도 HITL로: 요청 → 승인 → TTL → 만료 → Remediation

**에이전트가 "이 작업은 Game Gating 스킵해도 돼?"라고 요청하는 경우:**

```
Bypass 요청 (예: "Rate Limit 무시하고 Task 일괄 생성")
  ├─ 요청자 + 사유 + 기간 명시 (예: "3시간 동안만")
  ├─ HITL 게이트 1단계: 관리자 검증
  │  └─ "이 요청이 정말 필요한가?" 확인
  ├─ 승인 후, TTL(Time-To-Live) 설정
  │  └─ "이 bypass는 3시간 동안만 유효함"
  ├─ 3시간 경과 후 자동 만료
  │  └─ Rate Limit gate 다시 활성화
  └─ 만료 후 "이전에 bypass한 작업들, 이제 정상 게이트 통과?"
      └─ Remediation: 누적된 작업 재검증
```

**Bypass HITL의 핵심:**
- "이것도 게이트?"라고 생각할 수 있지만, 가장 중요함
- Bypass 자체가 비정상 신호 → 왜 필요했는지 기록
- TTL 만료 후, "아, 그때 왜 bypass가 필요했더라?" 후속 분석

---

### 다른 도메인 적용 예시

위의 Linear 사례가 task-management 중심이므로, 다른 도메인에서의 HITL 설계 예시:

- **고객 서비스 챗봇**: 환불/결제 관련 응답은 Level 2 (Approval Gate), 일반 FAQ는 Level 4 (Act-and-Escalate)
- **콘텐츠 모더레이션**: 민감 콘텐츠 판단은 Level 3 (Confidence Threshold 85%), 스팸 분류는 Level 5 (자동)
- **금융 거래 에이전트**: 고액 거래(>$10K)는 Level 1 (Manual), 소액 조회는 Level 5 (자동)

---

### 적용 교훈: AI 에이전트에서 HITL을 설계할 때 빠지기 쉬운 3가지 함정

#### 함정 #1: "Approval Gate는 선택"이라고 생각

**위험한 생각:**
> "에이전트 신뢰도가 95%니까, Approval Gate는 거의 필요 없겠지?"

**현실:**
- 에이전트 정확도 95% = 100건 중 5건은 틀림
- 틀린 5건이 "Task 전체 삭제" 같은 중요 작업이면? → SEV-1 사건
- 또한, 에이전트가 항상 정확하더라도, **운영자의 지시가 잘못될 수 있음**

**올바른 설계:**
- Approval Gate는 "에이전트 신뢰도"가 아니라 **"작업의 비용"**에 기반
- 비용(돌이킬 수 없는 행동) ↑ → Gate 엄격함 ↑

#### 함정 #2: "SLA 없이 '빨리 처리하세요'"

**위험한 생각:**
> "승인자가 바쁘더라도, 언젠가는 승인하겠지?"

**현실:**
- 승인 요청이 며칠 동안 미처리 → 에이전트 작업 계속 대기
- 비즈니스 임팩트: "우리 Task 자동화가 작동 안 해?" → 신뢰도 추락

**올바른 설계:**
- Tier A/B/C별 명확한 SLA (24/48/72시간)
- SLA 초과 시 **자동 에스컬레이션**
- "SLA"는 프로세스를 '빠르게' 하는 게 아니라, '일정하게' 하는 것

#### 함정 #3: "Bypass는 없어야 한다" (또는 "너무 흔함")

**위험한 생각:**
1. "게이트를 우회할 수 없게 구현하자" (게이트 과도)
2. "이 요청은 신뢰할 수 있으니 bypass 허용" (우회 부실)

**현실:**
- 응급 상황(예: 고객 서비스 중단)에는 bypass가 필수
- 하지만 bypass 자체가 비정상 신호 → 기록 + 승인 필수
- Bypass 후 정상화: TTL 만료 후 누적 작업 재검증

**올바른 설계:**
- Bypass **요청도 HITL**로 (관리자 승인 필수)
- TTL(시간 제한) 설정 (예: 1시간, 24시간)
- TTL 만료 후 자동 remediation (이전 bypass 작업들 정상 게이트 통과 확인)

---

### 결론: HITL 설계의 실천 체크리스트

HITL을 설계할 때, 다음 3가지 함정을 피하고 다음을 보장하세요:

- ✓ **모든 작업에 비용(가역성) 평가** → 비용 높을수록 gate 엄격
- ✓ **SLA는 명확하고, 초과 시 자동 에스컬레이션** → 프로세스 정체 방지
- ✓ **Bypass도 HITL로 관리** (TTL + 기록 + remediation) → 비정상 감지

## 교육자 관점: HITL 패턴의 실무 적용

> **참고**: 이 섹션은 HITL의 이론적 배경입니다. 실무 적용은 위의 사례들을 참고하세요.

### Agent 설계 패턴 5가지 중 HITL의 위치

에이전트 아키텍처에서 **5가지 핵심 패턴**이 있습니다. HITL은 이 중 가장 보수적이면서도 신뢰 구축에 필수적인 패턴입니다:

```
1. ReAct (Reasoning + Acting)
   └─ 단일 에이전트: 도구를 선택하고 실행

2. Supervisor (라우터)
   └─ 여러 에이전트 중 하나 선택 후 실행

3. Conditional Routing (조건부 분기)
   └─ 입력/상태에 따라 다른 워크플로우 선택

4. HITL (Human-in-the-Loop) ← 여기
   └─ 에이전트 결정 후 "이대로 진행할까?" 물어보기

5. Parallel (병렬 처리)
   └─ 여러 에이전트 동시 실행 후 통합
```

**HITL이 필요한 이유:** 신뢰받는 에이전트는 "내가 내 결정을 완전히 확신하지 못하는 상황에서 사람에게 물어본다"는 능력을 보여줘야 합니다.

---

### Supervisor → 조건부 분기 → HITL 노드 패턴

에이전트 설계 실무에서 다루는 **조건부 분기 + HITL 조합 패턴**:

```
[사용자 요청]
    ↓
[Supervisor 에이전트]
    (요청 분류: "Task 생성", "권한 변경", "데이터 삭제" 등)
    ↓
[조건부 분기]
    ├─ 위험도 낮음 (정보 조회)
    │  └─ 즉시 실행
    │
    ├─ 위험도 중간 (Task 생성, 상태 변경)
    │  └─ [HITL 노드 1]
    │     └─ "이 Task 생성 요청, 승인할까요?"
    │        ├─ 승인 → 계속 실행
    │        ├─ 거부 → 사용자에게 사유 알림
    │        └─ 수정 → 파라미터 변경 후 재실행
    │
    └─ 위험도 높음 (권한 변경, 삭제)
       └─ [HITL 노드 2] (더 엄격)
          └─ "이 작업은 매우 중요합니다. 최고 관리자 승인이 필요합니다."
             (위와 동일한 3가지 경로)
```

**이 패턴의 장점:**
- 위험도에 따라 **차등화된 검증** (모든 작업에 동일 레벨이 아님)
- 승인 여부뿐만 아니라 **거부와 수정**의 피드백 루프 제공
- 에이전트가 "아, 사람이 이렇게 판단했구나" 학습 가능

---

### 핵심 원칙: "Agent가 신뢰받으려면, 위험한 작업에서 반드시 물어봐야 한다"

**신뢰 구축의 심리학:**

| 에이전트 행동 | 사용자 반응 | 신뢰도 |
|----------|---------|--------|
| "이 Task를 생성했습니다" | "아, 내가 승인한 것도 아닌데?" | ⬇ 낮음 |
| "이 Task 생성을 제안합니다. 하시겠습니까?" | "아, 물어봤네. 좋다" | ⬆ 높음 |
| "이 Task 생성은 너무 위험해서 못 합니다. 관리자만 가능합니다" | "오, 자신이 없으면 말하는군" | ⬆⬆ 매우 높음 |

**프로 팁:**
> "에이전트가 95% 정확하면, 뭔가 모자란 게 있다는 뜻이다. 100% 자신할 수 없으면 물어보는 게 훨씬 낫다."

---

### 3가지 경로: 승인 / 거부 / 수정의 설계

HITL 노드에서 사용자는 3가지 선택지를 가집니다:

#### 1. 승인 (Approve)
```
HITL 게이트 → 사용자 "OK" → 에이전트 계속 실행 → 작업 완료
비용: 사용자의 1회 클릭 (2초)
```

#### 2. 거부 (Reject)
```
HITL 게이트 → 사용자 "No" + 사유 입력 → 에이전트 재실행 또는 중단
비용: 사용자 피드백 (30초~1분)
효과: 에이전트가 "아, 이 조건에서는 안 되네" 학습
```

#### 3. 수정 (Modify)
```
HITL 게이트 → 사용자 파라미터 수정 (예: Task 우선도 변경)
           → 에이전트 재실행 (수정된 파라미터로) → 승인 대기
비용: 사용자의 변경 입력 (1~3분)
효과: 에이전트와 사용자의 "협업" 경험
```

**실무 적용 팁:**
- "단순 승인/거부"만 제공하면 → 사용자가 답답함 (수정 불가)
- "수정 옵션"을 제공하면 → 사용자가 "내 상황을 이해하네" 느낌

### 실천 체크리스트: HITL 패턴을 올바르게 설계했는가?

- [ ] **다섯 가지 패턴을 검토했는가?** (ReAct/Supervisor/Conditional/HITL/Parallel)
  - HITL이 정말 필요한가, 아니면 자동화해도 되는가?

- [ ] **위험도 평가가 객관적인가?** (감정이 아닌 비용 기반)
  - "뭔가 위험해 보여" (X) → "이 작업은 돌이킬 수 없음" (O)

- [ ] **Supervisor → 조건부 분기 → HITL 흐름이 맞는가?**
  - 모든 요청이 HITL을 거쳐야 하나? → 아니다, 조건부여야 함

- [ ] **3가지 경로(승인/거부/수정)가 모두 설계되었는가?**
  - "승인 또는 거부"만 있으면 → 불완전 설계
  - "3가지 다 있음" → 좋은 설계

- [ ] **피드백 루프가 에이전트 학습으로 연결되는가?**
  - 사용자가 거부하거나 수정하면 → 에이전트 로그에 기록 → 향후 프롬프트 개선 시 반영
