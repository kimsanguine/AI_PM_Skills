## Agent Reliability Review

에이전트를 배포하기 전에 반드시 물어야 할 질문:

> "이 에이전트가 잘못 작동할 때 무슨 일이 생기는가?"

일반 제품 QA와 에이전트 신뢰성 검토의 차이:
- 일반 QA: 의도한 기능이 작동하는가? (Happy path)
- 에이전트 신뢰성: 의도하지 않은 상황에서 어떻게 반응하는가?

자율 에이전트는 오류가 조용히 발생합니다.  
사용자가 발견했을 때는 이미 여러 번 반복된 후입니다.

---

### 5가지 실패 도메인

**Domain 1 — Hallucination (허위 정보 생성)**

위험: 에이전트가 없는 정보를 있는 것처럼 생성  
영향: 잘못된 의사결정, 신뢰 손상

체크리스트:
- [ ] 출력에 사실 확인이 필요한 주장이 포함되는가?
- [ ] 소스가 명시되지 않은 수치/통계를 생성하는가?
- [ ] 에이전트가 "모른다"고 말할 수 있는 구조인가?
- [ ] 출력 검증 레이어(Reviewer Agent)가 있는가?

대응 방법:
```
- 사실 주장에 출처 명시 요구 프롬프트 추가
- 불확실한 내용은 "확인 필요" 태그 의무화
- 중요 출력은 Reviewer Agent 또는 Human 검토
```

---

**Domain 2 — Timeout (실행 중단)**

위험: 에이전트가 완료되지 않고 중간에 멈춤  
영향: 서비스 중단, 비용 낭비, 크론 실패 체인

체크리스트:
- [ ] 타임아웃 설정이 있는가?
- [ ] 타임아웃 발생 시 사용자에게 알림이 가는가?
- [ ] 무한 루프 가능성이 있는 로직이 있는가?
- [ ] 외부 API 호출에 타임아웃이 설정되어 있는가?

원인 분석:
```
일반적 원인:
- 웹 검색 무제한 반복 (→ web_search 횟수 제한으로 해결)
- 외부 API 응답 대기 (→ 타임아웃 + 재시도 횟수 제한)
- 컨텍스트 오버플로우로 인한 응답 지연
```

대응 방법:
```
- 크론 타임아웃: 예상 실행 시간 × 3
- web_search: 프롬프트에 "최대 5회" 명시
- 외부 API: timeout=30s + 3회 재시도 제한
```

---

**Domain 3 — Cost Explosion (비용 폭증)**

위험: 예상보다 훨씬 많은 토큰/API 비용 발생  
영향: 운영 비용 통제 불가, 예산 초과

체크리스트:
- [ ] 실행당 최대 토큰 한도가 설정되어 있는가?
- [ ] 반복 루프 내 LLM 호출이 있는가?
- [ ] 대용량 파일을 컨텍스트에 통째로 로드하는가?
- [ ] 월 비용 알림이 설정되어 있는가?

위험 패턴:
```
위험: for item in large_list: llm_call(item)
      → 항목 수만큼 API 호출 = 비용 N배
안전: 배치 처리 또는 요약 후 단일 호출
```

대응 방법:
```
- 실행당 토큰 예산 설정 (context-window-budget 스킬)
- 배치 처리로 API 호출 최소화
- 월 $N 초과 시 Slack/Telegram 알림
```

---

**Domain 4 — Security / Privacy (보안/개인정보)**

위험: 민감 정보 노출, 프롬프트 인젝션, 권한 초과  
영향: 개인정보 침해, 보안 사고, 법적 책임

체크리스트:
- [ ] 에이전트가 처리하는 데이터에 PII가 포함되는가?
- [ ] 외부 입력(웹 콘텐츠, 이메일)이 프롬프트에 직접 삽입되는가?
- [ ] 에이전트가 삭제/수정 권한을 가지고 있는가?
- [ ] API 키/비밀번호가 프롬프트에 노출되는가?

프롬프트 인젝션 위험:
```
위험 패턴:
웹페이지 내용 → 프롬프트에 직접 삽입
→ 악의적 지시 포함 가능 ("Ignore previous instructions...")

안전 패턴:
외부 콘텐츠를 UNTRUSTED로 마킹
→ 시스템 지시와 분리해서 처리
```

대응 방법:
```
- 최소 권한 원칙: 필요한 도구/권한만 부여
- 외부 입력 샌드박싱: UNTRUSTED 콘텐츠 명시
- 민감 데이터 마스킹: 로그에서 PII 제거
- GO 규칙 적용: 수정/삭제 전 인간 확인 필수
```

---

**Domain 5 — Cascading Failure (연쇄 실패)**

위험: 에이전트 A 실패 → 에이전트 B 실패 → 연쇄적 시스템 장애  
영향: 독립적인 에이전트들이 동시에 실패

체크리스트:
- [ ] 에이전트 간 의존관계가 있는가?
- [ ] 상위 에이전트 실패 시 하위 에이전트가 잘못된 데이터로 실행되는가?
- [ ] API 공유 자원(모델 API)이 동시 요청으로 쿨다운에 걸릴 수 있는가?
- [ ] 실패 격리(Circuit Breaker) 메커니즘이 있는가?

실제 사례:
```
2026-02 오전 8:30 동시 실행 문제:
2개 크론잡 동시 실행 → API 쿨다운 → 연쇄 실패
해결: 10분 간격으로 스케줄 분산
```

대응 방법:
```
- 스케줄 분산: 동시 실행 방지 (10분 간격)
- 실패 격리: Worker 실패가 Orchestrator를 중단시키지 않도록
- 재시도 제한: 무한 재시도 방지
- 독립적 실패 처리: 각 에이전트의 실패가 독립적으로 처리
```

### 배포 전 체크리스트 (Pre-deployment)

**필수 (배포 차단):**
- [ ] 타임아웃 설정 완료
- [ ] Anti-Goals 명시 완료
- [ ] 실패 처리 로직 구현
- [ ] 비용 상한 설정

**권장 (배포 후 조치 가능):**
- [ ] Reviewer Agent 또는 Human 검토 프로세스
- [ ] KPI 모니터링 설정
- [ ] 알림 채널 연결
- [ ] 첫 실행 수동 감시 (3회)

### Instructions

You are performing a reliability review for: **$ARGUMENTS**

**Step 1** — 5가지 실패 도메인 체크리스트 수행  
각 항목 Yes/No/N.A. 평가

**Step 2** — 위험 항목 상세 분석  
Yes로 표시된 위험 항목의 구체적 시나리오 서술

**Step 3** — 우선순위 위험 3개 선정  
발생 가능성 × 영향도로 Top 3 선정

**Step 4** — 대응 방안 제시  
각 위험 항목의 구체적 해결 방법

**Step 5** — 배포 판정  
배포 차단 항목 통과 여부 → 배포 가능 / 수정 필요 판정
