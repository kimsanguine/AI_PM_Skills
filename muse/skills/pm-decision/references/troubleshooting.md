# Troubleshooting — pm-decision 스킬

## 1) 적용한 패턴이 현재 맥락과 불일치

**증상:**
- 패턴을 적용했는데 상황이 좀 다른 것 같은 위화감
- "이건 약간 다른데?" 느낌
- 적용 결과가 예상과 다름

**확인:**
- 패턴의 활성화 조건: 현재 상황과 정확히 일치하는가?
- 비활성화 조건: 없는가? (있으면 적용 불가)
- 맥락 차이: 구체적으로 뭐가 다른가?

**조치:**
1. **Boundary Check 재실행**: 패턴을 참고만 하고 현재 맥락 재확인
2. **조건 재검토**: 활성화 조건이 50%만 맞으면 → 패턴 부분 적용 또는 조정
3. **새 TK 추출 검토**: "이게 기존 패턴과 다른 새로운 패턴인가?" → Yes면 `/pm-tacit-extract`
4. **팀원과 논의**: "이 패턴이 우리 상황에 적합한가?" 재검토
5. **기록**: 현재 상황과 차이점을 문서화 → 나중에 패턴 라이브러리 개선에 활용

## 2) 패턴 라이브러리에서 유사 패턴 0개 발견

**증상:**
- 라이브러리 검색 후 관련 패턴이 없음
- "이 상황은 과거에 본 적이 없나?"
- 판단 기준이 애매함

**확인:**
- 검색 키워드: 좋은 키워드로 검색했는가?
  - "긴급" → 더 구체적으로 "예상치 못한 요청", "프로젝트 일정 변화" 등
- 패턴 라이브러리 크기: 아직 패턴이 적은 초기 단계인가?
- 이게 정말 새로운 패턴인가?

**조치:**
1. **키워드 다양화**: "긴급" → "기한 단축", "우선순위 변경", "자원 압박" 등 여러 각도에서 검색
2. **비슷한 도메인 패턴 참고**: 직접 같은 상황은 아니지만, 유사한 판단 기준이 있는가?
3. **새 TK 추출**: 이것이 정말 새로운 패턴이면 `/pm-tacit-extract`로 즉시 기록
4. **임시 판단**: TK 추출 전까지는 "Why-First Decision Making" 같은 기본 패턴 적용
5. **라이브러리 성장 대기**: 패턴이 쌓일수록 유사 케이스 발견 가능성 ↑

## 3) 패턴을 따랐는데도 결과가 나쁨

**증상:**
- "Why-First Decision Making을 했는데 고객이 여전히 불만족"
- 또는 "Prototype-First로 했는데 프로토타입이 실패"
- "패턴이 틀렸나?"

**확인:**
- 패턴의 핵심을 정말 따랐는가?
  - Why-First: "왜"를 물었는가, 아니면 피상적으로 물었는가?
  - 재시도/폴백이 있었는가?
- 실행 방식의 문제:
  - 타이밍이 나빴나?
  - 실행자의 실수가 있었나?
  - 외부 요인이 변수였나?
- 패턴 자체의 한계:
  - 이 패턴이 모든 상황에서 통하는가?

**조치:**
1. **실행 리뷰**: "패턴은 맞았는데 실행이 부족했나?" 체크
   - Why-First 패턴이지만 충분히 깊이 있게 물었는가?
   - Prototype-First이지만 프로토타입이 충분히 검증되었는가?
2. **패턴 자체 평가**:
   - 이 패턴이 이 상황에 정말 적합한가?
   - 더 나은 패턴이 있는가?
3. **Iteration**: 패턴을 다시 적용하되, "피드백 기반 개선"
4. **TK 수정**: 필요시 패턴을 "이 조건 하에서는 작동하지 않음" 추가 설명
5. **실패 사례 기록**: "패턴 X를 따랐으나 Y 이유로 실패" 기록 → 라이브러리 강화

## 4) 팀원이 패턴을 잘못 이해하고 실행

**증상:**
- 패턴을 공유했는데 팀원이 다르게 이해
- "Stakeholder Energy Management를 했다"는데 모든 요청을 무시함
- 의도와 다른 결과

**확인:**
- 패턴 설명이 충분했는가?
  - 패턴의 목표를 명시했는가? ("팀 집중력 보존")
  - 활성화 조건을 구체적으로 설명했는가?
  - 비활성화 조건(예외)을 명시했는가?
- 예제가 있었는가? (추상적 설명보다 구체적 예제가 중요)
- 팀원의 이해도를 확인했는가?

**조치:**
1. **패턴 재설명**: "활성화 조건"과 "비활성화 조건" 분리해서 명확히
2. **구체적 예제 제시**: 추상적 설명 대신 "이런 경우는 O, 저런 경우는 X"
3. **결정 근거 공유**: "왜 이 패턴을 따르는가?" 팀원이 이해하게 설명
4. **피드백 루프**: 팀원이 적용했을 때 "어땠는가?" 물어보고 개선
5. **Written documentation**: 구두 설명보다 문서화 → 반복 학습 가능

## 5) 패턴과 정책이 충돌

**증상:**
- 패턴으로는 "A"를 권장
- 기업 정책으로는 "B"를 요구
- 어느 것을 따를까?

**확인:**
- 정책의 성격: 규제/법률 의무인가, 아니면 기업 내규인가?
- 패턴의 범위: 이 패턴이 정책 상황을 고려하도록 설계되었는가?
- 충돌의 본질: 근본적 충돌인가, 아니면 이해 부족인가?

**조치:**
1. **정책 우선**: 법적/규제 의무가 있으면 패턴을 무시하고 정책 준수
2. **정책 예외 신청**: 기업 내규면 "예외 요청" 절차 검토 가능
3. **패턴 재정의**: "정책 제약 하에서 이 패턴을 어떻게 적용할까?"
4. **문서화**: "이 경우는 정책 때문에 패턴을 적용하지 않음" 명시
5. **정책 개선 제안**: 장기적으로 "정책이 패턴을 반영하도록" 개선 요청
