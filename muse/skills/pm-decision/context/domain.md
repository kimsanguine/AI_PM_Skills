# Domain Context — pm-decision

## 1) Domain Scope

pm-decision은 **PM이 매일 반복하는 의사결정 상황에서 과거 경험 기반 패턴을 검색하여 현재 판단에 즉시 적용**하는 스킬입니다.

| 포함 | 제외 |
|-----|------|
| 의사결정 패턴 라이브러리 검색 | 기술 선택 조언 |
| 패턴 매칭 및 현재 상황 적용 | 비용 계산 (oracle) |
| 함정/인지 편향 체크 | 데이터 분석 (argus) |
| 패턴 기반 판단 근거 제시 | 신규 TK 추출 (pm-framework) |
| 과거 유사 결정 사례 제시 | Instruction 작성 (pm-engine) |

**핵심**: pm-decision은 "왜 이렇게 판단했는가"의 패턴 근거 데이터베이스입니다.

---

## 2) Primary Users

1. **PM (Product Manager)**
   - 반복되는 의사결정 시 라이브러리의 패턴을 즉시 참조
   - "이전에 비슷한 상황에서 어떻게 판단했나?" 빠르게 확인
   - 팀과의 의사결정 토론에서 패턴을 근거로 설명

2. **에이전트 (Agent)**
   - 의사결정이 필요한 상황에서 패턴 라이브러리 자동 검색
   - 유사 패턴 찾아 현재 상황에 적용
   - 함정을 사전에 회피하도록 Instruction에 반영

3. **팀 (Team)**
   - 신입 PM이 과거 결정 패턴을 학습 (Onboarding)
   - 의사결정 근거를 명확히 이해 (설명 가능성)
   - "우리 팀의 의사결정 원칙"으로 정규화

---

## 3) Required Inputs

### 필수 입력 1: 의사결정 상황 (Decision Scenario)
```
"마케팅팀이 '긴급하게 홍보 페이지를 추가해야 한다'고 요청했어.
 우리 팀은 지금 에이전트 개발 중이고, 마감이 2주 남았어.
 이 요청을 받아들일지, 거절할지 결정해야 해."
```

또는 구체적 맥락:
```
"우리가 지난 3개월 간 스코프 vs 일정 트레이드오프를 3번 결정했는데,
 매번 다르게 결정했어. 이번에는 패턴을 찾아서 일관되게 하고 싶어."
```

### 선택 입력: 인지 편향 체크 요청
```
"이 결정을 할 때 우리가 빠질 수 있는 심리적 함정이 뭐가 있을까?"
```

### 선택 입력: 함정 분석 (Failure Mode)
```
"과거에 비슷한 패턴으로 결정했을 때 실패한 케이스가 있어?"
```

## 4) Output Contract

### 출력 형식 1: 패턴 매칭 결과
```
[상황 분석]
• 현재 의사결정: 요청 우선순위 판단
• 유사도가 높은 패턴: "Stakeholder Energy Management"

[패턴 적용]
• 패턴명: Stakeholder Energy Management
• 활성화 조건: 여러 이해관계자의 동시 요청 ✓
• 판단 기준:
  - 각 요청을 "비즈니스 임팩트"로만 평가
  - 발신자의 직급/압박감은 우선순위 기준이 아님
  - 데이터 기반 우선순위 기준 사전 수립

[현재 상황 적용]
• 마케팅 요청의 임팩트: "고객 전환율 3% 증가 예상"
• 에이전트 개발 마감: 2주 → 비즈니스 임팩트 높음
• 권장 판단: 마케팅 요청 보류 (일정 우선)

[함정 체크]
• 주의사항: "긴급"이라는 표현 신뢰 금지
• 과거 사례: 비슷한 "긴급" 요청 80%는 사후 연기 가능했음
• 예방: 실제 마감일 먼저 확인
```

### 출력 형식 2: 함정/인지 편향 분석
```
[발견된 인지 편향]
• Sunk Cost Fallacy: "이미 4주 투자했으니 마무리해야 한다" X
• Planning Fallacy: "남은 일이 정말 2주면 끝날까?" 재검증
• Availability Heuristic: "최근에 본 사례"만 기억나는 경향

[과거 유사 결정과의 비교]
• 지난 3번의 유사 결정:
  1) 일정 우선 → 결과: 성공 (고객 만족도 높음)
  2) 스코프 우선 → 결과: 실패 (일정 지연, 품질 저하)
  3) 기간 연장 → 결과: 혼합 (일정은 지켰으나 리소스 낭비)

[권장]
패턴 데이터: 일정 우선이 장기적으로 성공률 높음
```

### 출력 형식 3: 패턴 라이브러리 검색 결과
```
[검색 결과]
• 검색어: "스코프 vs 일정"
• 매칭된 패턴 2개:

  1) Why-First Decision Making (일치도 95%)
     - 상황: 요청/아이디어가 들어왔을 때
     - 함정: 요청을 그대로 수행하고 "왜"를 묻지 않음

  2) Scope Creep Prevention (일치도 90%)
     - 상황: 프로젝트 진행 중 "이것도 추가하면 어때?"
     - 함정: 좋은 아이디어를 모두 넣으려다 아무것도 못 냄
```

## 5) Guardrails

### Guardrail 1: 패턴이 현재 맥락과 완벽히 일치하지 않을 수 있음
**규칙**: 패턴을 참고만 하고, 항상 현재 맥락 재확인
```
❌ "패턴 라이브러리에서 이렇게 나왔으니까 이대로 하자"
✅ "패턴은 이렇지만, 우리 상황이 조금 다른 부분이 있나 확인해보자"
```

### Guardrail 2: 이미 정해진 정책/프로세스가 있으면 우선
**규칙**: 패턴 참조보다 조직의 정책 준수가 우선
```
예: 회사의 "마케팅 요청 우선순위 기준"이 명시되어 있으면
    패턴 라이브러리보다 그 정책을 따름
```

### Guardrail 3: 법적/규제 의무는 패턴만으로 판단 금지
**규칙**: 법적 책임이 있는 결정은 전문가 검토 필수
```
❌ 패턴만으로: "GDPR 대응은 우리 패턴에서 A방식이 나왔으니 이렇게 하자"
✅ 법무팀 검토 후: "법무팀이 승인한 A방식을 따르되, 참고로 우리의 과거 패턴도..."
```

### Guardrail 4: 패턴의 근거가 무엇인지 항상 명시
**규칙**: 패턴을 권장할 때 "왜 이 패턴이 더 나은가"의 근거 명시
```
❌ "Stakeholder Energy Management 패턴을 따르세요"
✅ "Stakeholder Energy Management 패턴을 따르는 이유:
    지난 10번의 유사 결정에서 이 패턴이 8번 성공했고,
    팀의 집중력 분산도 30% 줄어들었습니다"
```

### Guardrail 5: 패턴 적용 후 결과 추적
**규칙**: 패턴을 따르고 난 후 "실제로 예상대로 나왔나" 검증
```
기록:
- 패턴명: Stakeholder Energy Management
- 적용 날짜: 2026-03-07
- 기대 결과: "긴급 요청 우선순위 50% 감소"
- 실제 결과: 2주 후 확인 예정
```
