# Bad Example — pm-decision 오용

## 사용자 요청
"에이전트 신규 기능을 만들지 기존 도구를 쓸지,
어느 쪽이 더 나을까? 기술적으로 뭐가 좋은지 알려줘."

## 거절 이유
- pm-decision의 역할: 반복된 의사결정 패턴을 라이브러리에서 찾아 제시
- 사용자의 요청: 기술적 판단 (Build vs Buy의 기술 측면)
- 이것은 pm-engine의 TK-007(Build vs Buy 2주 법칙)을 통한 판단이 맞음
- pm-decision은 "패턴 라이브러리"이지 "기술 조언"이 아님

## 올바른 라우팅
- pm-engine 사용: `/pm-decision-log [Build vs Buy 상황]`
  → TK-007, TK-003 로드하여 비용/시간 기반 판단
- oracle 사용: cost-sim으로 실제 비용 시뮬레이션

## 수정 방향
사용자가 다시 요청: "우리 팀은 지난 3개월간 같은 Build vs Buy 결정을 3번 반복했는데,
매번 다르게 결정했어. 이번엔 일관된 패턴으로 결정하고 싶은데,
예전 결정들을 리뷰해서 어떤 패턴이 더 좋은지 보여줄래?"

→ 그러면 pm-decision이 적절:
  "과거 3번의 Build vs Buy 결정을 분석하여,
   성공한 패턴과 실패한 패턴을 제시하고,
   이번 결정에 적용할 최적 패턴 권장"
