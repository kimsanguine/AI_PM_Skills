# Good Example — pm-framework TK 추출

## 사용자 요청
"오늘 회의에서 중요한 배움이 있었어.
에이전트를 만들 때, 처음부터 모든 기능을 다 넣으려고 했더니 3배 오래 걸렸는데,
핵심 기능 1개만 먼저 만들고 피드백 받으니까 훨씬 빨라졌어.
이걸 TK 유닛으로 구조화해서 PM-ENGINE에 저장해줄래?"

## 승인 이유
- pm-framework의 핵심: PM의 경험을 명시적 TK로 변환하여 재사용 가능하게 하기
- 사용자의 경험: 반복적으로 검증된 판단 기준 (MVP 우선 원칙)
- 이것은 Type 2 (Failure Pattern) + Type 3 (Heuristic)에 해당

## 예상 처리
1. 상황 청취: "모든 기능 포함 vs 핵심만 먼저" 두 접근법의 결과 명확히
2. 암묵지 패턴 포착: "처음부터 모든 기능을 넣으면 망한다", "MVP로 시작하면 빠르고 피드백이 명확하다"
3. TK 유형 분류: Type 2 (Failure Pattern) — "처음부터 모든 기능을 넣으면 망한다"
4. TK 구조화:
   ```
   TK-NNN: Minimum Viable Agent

   📌 패턴: 에이전트 설계 초안을 잡을 때,
            이 에이전트가 해야 할 단 하나의 핵심 기능만 정의하고
            그 핵심만으로 작동하는 최소 버전을 먼저 배포한다.

   🟢 활성화 조건: 신규 에이전트 설계, MVP 정의, 배포 계획 수립 시
   🔴 비활성화 조건: 규제/컴플라이언스로 사전 기능이 모두 필요한 경우
   💡 Why: 초기에 기능이 많을수록 복잡도 급증, 실패 원인 파악 어려움.
           MVP로 시작하면 피드백이 명확하고, 확장도 빠름.
   🔗 연관 TK: TK-015(Prototype-first), TK-010(그로스는 리텐션 다음)
   ```
5. 연관 TK 연결: 기존 TK 중 프로토타입, 그로스 관련 TK와 연결
6. PM-ENGINE-MEMORY에 저장: TK-090으로 추가 (예시)
7. 다음 단계 권고: `/tk-to-instruction`으로 에이전트 Instruction에 반영
