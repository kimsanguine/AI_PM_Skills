# Domain Context — pm-engine

## 1) Domain Scope

pm-engine은 **PM의 축적된 판단 기준(TK)을 에이전트가 실행 중에 동적으로 참조하여 의사결정 품질을 높이는 시스템**입니다.

| 포함 | 제외 |
|-----|------|
| PM-ENGINE-MEMORY의 TK 검색/참조 | 일반 LLM 기술 조언 |
| TK를 에이전트 Instruction으로 변환 | 프롬프트 엔지니어링 팁 |
| 새로운 TK 추출 및 저장 | 에이전트 아키텍처 설계 |
| TK 간 연관성 매핑 (지식 그래프) | 모델 선택, 인프라 설계 |
| TK 활성화/비활성화 조건 확인 | 비용 계산, 데이터 분석 |

**핵심**: pm-engine은 "왜 이렇게 판단하는가"의 근거 데이터베이스입니다.

---

## 2) Primary Users

1. **PM (Product Manager)**
   - PM의 경험에서 TK를 추출하여 저장
   - 일상 의사결정 시 TK를 참조하여 판단 품질 향상
   - TK가 축적될수록 자신의 판단을 자동화할 수 있음

2. **에이전트 (Agent)**
   - 운영 중 현재 상황과 관련된 TK를 동적으로 검색
   - TK의 "왜"를 이해하고 맥락에 맞는 판단 수행
   - Instruction에 반영된 TK를 따라 의사결정 자동화

3. **팀 (Team)**
   - PM의 판단 기준을 공유받아 일관된 의사결정
   - 새로운 팀원이 PM의 사고방식을 빠르게 습득
   - 의사결정 근거를 "TK 번호"로 참조하여 토론 효율화

---

## 3) Required Inputs

### 필수 입력 1: PM 경험 또는 판단 상황
```
"어제 에이전트 MVP를 배포할 때,
기능을 하나만 넣고 시작했더니 피드백이 빨리 들어왔어."
```
또는
```
"우리는 새 기능을 만들지 아니면 기존 도구를 쓸지 2주 규칙으로 판단해."
```

### 필수 입력 2: TK 검색 쿼리 (상황 맥락)
```
"지금 스코프 크리프로 일정이 밀리고 있는데,
이런 상황을 어떻게 처리해야 하는지 TK를 찾아줘."
```

### 선택 입력: TK 번호 직접 제시
```
"TK-001과 TK-009를 우리 현재 상황에 적용해줘."
```

## 5) Guardrails

### Guardrail 1: TK와 일반 LLM 지식의 충돌
**규칙**: PM-ENGINE-MEMORY의 TK가 일반 LLM 지식과 충돌하면 TK 우선
```
❌ "일반적으로 모든 기능을 먼저 계획하는 것이 좋습니다"
✅ "TK-012에서는 스펙 완성 후 개발이 아니라
    프로토타입 → 검증 → 스펙 순서를 추천합니다"
```

### Guardrail 2: 활성화/비활성화 조건 확인 필수
**규칙**: TK는 항상 조건이 맞을 때만 적용
```
❌ TK-001(긴급 요청 우선순위 판단)을
   실제 SLA 걸린 장애 대응에 적용 (비활성화 조건 위반)

✅ TK-001을 마케팅팀의 "긴급하다"는
   느낌만 있는 요청에 적용 (활성화 조건 만족)
```

### Guardrail 3: TK 부족 영역에서는 데이터 검증 필수
**규칙**: 새 제품, 새 시장처럼 TK가 쌓이지 않은 영역에서는 TK만 믿지 말 것
```
❌ "TK가 없으니까 판단 안 함"
✅ "TK가 없으므로 가정을 명시하고
    2주 내 데이터 검증 계획 수립"
```

### Guardrail 4: 숨겨진 가정 명시
**규칙**: TK 적용 시 "이 판단이 가정하는 것"을 명시
```
예: TK-003(에이전트 비용 10배 법칙) 적용 시:
- 가정: 모델 가격이 현재 수준 유지
- 가정: 사용 패턴이 현재와 유사
- 재검증 시점: 매 분기 가격 체크
```

### Guardrail 5: TK의 유효 기한 확인
**규칙**: 시간이 지나면서 시장 변화로 TK가 부실화될 수 있음
```
예: TK-008(경쟁사 카피는 해자가 아님)
- 초안 작성: 2026-03
- 마지막 검증: 2026-03
- 재검증 필요: 2026-06 (분기별)
```
