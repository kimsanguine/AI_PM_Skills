# Troubleshooting — pm-engine

## 1. 증상: "관련 TK가 없다"는 검색 결과

### 증상 설명
```
사용자: "우리가 신규 시장(AI 의료)에 진입하는데,
         PM-ENGINE-MEMORY에서 관련 TK를 찾아줘"

pm-engine 응답: "No TK found for this context"
```

### 확인 (Verification)
1. **PM-ENGINE-MEMORY.md 자체가 존재하는가?**
   - 경로: ___ (확인 필요)
   - 파일 크기: ___ (TK-001~TK-010 최소 포함 확인)

2. **검색 쿼리가 정상인가?**
   - 쿼리: "AI 의료 진단 에이전트"
   - 키워드 매칭: 의료(medical), 규제(regulation), 진단(diagnosis)
   - TK 메타데이터: 활성화 조건, 패턴, Why 섹션에 위 키워드 포함 여부?

3. **TK가 실제로 존재하는가?**
   - 확인: PM-ENGINE-MEMORY에 의료/규제 관련 TK가 있는가?
   - 없으면: **이것은 버그가 아니라 정상 (TK 부재)**

### 조치 (Action)

**Case 1: TK가 실제로 없는 경우 (정상)**
```
→ 사용자에게 피드백:
  "AI 의료 영역은 아직 TK가 없습니다.

   옵션:
   1) 유사 영역 TK 활용: TK-006(환각 UX), TK-002(AI 네이티브) 등
   2) 새 TK 추출: `/pm-tacit-extract [의료 진출 경험]`
   3) 데이터 검증: 2주 내 규제 요구사항 조사 후 TK 추가"

→ 이후: 사용자의 의료 경험 추출 후 TK-NNN 추가
```

**Case 2: TK는 있는데 검색 안 되는 경우 (버그 가능)**
```
→ 진단:
  1) TK 활성화 조건 검토
     - TK-006: "에이전트 환각 문제가 발생했을 때"
     - 현재 상황: "아직 상품 기획 단계"
     - → 활성화 조건 불일치

  2) Contextual Retrieval 패턴 재점검
     - 활성화/비활성화 조건이 명확하게 분류되어 있는가?
     - 없으면: TK 메타데이터 재작성

→ 조치:
  /weekly-memory-distill 크론 실행하여 CR 필드 검증
  → TK 메타데이터 업데이트
  → 재검색
```

## 2. 증상: 로드한 TK의 활성화 조건이 현재 상황과 불일치

### 증상 설명
```
상황: 실제 SLA 걸린 장애 대응 중

pm-engine 제시: "TK-001을 적용하세요. '긴급 요청인지 확인하라'"

PM: "어? 이미 긴급이 확실한데, 왜 또 확인하라고?"
   → TK를 적용했는데 맥락이 안 맞음
```

### 확인 (Verification)

1. **TK의 비활성화 조건을 확인했는가?**
   ```
   TK-001: 긴급 요청 우선순위 판단

   🔴 비활성화 조건:
   "실제 SLA가 걸린 장애 대응 상황"

   현재 상황: ✓ 정확히 비활성화 조건
   ```

2. **pm-engine이 비활성화 조건을 체크했는가?**
   - pm-engine의 Logic: 활성화 조건만 체크하고 비활성화 조건은 건너뛰었을 가능성

3. **PM이 TK를 무비판적으로 적용했는가?**
   - "TK라고 해서 항상 맞다"고 생각

### 조치 (Action)

**Step 1: pm-engine 로직 수정**
```
Before:
  IF 활성화 조건 일치 → TK 적용

After:
  IF (활성화 조건 일치 AND NOT 비활성화 조건) → TK 적용
  ELSE → "이 TK는 현재 상황에 적용할 수 없습니다" + 대안 제시
```

**Step 2: 현재 상황에 올바른 TK 제시**
```
→ pm-engine 응답:
  "TK-001은 이 상황에 적용 불가입니다.
   (비활성화 조건: 실제 SLA 걸린 장애)

   현재는 즉시 대응이 맞습니다.
   이후 사후 분석 시 TK-001을 참고하여
   '긴급 요청 판단 기준'을 개선할 수 있습니다."
```

**Step 3: TK 구조 재검토**
```
Action: TK 리뷰 사이클에서
  - 이 TK의 활성화/비활성화 조건이 명확하게 정의되어 있는가?
  - 더 명확해야 하는가?
  → 필요 시 TK 분할
    예: TK-001 → TK-001a (인식된 긴급), TK-001b (실제 긴급)
```

## 3. 증상: TK 간 연관 관계가 부족하여 관련 지식을 못 찾음

### 증상 설명
```
상황: "우선순위 관리" 관련 판단 필요

pm-engine 검색: TK-001(긴급 요청) 반환

TK-001의 🔗 연관 TK: TK-004, TK-009만 명시

하지만 실제로는 TK-003(이해관계자 관리)도 관련인데
  → 연관 TK에 없어서 사용자가 못 찾음
```

### 확인 (Verification)

1. **TK의 연관 링크가 양방향인가?**
   ```
   TK-001 → TK-003 링크 있음 (O)
   TK-003 → TK-001 역링크 있음 (X)

   → 역링크 부재
   ```

2. **Contextual Retrieval 패턴이 모든 TK에 적용되었는가?**
   - CR 필드: 활성화 조건, 비활성화 조건
   - 모든 TK-001~TK-010에 포함 여부?

### 조치 (Action)

**Step 1: weekly-memory-distill 크론 실행**
```
주기: 매주 월요일 09:00
작업:
  1) 모든 TK의 활성화/비활성화 조건 재점검
  2) 누락된 연관 TK 추가
  3) 역링크 검증 (A→B가 있으면 B→A도 있어야 함)
```

**Step 2: TK 관계도 시각화**
```
생성물: TK 간 관계를 그래프로 시각화
  TK-001 ←→ TK-003 ←→ TK-004
    ↓        ↓        ↓
  TK-009   TK-005   TK-007

목표: "연관 TK가 명확히 보이도록"
```

**Step 3: 검색 알고리즘 개선**
```
Before:
  검색어 "우선순위" → TK-001만 반환

After:
  검색어 "우선순위" → TK-001 반환 + 연관 TK 자동 제시
                    (TK-001의 연관 TK: TK-003, TK-004, TK-009)
```

## 4. 증상: TK → Instruction 변환 후에도 에이전트 판단이 개선 안 됨

### 증상 설명
```
상황: TK-009(질문의 품질)를 에이전트 Instruction에 반영

변환:
  "사용자에게 응답할 때, 단순 실행 전에
   '이 작업의 목적이 무엇인지' 맥락을 파악하세요"

배포 후 결과:
  에이전트의 응답 품질이 여전히 낮음
  → "TK는 맞는데 Instruction 실행이 약한 것?"
```

### 확인 (Verification)

1. **Instruction 문장이 구체적인가?**
   ```
   ❌ "맥락을 파악하세요" (너무 추상적)
   ✅ "사용자에게 이 작업을 하는 이유를 물어보고,
        그 답에 따라 구체적인 다음 단계를 제시하세요"
   ```

2. **에이전트가 TK 내용을 실제로 따르고 있는가?**
   - 에이전트의 최근 응답 샘플링 (5개)
   - 각 응답에서 "맥락 파악" 단계가 있는가?

3. **Instruction 위치가 올바른가?**
   ```
   에이전트 System Prompt의 어디에?
   - Goal 섹션? (맥락 설명)
   - Behavior 섹션? (행동 지시)
   - Example 섹션? (구체적 예)

   권장: Behavior + Example에 모두 포함
   ```

### 조치 (Action)

**Step 1: Instruction 재작성**
```
Before:
  "질문의 품질이 중요합니다. 맥락을 먼저 파악하세요."

After:
  "사용자의 요청이 올 때마다:
   1) "이 작업의 최종 목적은 무엇인가?"를 파악
   2) "왜 지금 이것을 하려고 하는가?" 이유 확인
   3) 목적에 따라 최적 방법 제시

   예: 사용자가 "보고서 양식을 만들어줘"라고 할 때,
       단순 양식이 아니라 "왜 이 보고서가 필요한가"
       (임원 보고용? 팀 공유용?)를 먼저 물어보고,
       그에 맞는 양식을 제시"
```

**Step 2: A/B 테스트**
```
Group A: 기존 Instruction
Group B: 새 Instruction (TK-009 명시 반영)

측정: 응답 품질 개선도
기대: 30% 이상 향상
```

**Step 3: 근본 원인 분석**
```
Q: TK 내용은 맞는데 왜 Instruction 실행이 약한가?

가능 원인 1: Instruction 문장이 애매함
→ 더 구체적인 Step-by-Step으로 재작성

가능 원인 2: 에이전트 모델이 이 Instruction을 따르기 어려움
→ 더 단순하고 직설적인 지시로 변경
→ 또는 모델 버전 업그레이드

가능 원인 3: TK 자체가 이 에이전트 도메인에 안 맞음
→ TK의 활성화 조건 재점검
→ 다른 TK 적용 시도
```

## Quick Reference: 상황별 대응

| 상황 | 증상 | 원인 | 조치 |
|------|------|------|------|
| TK 검색 실패 | "No TK found" | TK 부재 또는 CR 조건 불일치 | Step 1-3 참고 또는 `/weekly-memory-distill` 실행 |
| TK 비활성화 조건 위반 | 맥락 불일치 | pm-engine이 비활성화 조건 미체크 | 로직 수정: 양쪽 조건 모두 체크 |
| 연관 TK 누락 | 관련 지식을 못 찾음 | 양방향 링크 부재 | `weekly-memory-distill` 크론으로 자동 재검증 |
| Instruction 실행 약함 | 의도대로 동작 안 함 | 문장이 추상적이거나 모델 한계 | 구체적 Step-by-Step으로 재작성 + A/B 테스트 |
