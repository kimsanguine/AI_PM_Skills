# Troubleshooting — instruction 스킬

## 1) Anti-Goals 추상화

**증상:**
- 작성된 Anti-Goals: "확신 없는 답 하지 않기", "정확하게 해줘", "빠르게 해줘"
- 문제: 에이전트가 어떻게 행동해야 하는지 불명확
- 결과: 실행 단계에서 판단이 미뤄짐

**확인:**
- 각 Anti-Goals가 구체적인 시나리오로 작성되었는가?
- 에이전트가 "이 상황에서는 이렇게 행동한다"고 판단할 수 있는가?

**조치:**
- 각 Anti-Goals를 구체적 시나리오로 재작성:
  ```
  ❌ "정확하게 해줘"
  ✅ "신뢰도 < 0.7이면 '❓ 확인 필요' 표시하고, 확신 있는 정보와 구분"
  ```

## 2) Tool 사용 조건 누락

**증상:**
- Tool 목록에는 도구명과 용도만 있음
- "언제 사용할지" 조건과 "제한"이 명시되지 않음
- 구현팀이 모든 상황에서 도구를 과다 호출

**확인:**
- 각 도구에 "사용 조건" 컬럼이 있는가?
- 각 도구에 "호출 제한" 컬럼이 있는가?
- 최소 권한 원칙이 적용되었는가?

**조치:**
- Tool 테이블 재정의:
  ```
  | 도구 | 용도 | 사용 조건 | 호출 제한 |
  |------|------|----------|---------|
  | web_search | 최신 정보 | 내부 캐시 부족 시만 | 최대 5회/실행 |
  ```

## 3) Memory 3계층 구조 혼동

**증상:**
- Memory 섹션에 "파일들을 로드합니다"만 있음
- 단기/장기/절차적 메모리 구분 없음
- 어떤 파일을 항상 로드할지, 조건부로 로드할지 불명확

**확인:**
- Working Memory(단기): 현재 세션에서만 필요한 정보
- Long-term Memory(장기): 세션 간 유지해야 할 정보
- Procedural Memory(절차적): 반복 판단 패턴 (SKILL.md)

**조치:**
- 3계층 분리:
  ```
  Working Memory:
  - SOUL.md (~600 tokens)
  - 오늘 메모리 파일 (~1500 tokens)

  Long-term Memory:
  - 어제 뉴스 캐시 (중복 제거용)

  Procedural Memory:
  - news/summarization.md (요약 스타일)
  - news/credibility-check.md (신뢰도 판단)
  ```

## 4) Output Format이 채널에 맞지 않음

**증상:**
- Output Format: Markdown 테이블
- 채널: Telegram
- 문제: Telegram은 마크다운 테이블 미지원 → 렌더링 오류

**확인:**
- 선택한 채널이 무엇인가?
- 그 채널이 지원하는 포맷은 무엇인가?

**조치:**
- 채널별 포맷 최적화:
  ```
  ❌ Telegram에 Markdown 테이블
  ✅ Telegram에 리스트 형식 + 이모지 활용
  ```

## 5) Failure Handling 불완전성

**증상:**
- Failure Handling: "에러가 발생하면 알림"
- 문제: 구체적인 에러 유형별 대응이 없음
- 결과: 구현팀이 각자 다르게 처리

**확인:**
- 4가지 이상 현실적 실패 시나리오가 있는가?
- 각 시나리오별 "감지 방법"과 "구체적 행동"이 명시되었는가?

**조치:**
- 실패 시나리오별 구체화:
  ```
  | 시나리오 | 감지 방법 | 행동 |
  |---------|---------|------|
  | API 타임아웃 | timeout 에러 | 3회 재시도 → 실패 시 어제 결과 사용 |
  | 뉴스 검색 0건 | empty 결과 | 어제 상위 3건 재발송 + "신규 뉴스 없음" 표시 |
  ```
