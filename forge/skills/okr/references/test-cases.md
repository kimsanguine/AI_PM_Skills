# Test Cases — okr 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "morning-briefing 에이전트의 OKR을 설계해 줄 수 있어? 시간 절감, 신뢰도, 비용을 측정하고 싶어."
   - 이유: 에이전트 이름, 측정할 3개 지표(시간/신뢰도/비용) 명시 → Step 1~5 설계 진입 가능

2. "우리 고객 지원 에이전트가 배포되었어요. 이제 분기 OKR을 세우고 성과를 추적해야 해요."
   - 이유: 배포 완료 + OKR 설정 필요 명확 → Step 1 Objective부터 시작

3. "How do I measure success for our AI research agent? We need both business and operational KRs."
   - 이유: 비즈니스 + 운영 KR 분리 요청 명시 → 2축 OKR 설계 진입 가능

4. "에이전트 포트폴리오 3개의 우선순위를 정해야 하는데, 각각의 KR 달성률로 평가하고 싶어요."
   - 이유: 에이전트 비교/우선순위 → OKR 기반 포트폴리오 관리 설계

5. "비용 KR을 초과했어요. 무엇이 문제일까요? 다음 분기는 어떻게 조정할까요?"
   - 이유: 현재 OKR 성과 리뷰 + 다음 분기 조정 → Step 4~5 측정/리뷰 사이클 설계

### Should NOT Trigger (5)

1. "우리 전체 회사 OKR을 설계해 줄 수 있어?"
   - 올바른 라우팅: 회사 수준 OKR → 일반 OKR 프레임워크 (에이전트 특화 아님)

2. "에이전트가 너무 느려요. 어떻게 최적화할까요?"
   - 올바른 라우팅: 성능 최적화 → `agent-plan-review` 또는 기술 스킬 (OKR 범위 밖)

3. "우리 에이전트의 비용이 너무 많이 들어요. 어떤 모델을 써야 할까요?"
   - 올바른 라우팅: 비용 분석/모델 선택 → `oracle/cost-sim` 또는 `atlas/router`

4. "이번 달 API 비용이 얼마인지 확인해 줄 수 있어요?"
   - 올바른 라우팅: 비용 추적/모니터링 → `argus/burn-rate` (실시간 모니터링)

5. "에이전트의 신뢰성을 개선하려면 뭘 해야 할까요?"
   - 올바른 라우팅: 신뢰성/SLO 설계 → `argus/reliability` 스킬

## 2) Edge Cases

### 경계 사례 (5)

1. **초기 배포 직후 데이터 부재로 베이스라인 설정 불가**
   - 입력: "에이전트를 방금 배포했는데, KR 설정 전에 데이터를 어떻게 모을까요?"
   - 예상 행동: 처음 2주를 "데이터 수집 기간"으로 설정 → 베이스라인 측정 완료 후 다음 주부터 정식 OKR 시작
   - 근거: 베이스라인 없이 목표 설정 불가 → Step 2~3 KR을 가정값으로 임시 설정 후 2주 데이터 기반 재협상

2. **비즈니스 임팩트 KR이 측정 불가능한 경우 (예: "사용자 만족도")**
   - 입력: "에이전트의 '사용자 만족도'를 높이는 게 목표인데, 이걸 어떻게 측정해요?"
   - 예상 행동: 추상적 지표 → 측정 가능한 프록시 지표로 변환 (예: "재사용률 80%", "에이전트 호출 주 3회 이상")
   - 근거: KR은 반드시 정량화 가능해야 함 → 도메인 특화 프록시 지표 정의

3. **KR이 너무 높게 설정되어 분기 중반부터 달성 불가능 확실할 때**
   - 입력: "설정한 KR을 보니 이미 불가능해 보여요. 조정할까요?"
   - 예상 행동: 즉시 KR 재협상 (낮게 재설정) → 재설정 이유 문서화 (아키텍처 문제/스코프 미포함 등) → 다음 분기에 반영
   - 근거: KR은 야심차되 달성 가능해야 함 → 중간 수정은 정상 (초기 추정 오차)

4. **여러 에이전트의 KR이 충돌할 때 (예: A 비용↑ vs B 비용↓)**
   - 입력: "고객 지원 에이전트는 비용을 늘려야 하는데, 전체 예산 한도 때문에 다른 에이전트 비용을 줄여야 해요."
   - 예상 행동: 포트폴리오 리뷰 → 우선순위 재조정 (고임팩트 에이전트에 리소스 집중) → 저임팩트 에이전트 일시 중지 검토
   - 근거: 총합 비용 한도는 고정 → 우선순위 기반 재할당 필수

5. **월간 리뷰 시 데이터 부재로 KR 달성률 계산 불가**
   - 입력: "로그를 남기지 않아서 지난 달 KR 성과를 계산할 수 없어요."
   - 예상 행동: 현재 달부터 자동 로깅 설정 (성공/실패 건수, 응답 시간 등) → 지난 달은 추정값으로 임시 평가 → 다음 달부터 정확한 데이터 기반 리뷰
   - 근거: 측정 불가능 = 개선 불가능 → 자동화 로깅 필수
