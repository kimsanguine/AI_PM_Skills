# Test Cases — ctx-budget 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "morning-briefing 에이전트의 컨텍스트 윈도우 예산을 계획해 줄 수 있어? SOUL.md, 메모리 파일, SKILL.md 여러 개를 로드해야 해."
   - 이유: 에이전트 이름 명시, 로드할 파일 목록 제시 → Step 1 파일 목록부터 시작 가능

2. "We're running a customer support agent that loads multiple tools and documents. The context window is getting tight around 85% usage. Can you help optimize?"
   - 이유: 컨텍스트 사용률(85%) 명시 → 즉시 Step 3 최적화 제안 진입 가능

3. "우리 에이전트가 Haiku 모델을 쓰고 있는데, 프롬프트 + 3개 메모리 + 도메인 문서를 로드하면 예산이 얼마나 될까?"
   - 이유: 모델 명시(Haiku), 로드 항목 정확 → 토큰 추정 직접 진행 가능

4. "에이전트 컨텍스트 오버플로우 에러가 자주 나요. 어디가 문제일까요?"
   - 이유: 컨텍스트 오버플로우 증상 → Step 1~3을 통해 예산 재분석 후 문제 파악

5. "Context window budget plan for a research agent that dynamically loads different documentation based on user queries. Need a dynamic loading strategy."
   - 이유: 동적 로딩 전략 요청 명시 → Step 4 memory_search 기반 설계 진입

### Should NOT Trigger (5)

1. "프롬프트를 최적화해서 토큰을 줄이고 싶어요."
   - 올바른 라우팅: 프롬프트 문장 단축 → `forge/prompt` (프롬프트 CRISP 최적화)

2. "우리 에이전트 아키텍처 전체가 맞는지 검토해 주세요."
   - 올바른 라우팅: 전체 설계 검증 → `agent-plan-review` (아키텍처 최상위 레벨)

3. "어떤 모델(Haiku vs Sonnet vs Opus)을 써야 품질과 비용 밸런스가 최고일까?"
   - 올바른 라우팅: 모델 선택 최적화 → `atlas/router` (모델별 성능/비용 비교)

4. "메모리 아키텍처를 다시 설계해야 해요. 단기/장기/절차적 메모리가 어떻게 구성되어야 하죠?"
   - 올바른 라우팅: 메모리 시스템 전체 설계 → `atlas/memory-arch`

5. "컨텍스트를 줄이려고 요약본을 만들었어요. 이제 원본과 요약본을 둘 다 로드할지 말지 판단해 줄 수 있어요?"
   - 올바른 라우팅: 파일 콘텐츠 비교/선택 → 시간 투자 대비 ctx-budget 범위 밖 (도메인 특화 의사결정)

## 2) Edge Cases

### 경계 사례 (5)

1. **모델별 토큰 환산율이 다를 때 예측 오차 발생**
   - 입력: "Haiku 기준으로 계산했는데, 나중에 Sonnet으로 바꿨어요. 토큰이 더 들 거 같은데?"
   - 예상 행동: 각 모델별 환산율 재적용 → 새로운 토큰 추정 계산 → 70%/85% 임계값 재평가
   - 근거: Haiku/Sonnet/Opus마다 토큰 환산이 다름 (마크다운 1KB ≈ 250-350 tokens인데, 모델에 따라 ±10% 변동) → 모든 계산을 모델별로 분리해야 함

2. **필수/조건부 분류가 모호해지는 경우 (예: "관련성 높은 섹션"의 정의)**
   - 입력: "MEMORY.md에서 어떤 부분을 조건부로 로드할지 정의하기가 어려워요. 관련성 기준이 뭐죠?"
   - 예상 행동: 추상적 조건("관련성") → 구체적 규칙으로 변환 (키워드, 검색 스코어, 타임스탬프 등) → memory_search 조건식 재정의
   - 근거: memory_search 도구는 반드시 정량화 가능한 조건이 필요 → 예: "score > 0.7" 또는 "keyword in ['OKR', '전략']"

3. **동적 로딩 전략을 구현했는데 memory_search 도구가 없는 경우**
   - 입력: "memory_search 함수가 없어요. 그럼 동적 로딩을 어떻게 해요?"
   - 예상 행동: 동적 로딩 → 정적 로딩 + 부분 파일 로드로 fallback → 기본값을 조건부 로드에 맞춰 재설정
   - 근거: Fallback 계획 필수 → 도구 미지원 시에도 작동 가능한 대체 로딩 방식 설계

4. **70%~85% 구간에서 여러 파일을 추가해야 하는데 모두 넣으면 예산 초과**
   - 입력: "중요한 도메인 문서 3개를 더 추가해야 하는데, 지금도 75%예요. 모두 추가하면 안 돼요."
   - 예상 행동: 우선순위 재평가 → "작업 여유분" 또는 "대화 히스토리" 항목 축소 우선 검토 → 불가능하면 아키텍처 재검토 권장 (agent-plan-review 연계)
   - 근거: 컨텍스트 한도는 절대값 → 추가 파일 들이기 전에 기존 항목 재최적화 필수

5. **처음 2주 데이터 수집 후 실제 사용 패턴이 예측과 완전히 다른 경우**
   - 입력: "계획할 땐 매일 메모리를 로드할 거라고 생각했는데, 실제로는 주 1회만 로드되네요. 예산 재계산이 필요할까요?"
   - 예상 행동: 실제 로드 빈도 수정 → 토큰 예산 재계산 (로드 빈도 ↓ → 총 토큰 ↓) → 여유분 재할당
   - 근거: 초기 예측은 가정일 뿐 → 실제 데이터 기반 조정이 더 정확 → 다음 달부터 새로운 기준선으로 재설정
