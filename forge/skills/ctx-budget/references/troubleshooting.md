# Troubleshooting — ctx-budget 스킬

## 1) 컨텍스트 윈도우 사용률이 85% 이상으로 치솟는 경우

**증상:**
- "에이전트 응답 품질이 갑자기 떨어졌어요."
- "토큰 계산했을 때는 70% 안쪽이었는데, 실제로 실행하면 85% 이상이에요."
- "대화 히스토리가 길어지면 컨텍스트 오버플로우 에러가 나요."

**확인:**
- 최근 3-5회 에이전트 실행 로그에서 "실제 컨텍스트 윈도우 사용률" 측정 (token-counting API 또는 로그)
- 예상 항목별 토큰 vs 실제 측정값 비교 → 어디서 오차가 났는가?
- 대화 히스토리가 예상보다 많이 누적되었는가?
- 조건부 로드 설정이 제대로 작동했는가?

**조치:**
1. 즉시 대응: 대화 히스토리 길이 제한 설정 → 최근 5개 turn만 유지, 나머지는 요약본으로 치환
   ```python
   if context_usage > 85%:
       history = compress_history(keep_last_n=5)  # 최근 5개만
   ```
2. 원인 분석: 예상 토큰 vs 실제 측정값 표 작성
   - 예상 SOUL.md: 600 tokens, 실제: 850 tokens (오차 +250)
   - 이런 식으로 각 항목 비교 → 가장 큰 오차 항목 파악
3. 파일별 재측정: token-counting API 사용해서 정확한 토큰 값 재계산
   ```python
   import anthropic
   client = anthropic.Anthropic()
   count = client.messages.count_tokens(model="claude-sonnet-4", ...)
   print(f"Actual tokens: {count.input_tokens}")
   ```
4. Step 3 필수/조건부/제외 분류 재검토:
   - 현재 "필수"로 분류한 파일을 "조건부"로 이동 가능한가?
   - 요약본으로 대체 가능한 항목이 있는가?
5. 다음 달 예산 재협상: 새로운 기준선 기반 Step 2 재계산

## 2) 동적 로딩 전략(memory_search)이 작동하지 않는 경우

**증상:**
- "memory_search 조건을 설정했는데 제대로 된 문서가 로드되지 않아요."
- "검색 점수가 0.3 이상인 섹션만 로드하기로 했는데, 아무것도 안 나옵니다."
- "조건문이 너무 복잡해서 디버깅하기 어려워요."

**확인:**
- memory_search 도구가 실제로 구현되어 있는가? (에이전트 설정 파일에서 tools 목록 확인)
- 검색 쿼리가 명확한가? (예: 빈 쿼리 또는 너무 구체적인 쿼리)
- 검색 결과의 relevance score 분포 확인 (모두 0.1 이하면 문서와 쿼리 미스매치)
- 조건 로직이 AND/OR 으로 정확히 설정되었는가?

**조치:**
1. memory_search 도구 가용성 확인
   ```python
   available_tools = get_agent_tools()
   if "memory_search" not in available_tools:
       print("memory_search not available, using static loading")
       # Fallback: 정적 로드로 전환
   ```
2. 검색 쿼리 단순화 및 테스트
   - "OKR 또는 전략" (너무 넓음) → "Q1 OKR 목표" (구체적)
   - 1회 테스트 실행 → relevance score 분포 확인
3. 조건문 분해해서 각 조건별로 테스트
   ```python
   # 복잡한 조건 →
   if (score > 0.7 and "전략" in text) or (date >= today - 7days):

   # 분해해서 →
   result = memory_search("전략")  # score 확인
   print(f"전략 검색 결과: {len(result)} 개, 평균 score: {avg(score)}")
   # → 조건값 조정
   ```
4. Fallback 설정: memory_search 실패 시 정적 로드로 자동 전환
5. 다음 사이클: 실제 검색 결과 기반으로 조건값 재보정

## 3) 토큰 추정값과 실제 값의 큰 오차 (50% 이상)

**증상:**
- "마크다운 파일 10KB라고 생각했는데 실제로는 3,000 tokens가 아니라 5,500 tokens 나왔어요."
- "코드 파일 크기 계산이 완전히 틀렸어요."
- "모델마다 토큰이 다르게 계산되는데 어떻게 대응해야 하나요?"

**확인:**
- 파일 크기 측정 방식이 올바른가? (단순 파일 크기 vs 라인 수 vs 단어 수)
- 파일 형식별 환산율이 맞는가? (마크다운 vs 코드 vs JSON 마다 다름)
- token-counting API를 실제로 사용했는가? (추정값 vs 측정값)
- 모델별 환산율 차이를 반영했는가? (Haiku vs Sonnet)

**조치:**
1. token-counting API로 정확한 토큰 측정 (추정 대신)
   ```python
   # 추정 (부정확)
   tokens = file_size_kb * 300  # 300은 추정값

   # 측정 (정확)
   actual_tokens = count_tokens_via_api(file_content, model="claude-sonnet-4")
   ```
2. 파일별 환산율 테이블 재작성 (추정 대신 실제 측정값 기반)
   | 파일 형식 | 크기 | 실제 토큰 | 환산율 |
   | — | — | — | — |
   | Markdown | 2KB | 850 | 425/KB |
   | Python | 2KB | 600 | 300/KB |
   | JSON | 1KB | 250 | 250/KB |

3. 모델별 환산율 차이 적용 → Haiku는 Sonnet보다 ±5% 다를 수 있음
   - Sonnet 기준: 1,000 tokens
   - Haiku: 1,000 × 0.95 = 950 tokens (약간 효율적)
4. 향후 계획: 추정 대신 항상 API로 정확하게 측정

## 4) 필수/조건부/제외 분류가 모호할 때

**증상:**
- "이 파일을 필수로 분류해야 하는지 조건부로 분류해야 하는지 모르겠어요."
- "조건부 로드 기준이 자의적이거나 명확하지 않습니다."
- "매번 다르게 분류하게 되는 것 같아요. 원칙이 있을까요?"

**확인:**
- 필수: 에이전트가 매 실행마다 항상 필요한 파일인가? (SOUL.md, 오늘 메모리 등)
- 조건부: 특정 조건(키워드, 시간, 사용자)에서만 필요한 파일인가? (도메인 메모리, 관련 SKILL.md)
- 제외: 에이전트 실행에 전혀 필요 없는 파일인가? (2주 이전 메모리, 무관한 도메인)

**조치:**
1. 명확한 분류 기준 정의:
   ```
   필수: 로드하지 않으면 에이전트 기본 동작 불가
   → SOUL.md, USER.md, 오늘 메모리 (일일 데이터)

   조건부: 입력/도메인/시간에 따라 선택적 로드
   → MEMORY.md (1달 이전 요약), 도메인 메모리 (마케팅/재무)
   → 조건: keyword match or user role

   제외: 총합 파일 크기, 무관성, 보안
   → 2주+ 이전 메모리 (MEMORY.md 요약본으로 통합)
   → 다른 팀의 도메인 문서
   ```
2. 팀과 함께 분류 의사결정 미팅 → 합의 문서화
3. 월별 재검토: 실제 사용 패턴 기반으로 분류 조정

## 5) 70% 도달 후 새로운 파일 추가 요청이 들어올 때

**증상:**
- "중요한 도메인 문서를 추가해야 하는데 이미 75%라서 안 돼요."
- "다른 팀에서 SKILL.md를 추가해 달라고 하는데, 컨텍스트가 부족해요."
- "임계값을 88%까지 올려도 되지 않을까요?"

**확인:**
- 70% 임계값을 높이는 것이 성능 저하로 이어지지 않을까? (응답 품질, 레이턴시)
- 현재 "작업 여유분"으로 할당된 토큰이 충분한가?
- 기존 필수 항목 중에 축소 가능한 항목이 있는가?

**조치:**
1. 먼저 기존 항목 최적화:
   - 요약본 우선: 원문 대신 3줄 요약으로 치환 → ~80% 토큰 절감
   - 대화 히스토리 압축: 최근 10개 → 최근 5개로 제한 → ~40% 절감
   - 관련 없는 파일 제외: 다른 팀 SKILL.md는 조건부로 변경
2. 위 조치로도 부족하면, 새 파일 추가 아젠다를 다음 분기로 미루기
3. 임계값 상향(70% → 85%) 검토: 장기적으로 아키텍처 재설계 필요 (agent-plan-review 연계)
