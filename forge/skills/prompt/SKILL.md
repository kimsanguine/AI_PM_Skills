## Prompt Engineering for PMs

## Core Goal

- PM 관점에서 프롬프트를 의도 명확화와 판단 기준 정의의 관점으로 설계하여 기술자 중심의 토큰 최적화를 보완
- CRISP 프레임워크(Context/Role/Instruction/Scope/Parameters)를 체계적으로 적용하여 재사용 가능하고 일관성 있는 프롬프트 구조화
- 7가지 실패 패턴 회피 + Why-First 원칙으로 예외 상황에서도 에이전트가 의도에 맞게 판단하도록 설계

---

## Trigger Gate

### Use This Skill When

- System Prompt 또는 Agent Instruction 초안 작성 후 검토 및 최적화 필요
- 에이전트가 일관되지 않은 판단을 보이거나 예상 밖의 행동을 할 때 (프롬프트 디버깅)
- Why-First 원칙에 따라 기존 프롬프트를 재작성할 필요가 있을 때
- PM이 프롬프트의 의도를 명확히 하고 기술자에게 전달해야 할 때

### Route to Other Skills When

- 7요소 Instruction 설계 필요 → `forge/instruction` 스킬로 라우팅
- 컨텍스트 윈도우 최적화 필요 → `forge/ctx-budget` 스킬로 라우팅 (토큰 절감)
- 판단 기준 구체화 필요 → `muse/pm-engine` 스킬로 라우팅 (PM 사고 엔진)

### Boundary Checks

- CRISP 프레임워크는 최소한의 가이드라인이지, 모든 5요소를 항상 명시할 필요는 없음 (상황별 조정 가능)
- 프롬프트는 기술적 완벽성이 아닌 "의도 명확성"이 우선
- 7가지 실패 패턴은 체크리스트이지 절대 규칙은 아니며, 상황에 따라 일부는 의도적으로 허용 가능

---

## Failure Handling

| 실패 상황 | 감지 | 대응 |
|----------|------|------|
| CRISP 5요소 중 일부가 누락되어 에이전트 행동이 모호 | 프롬프트를 읽었을 때 "그럼 이 상황에서는 뭘 해야 해?"라는 질문이 남음 | 누락된 요소를 명시적으로 추가 (특히 Scope와 Parameters는 선택이 아닌 필수) |
| Anti-Goals가 있지만 예외 상황에 대한 판단 기준이 없음 | "절대 하지 말아야 할 것"은 명확한데 "해야 할 것"이 불명확 | Anti-Goals와 대응되는 "When to do"를 동시에 정의 |
| Why-First 원칙 미적용 — 에이전트가 기계적으로만 실행 | 프롬프트에 "단계적으로", "좋은 결과를 위해" 등 기술적 용어만 있고, "왜 이 작업이 중요한가"가 없음 | 각 주요 지시마다 "이유(Why)" 1줄 추가 |
| 판단 기준이 정성적이어서 에이전트가 해석에 헷갈림 | "중요한", "좋은", "신뢰도 높은" 등 정성적 표현만 사용 | 정량적 기준으로 재정의 (임계값, 점수, 예시) |
| 프롬프트가 너무 길어서 실제 작업 입력 토큰 부족 | 컨텍스트 윈도우의 50% 이상이 시스템 프롬프트 차지 | forge/ctx-budget 스킬로 라우팅하여 압축 전략 수립 |

---

## Quality Gate

- [ ] CRISP 5요소 중 최소 4개 명시 (Context/Role/Instruction/Scope/Parameters) (Yes/No)
- [ ] 목표가 명확하고 모호하지 않음 (Yes/No)
- [ ] Anti-Goals 명시 (최소 2개 이상) (Yes/No)
- [ ] 판단 기준이 정량적 또는 구체적 예시로 정의 (Yes/No)
- [ ] 7가지 실패 패턴 체크 (각 패턴 회피 여부 확인) (Yes/No)
- [ ] Why-First 원칙 적용 (주요 지시마다 "왜"가 명시) (Yes/No)
- [ ] 실패 처리 케이스 정의 (데이터 없음/불확실 시 어떻게) (Yes/No)

## Examples

### Good Example

```markdown
# Prompt — morning-briefing 뉴스 요약

## Context
당신은 이든의 정보 수집 파트너입니다.
매일 아침 8시에 자동 실행되어, 전 날 수집한 AI/에이전트 관련 뉴스를 이든에게 전달합니다.
이든은 기술 수준이 높은 PM이므로, 표피적 요약보다 PM 관점의 인사이트를 중요시합니다.

## Role
당신은 뉴스 큐레이터(PM 관점)로서,
단순 요약이 아닌 "이 뉴스가 이든의 에이전트 업무에 어떤 의미가 있는지"를 판단하고 전달합니다.

## Instruction
다음 순서로 진행하세요:

1. 입력받은 뉴스 5개를 검토합니다.
   - 왜: 관련성 있는 뉴스만 처리하기 위해
2. 각 뉴스마다 3줄 이내로 요약합니다.
   - 첫 줄: 핵심 사실
   - 둘째 줄: 기술/제품 상세
   - 셋째 줄: PM 관점 인사이트
   - 왜: PM이 빠르게 의사결정에 필요한 정보를 얻기 위해
3. 신뢰도가 낮은 기사(확인 필요한 정보)는 명시합니다.
   - 왜: 잘못된 정보로 인한 의사결정 오류 방지

## Scope
포함:
- AI 모델, 에이전트, 프롬프트 엔지니어링 관련 뉴스
- 비용 최적화, API 가격 변동 뉴스
- 경쟁사 에이전트 도입 사례

제외:
- 스팸/뉴스레터 (공식 미디어만)
- 학술 논문만 있는 기사 (실무 적용이 불명확)
- 영어 뉴스를 불완전하게 번역한 부분

제한:
- 요약은 3줄 정확히 (더 길면 안 됨)
- 신뢰도 표시는 "❓ 확인 필요" 한 글자로만

## Parameters
형식: Markdown (이모지 활용 가능)
길이: 요약 3줄 × 뉴스 5개 = 최대 15줄
언어: 한국어 (고유명사는 영문 병기)
톤: 간결하고 PM 친화적 (마케팅 톤 X, 객관적 톤)

예시:
```
📰 3월 7일 AI 뉴스 요약

1️⃣ Anthropic, Claude Opus 4.6 출시 — 프롬프트 캐싱 50% 비용 절감
기술: 컨텍스트 윈도우 캐싱으로 장기 컨텍스트 비용 획기적 감소
💡 PM 인사이트: OpenClaw의 morning-briefing 에이전트 비용 재계산 필요 (월 $50 → $25로 절감 가능 추정)

2️⃣ ...
```

---

## Anti-Goals
1. 추측 기반 요약 금지 — 불명확한 내용은 "❓ 확인 필요"로 표시
2. 마케팅 톤으로 작성 금지 — 객관적 사실 중심
3. 뉴스 전문을 요약으로 착각 금지 — 3줄은 절대 최소 단위, 확장 금지

---

## 판단 기준 테이블

| 상황 | 행동 | 이유 |
|------|------|------|
| 신뢰도 높음 (공식 보도, 다중 출처) | 그대로 요약 | 확인 필요 표시 X |
| 신뢰도 낮음 (1개 출처, 확인 불가) | "❓ 확인 필요" 표시 | 오류 정보로 인한 의사결정 오류 방지 |
| 뉴스이지만 무관함 (예: 주식 가격) | 제외 | 이든의 에이전트 업무와 무관 |
| 영어 뉴스 | 완전 번역 후 요약 | 정보 손실 방지 |

---

## 실패 처리
- 뉴스가 5개 미만 → 있는 것만 처리 ("신규 뉴스 N개" 명시)
- 신뢰도 판단 어려움 → "❓ 확인 필요" 마크
- API 오류 → "수집 실패" 알림 + 어제 상위 뉴스 대신 전송
```

### Bad Example

```markdown
# Prompt — 뉴스 요약

당신은 뉴스를 요약합니다.

좋은 요약을 만들어주세요.

3줄 이내로 정리하고, 정확하게 하되 창의적으로 작성해주세요.

결과를 Telegram으로 전송해주세요.

---

문제점:
- Context 없음 → 누가 읽을지, 어떤 뉴스인지 불명확
- Role이 추상적 ("뉴스 요약기" 만 있고, PM 관점이 없음)
- Instruction이 기술적만 있음 ("3줄", "정확하게") — 왜(Why)가 없음
- Anti-Goals 없음 → 하지 말아야 할 것이 모호
- "창의적 + 정확" = 충돌하는 요구사항 (우선순위 불명확)
- Scope (포함/제외) 정의 없음 → 어떤 뉴스를 처리할지 불명확
- 판단 기준이 정성적 ("좋은", "창의적") → 에이전트가 해석에 헷갈림
- 실패 처리 없음 → 뉴스가 없으면? 신뢰도 낮으면?
```

## Prompt Engineering for PMs

프롬프트 엔지니어링을 기술자 관점과 PM 관점에서 보면 완전히 다릅니다.

| 기술자 관점 | PM 관점 |
|---|---|
| 토큰 최적화 | 의도 명확화 |
| 구문 형식 | 판단 기준 정의 |
| Few-shot 예시 | 아웃컴 명세 |
| Temperature 조정 | Anti-Goals 설정 |

> "프롬프트는 신입 직원에게 주는 업무 지시서다.  
> 기술이 아니라 커뮤니케이션의 문제다."

---

### CRISP 프레임워크

PM이 쓸 수 있는 프롬프트 설계 5요소:

**C — Context (맥락)**
에이전트가 알아야 할 배경 정보
```
당신은 [역할]입니다.
사용자는 [누구]이고, 현재 [상황]에 있습니다.
```

**R — Role (역할)**
에이전트의 전문성과 페르소나
```
당신은 PM 20년 경력의 AI 에이전트 전문가입니다.
기술적 판단보다 '왜'를 먼저 설명하는 방식을 선호합니다.
```

**I — Instruction (지시)**
구체적인 행동 지시 — 무엇을, 어떻게, 순서대로
```
다음 순서로 진행하세요:
1. [입력] 분석
2. [프레임워크] 적용
3. [출력 형식]으로 결과 제시
```

**S — Scope (범위)**
해야 할 것과 하지 말아야 할 것
```
포함: [명시적으로 해야 하는 것]
제외: [절대 하지 말아야 하는 것 — Anti-Goals]
제한: [길이, 언어, 형식 제한]
```

**P — Parameters (파라미터)**
출력 형식, 길이, 톤, 채널
```
형식: [Markdown / Plain / JSON]
길이: [최대 N줄]
언어: [한국어]
톤: [간결하고 실용적으로]
```

### 프롬프트 실패 패턴 7가지

**Pattern 1 — 목표 모호성**
```
❌ "뉴스를 요약해줘"
✅ "오늘 AI 에이전트 관련 뉴스 5건을 각 3줄 이내로 요약하고,
    PM 관점의 인사이트 1줄을 추가해줘"
```

**Pattern 2 — Anti-Goals 누락**
```
❌ "최선을 다해 분석해줘"
✅ "분석해줘. 단, 확실하지 않은 내용은 추측하지 말고
    '확인 필요'로 표시해줘"
```

**Pattern 3 — 출력 형식 미명시**
```
❌ "결과 알려줘"
✅ "결과를 다음 형식으로 Telegram에 전송 가능한 텍스트로 작성:
    - [제목]
    - [3줄 요약]
    - [행동 제안 1개]"
```

**Pattern 4 — 컨텍스트 과부하**
```
❌ [5000줄 파일 전체 로드] + "분석해줘"
✅ 관련 섹션만 추출 후 → "다음 내용을 기준으로 분석해줘: [핵심 섹션]"
```

**Pattern 5 — 판단 기준 부재**
```
❌ "중요한 이메일을 선별해줘"
✅ "다음 기준으로 중요도를 판단해줘:
    높음: 직접 답장 필요, 마감 포함, 의사결정 요청
    중간: 참고 필요, FYI
    낮음: 뉴스레터, 마케팅, 자동 발송"
```

**Pattern 6 — 실패 처리 누락**
```
❌ "데이터를 가져와서 분석해줘"
✅ "데이터를 가져와서 분석해줘.
    데이터가 없으면: '데이터 없음'으로 표시하고 계속 진행해줘.
    API 오류 시: 오류 메시지를 포함해서 결과를 전송해줘."
```

**Pattern 7 — 역할 충돌**
```
❌ "창의적이고 정확하고 간결하게 작성해줘"
    (창의성 ↔ 정확성 ↔ 간결성은 종종 충돌)
✅ "정확성을 최우선으로, 간결하게 작성해줘.
    창의적 표현보다 명확한 전달을 선호해줘."
```

### PM 판단을 프롬프트에 녹이는 방법

PM의 핵심 가치는 **판단 기준**을 명확히 하는 것입니다.

**판단 기준 명시화 패턴:**

```
[상황 A]이면 → [행동 X]
[상황 B]이면 → [행동 Y]
판단이 어려우면 → [에스컬레이션 방법]
```

예시 (이메일 우선순위 에이전트):
```
발신자가 CEO/임원이면 → 항상 '높음'
마감일이 24시간 이내이면 → '높음'
키워드 [계약, 결제, 긴급]이면 → '높음'
뉴스레터/자동발송이면 → 자동 제외
판단이 불확실하면 → '확인 필요' 태그 추가
```

---

### Why-First 프롬프트 원칙

이든의 PM 철학 — 기술이 아닌 의도를 먼저:

```
❌ 기술 중심: "Chain-of-thought로 step-by-step 분석해줘"
✅ 의도 중심: "이 결정이 왜 중요한지 먼저 설명하고,
               그 이유를 기반으로 분석해줘"
```

에이전트가 "왜"를 이해하면:
- 예외 상황에서도 올바른 방향으로 판단
- 지시가 불완전해도 의도에 맞게 처리
- 새로운 상황에 유연하게 대응
