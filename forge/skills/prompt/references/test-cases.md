# Test Cases — prompt 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "뉴스 요약 프롬프트를 PM 관점으로 최적화해줄래? 지금은 자꾸 확신 없는 요약을 내놓고, 판단 기준이 명확하지 않아."
   - 이유: 판단 기준 명확화 필요

2. "System Prompt 토큰이 너무 많아. CRISP 프레임워크로 재구조화해서 압축해줄래?"
   - 이유: 토큰 최적화 + 명확성 개선

3. "에이전트가 자꾸 환각(거짓 정보)을 출력해. Anti-Goals와 신뢰도 판단 기준을 추가해줄래?"
   - 이유: 환각 방지 기준 명시

4. "프롬프트에서 '창의적', '정확한', '간결하게' 같은 말들이 너무 많아. 실제로 어떻게 행동해야 하는지 구체화해줄래?"
   - 이유: 정성적 표현 → 정량적 기준

5. "현재 프롬프트를 Why-First 원칙으로 재작성해줄래? 에이전트가 예외 상황에서 올바르게 판단하도록."
   - 이유: 의도 기반 설계

### Should NOT Trigger (5)

1. "7요소 Instruction을 완전히 설계해줄래?"
   - 올바른 라우팅: `forge/instruction`

2. "컨텍스트 윈도우를 70% 이하로 유지하도록 토큰 예산을 계획해줄래?"
   - 올바른 라우팅: `forge/ctx-budget`

3. "PM 암묵지를 추출해서 TK 유닛으로 만들어줄래?"
   - 올바른 라우팅: `muse/pm-engine`

4. "프롬프트 엔지니어링의 최신 기법을 알려줄래?"
   - 올바른 라우팅: 학습 자료 제공

5. "Chain-of-Thought 프롬프팅 기법을 적용해줄래?"
   - 올바른 라우팅: 기술적 구현 (프롬프트 스킬이 아닌 개발자 작업)

## 2) Functional Tests (Given-When-Then)

### Test 1: CRISP 5요소 적용

**Given:**
- 초안 프롬프트: "뉴스를 잘 요약해줄래?"

**When:**
- CRISP 프레임워크 적용 요청

**Then:**
- Context: "당신은 PM 담당자의 정보 수집 파트너입니다. 매일 아침 8시 자동 실행..."
- Role: "뉴스 큐레이터(PM 관점)"
- Instruction: "3줄 이내 요약 + PM 인사이트"
- Scope: "포함: AI 뉴스, 제외: 스팸"
- Parameters: "형식: Markdown, 길이: 3줄, 언어: 한국어"

---

### Test 2: 판단 기준 정량화

**Given:**
- 초안 기준: "신뢰도 높은 기사", "정확한 요약"

**When:**
- 정량적 기준 재정의 요청

**Then:**
- 신뢰도 높음: "공식 보도, 다중 출처, 확인 가능"
- 신뢰도 낮음: "1개 출처, 확인 불가" → "❓ 확인 필요" 표시
- 정확함: "원본 의미 손실 < 5%" → 기사 3줄 정확히

---

### Test 3: 7가지 실패 패턴 체크

**Given:**
- 프롬프트 초안

**When:**
- 7가지 패턴(목표 모호성/Anti-Goals 누락/출력 형식 미명시/컨텍스트 과부하/판단 기준 부재/실패 처리 누락/역할 충돌) 검사

**Then:**
- Pattern 1 (목표 모호성): "뉴스를 잘 요약" → "AI 에이전트 관련 뉴스 5건을 3줄 이내로 요약하고, PM 관점 인사이트 1줄 추가"
- Pattern 2 (Anti-Goals): "절대 확신 없는 내용을 사실처럼 전달하지 마세요" 추가
- Pattern 3 (출력 형식): "Telegram Markdown, 최대 500자" 명시
- Pattern 4 (컨텍스트 과부하): 프롬프트 1000자 이내 확인
- Pattern 5 (판단 기준): 신뢰도별 행동 테이블 추가
- Pattern 6 (실패 처리): "뉴스 5개 미만 → 있는 것만 처리" 명시
- Pattern 7 (역할 충돌): "정확성 > 창의성" 명확히

## 3) Error Cases

### Error 1: Anti-Goals 부재로 인한 환각

**상황:**
- 프롬프트에 목표만 있고 Anti-Goals 없음
- 에이전트가 확실하지 않은 내용도 사실처럼 전달

**대응:**
- Anti-Goals 추가: "확신 불가능하면 '❓ 확인 필요' 표시", "추측은 '의견'으로 명시"

---

### Error 2: 판단 기준 정성적으로 인한 해석 차이

**상황:**
- "중요한 정보는 강조해주세요"
- 에이전트마다 다르게 해석

**대응:**
- 정량적 기준 제시: "마감일 포함, CEO 발신, 키워드[계약/결제/긴급] → 항상 강조"

## 3) Edge Cases

| # | 쿼리 | 판정 | 이유 |
|---|------|------|------|
| E1 | "프롬프트 최적화를 하고 싶은데, 지금 에이전트가 이미 잘 작동 중이야. 개선의 필요성이 있을까?" | ⚠️ 경계 | "작동한다" ≠ "최적화된다". 만약 "토큰이 많거나" "판단 기준이 모호하거나" "환각이 있으면" prompt 스킬 유효 |
| E2 | "프롬프트를 한국어로 다시 쓸지, 영어로 쓸지 결정해야 해. 이것도 prompt 스킬 범위인가?" | ⚠️ 경계 | prompt 스킬은 "언어 선택"이 아니라 "명확성 개선"에 집중. 다만 "한국어 프롬프트가 더 정확하다" 판단 후라면 prompt 스킬로 한국어 재작성 가능 |
| E3 | "프롬프트에 "창의적으로 하기"라는 표현이 있는데, 이걸 구체화하려면 어떻게 해야 할까?" | ✅ Trigger | 정성적 표현을 정량적 기준으로 변환은 prompt 스킬의 핵심. "창의적 = 3가지 이상의 새로운 아이디어 제시"로 정의 |
| E4 | "프롬프트 토큰이 많아서 비용이 높아. 압축하고 싶은데 품질을 안 잃을 수 있을까?" | ✅ Trigger | CRISP 프레임워크로 재구조화하면 "불필요한 예시 제거 + 핵심만 남기기" 가능. 하지만 극단적 압축은 prompt 스킬보다 ctx-budget 스킬이 더 적합 |
| E5 | "프롬프트에 "사실만 말하고 추측은 하지 마"라고 썼는데, 에이전트가 계속 추측 같은 답을 해. 프롬프트 문제일까?" | ✅ Trigger | Anti-Goals 부재 또는 정의 불명확. prompt 스킬로 "추측 기준" 재정의: "신뢰도 < 0.7이면 '❓ 확인 필요' 표시" 등 구체화 |

## 4) With/Without Skill 비교

### 시나리오: CRISP 없이 프롬프트만 작성

**Without Prompt Skill:**
- 프롬프트: "뉴스를 잘 요약해줄래?"
- 에이전트 행동: 예측 불가능, 매번 다른 해석

**With Prompt Skill:**
- Context/Role/Instruction/Scope/Parameters 명확
- Why-First로 의도 기반 설계
- 예외 상황에서도 일관된 행동
