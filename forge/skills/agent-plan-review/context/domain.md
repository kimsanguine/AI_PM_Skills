# Domain Context — agent-plan-review 스킬

## 1) Domain Scope

**이 스킬이 소유하는 영역:**
- 구현 전 에이전트 설계의 4축(범위, 아키텍처, 인스트럭션, 신뢰성) 검증
- Step 0(범위 점검), 아키텍처 리뷰, 인스트럭션 품질, 운영 신뢰성, 비용 & 스케일 리뷰
- 장애 모드 매트릭스 작성 및 치명적 gap 식별
- 의견이 담긴 권고안과 트레이드오프 제시

**이 스킬이 소유하지 않는 영역:**
- Instruction 7요소 설계 자체 (→ `forge/instruction`)
- 컨텍스트 윈도우 예산 계획 (→ `forge/ctx-budget`)
- 비용 시뮬레이션 (→ `oracle/cost-sim`)
- 신뢰성/SLO 정의 (→ `argus/reliability`)

## 2) Primary Users

- **PM/리드 엔지니어**: 구현 전 설계 검증 및 go/no-go 판단
- **기술 리더**: 아키텍처 결정의 트레이드오프 확인
- **운영 담당**: 장애 모드와 모니터링 계획 수립

## 3) Required Inputs

**필수 입력:**
1. 에이전트 이름
2. PRD 또는 설계 문서 (최소 Instruction 정의)
3. 아키텍처 다이어그램 (있으면)

**선택 입력:**
- 현재 프로토타입 코드
- 비용 예산 제약

## 4) Output Contract

**산출물:**
- Step 0 범위 점검 결과 (3가지 질문 답변)
- 4개 섹션 리뷰: 각 섹션에 이슈 번호 + 옵션 + 권고안
- 장애 모드 매트릭스 (최소 5개 유형)
- 완료 요약 (이슈 수, 범위 외, 기존 자산, 미결정 사항)

| 항목 | 보증 |
|------|------|
| Step 0 | 3개 질문 답변 완료 |
| 범위 옵션 | 축소/전체/압축 중 선택 |
| 다이어그램 | 필수 다이어그램 2개 이상 |
| 장애 모드 | 5개 이상, 치명적 gap 수 명시 |
| 미결정 사항 | 있으면 명확히 기록 |

## 5) Guardrails

**라우팅 규칙:**
- Instruction 미완성 → `forge/instruction` 먼저 완료
- 컨텍스트 윈도우 최적화 → `forge/ctx-budget`
- 비용 추정 → `oracle/cost-sim`
- 신뢰성/SLO → `argus/reliability`

**품질 기준:**
- Step 0은 절대 건너뛰지 마라
- 장애 모드 매트릭스는 최소 5개 유형 필수
- 영향도 높음/매우높음인데 설계 미반영된 항목은 "치명적 gap"으로 표시

## 6) Working Facts

**리뷰 시간 추정:**

| 범위 | 시간 (예시값, 도메인에 따라 조정) | 섹션 깊이 | 산출물 규모 |
|------|------|---------|-----------|
| 축소 | 30분 | 최소화 (MVA 중심) | 1~2페이지 |
| 전체 | 120분 | 4개 섹션 × 4개 이슈 | 8~12페이지 |
| 압축 | 60분 | 핵심만 (이슈 1개 + 장애 모드) | 4~6페이지 |

**복잡도 경고 신호 (예시값, 도메인에 따라 조정):**
- 에이전트 3개 이상 오케스트레이션
- 도구 5개 이상 연결
- 컨텍스트 윈도우 50% 이상 차지
- 외부 API 3개 이상

**TO BE UPDATED by reviewer:**

- 실제 장애 사례 기반 모드 목록 확대
- 치명적 gap의 기준 명확화

## 8) 참고 사례: Skill Creation Gate의 검증 체계

> **아래는 특정 프로덕션 환경에서의 사례입니다. 조직과 도메인에 따라 다르게 설계할 수 있습니다.**

### Core-5 하드 검증 패턴 (내부 프로세스 추천 — 조직에 맞게 조정 가능)

agent-plan-review의 검증 프로세스는 Skill Creation Gate와 동일한 5가지 필수 요소를 강제한다:

| 필수 요소 | 검증 방식 | agent-plan-review의 적용 |
|----------|---------|------------------------|
| **owner** | 책임자 명시 (Not TBD) | 리뷰 주관자 이름 또는 팀 지정 필수 |
| **due_date** | 검증 완료 기한 명시 | Step 0 범위 점검 → 본 리뷰 → 산출물 제출의 3단계 일정 필수 |
| **problem_statement** | 3요소 모두 명시: "현재 상태" + "바라는 상태" + "왜 중요한가" | 에이전트 설계의 문제점 또는 미확인 영역 3가지 이상 명시 필수 |
| **priority** | P0/P1/P2/P3 등급화 (우선순위 명확) | 장애 모드 매트릭스 중 치명적 gap을 P0로, 나머지를 P1/P2로 구분 |
| **trace_links** | 관련 문서/PR/이슈로의 역추적 가능 | 대상 에이전트의 Instruction, PRD, 기존 리뷰 문서로 링크 필수 |

**Core-5 검증 실패 시**: 리뷰 진행을 중단하고 5가지 요소 모두를 명시하도록 요청.

### 피드백 개선 규격: What/Why/Fix/Example + Good/Bad Example 동시 제공

리뷰 결과 각 이슈마다 4-튜플 피드백:

```
이슈 #N: [문제 제목]

What: [무엇이 문제인가? 객관적 관찰]
- 장애 모드 매트릭스에 5개 미만의 유형만 식별됨
- 영향도 높음/매우높음인데 설계에 미반영된 항목이 3개 있음

Why: [왜 중요한가? 영향도 설명]
- 미반영된 장애 모드는 프로덕션 환경에서 예상치 못한 장애로 이어짐
- 구현 후 반은 설계 변경이 필요한 비용 발생

Fix: [어떻게 해결할까? 액션 아이템]
1. 도구 호출 실패 → API 타임아웃, 인증 실패, Rate Limit 구분하여 3개 추가
2. 메모리 접근 실패 → 파일 없음, 손상, 권한 부족으로 세분화
3. 각 항목에 "설계에 반영했는가?" 컬럼 추가해서 Yes/No 명시

Example:
- Good: "메모리 로드 실패(확률: 낮음, 영향도: 높음): Fallback 텍스트 메모리 사용"
- Bad: "메모리 실패가 발생할 수 있음" (누가? 어떻게? 대응은?)
```

### 반복 실패 시 피드백 우선순위 재정렬 로직

사용자가 같은 이슈로 3회 이상 리뷰 실패 시:

1. **첫 번째 실패**: "상세 피드백 제공 + Fix 액션 아이템 명시"
2. **두 번째 실패**: "이전 피드백 + 추가 설명 + 참고 사례 제시"
3. **세 번째 실패**: 우선순위 재정렬
   - **critical**: 치명적 gap (구현 중단 필요)
   - **non_critical**: 개선 사항 (차기 버전 가능)
   - 불완전한 이슈는 "해결 불가능"으로 표시하고 문서화하기로 전환

**예**:
- 처음: "장애 모드 5개 미만" → P1로 지적
- 두 번째 실패: 구체적 모드 추가 사례 5개 제시
- 세 번째 실패: "이 부분은 운영 단계에서 모니터링해서 보완하자"는 차기 계획으로 변경

### 적용 교훈: 에이전트 계획 검토에서 Gate 검증이 왜 중요한가

**문제 상황**: 대부분의 팀이 리뷰를 진행하지만, 정량화되지 않아 "누군가는 뭔가 봤지만 뭘 봤는지 모름"

**Gate 검증의 역할**:
- Owner 명시 → 책임 추적성 보장 (누가 이 리뷰를 주도하는가?)
- Due Date 명시 → 리뷰가 구현을 지연하지 않도록 일정 관리
- Problem Statement → 리뷰의 목표가 명확 (대상 에이전트의 어느 부분을 검증하나?)
- Priority → 어떤 gap을 먼저 해결할지 명시적 우선순위
- Trace Links → 리뷰 결과가 Instruction, PRD, 구현 코드로 추적 가능

**실제 효과**:
- 구현 중 "아, 이건 이미 리뷰에서 논의했네"라는 재작업 30% 감소
- 차기 유사 에이전트 리뷰 시 이전 리뷰의 장애 모드 재사용 가능 (속도 2배)
