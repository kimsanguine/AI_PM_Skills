# Troubleshooting — prd 스킬

## 1) 섹션 간 충돌 (Instruction vs Output 불일치)

**증상:**
- Section 2 Instruction에는 "상세하고 복잡한 분석 제시"
- Section 6 Output에는 "최대 1000자 간결함"
- 실제 구현 시 두 요구사항이 충돌

**확인:**
- Section 2와 Section 6을 읽어보면 모순이 명확한가?
- "상세함"과 "간결함" 중 어느 것이 우선인가?

**조치:**
1. 어느 것이 정답인지 재결정 (사용자와 협의)
2. 양쪽 모두 업데이트 (일관성 확보)
   - 예시: "정확성 우선 간결함" → Output 최대 1500자로 상향, 또는 "간결함 우선" → Instruction에 "5줄 이내 핵심만 제시" 명시

## 2) Tools 섹션 불완전성

**증상:**
- Tools 섹션에 도구 목록과 용도만 있음
- "언제 쓸지", "얼마나 쓸지" 조건 명시 안 됨
- 구현팀이 모든 상황에서 도구를 과다 호출

**확인:**
- 각 도구에 "사용 조건" 컬럼이 있는가?
- 각 도구에 "호출 제한" 컬럼이 있는가?

**조치:**
- forge/instruction 스킬로 라우팅
- 각 도구별 상세 조건 정의:
  ```
  | 도구 | 용도 | 사용 조건 | 호출 제한 |
  |------|------|----------|---------|
  | web_search | 최신 정보 | 내부 캐시 부족 시만 | 최대 5회/실행 |
  ```

## 3) Failure Handling 추상화

**증상:**
- Failure Handling에 "데이터 오류 발생 시 처리"
- 실제로 어떤 데이터 오류인지 명확하지 않음
- 어떻게 대응할지도 구체적이지 않음

**확인:**
- 각 실패 시나리오가 구체적인 상황인가?
- 대응 방법이 actionable(실행 가능)한가?

**조치:**
- 추상적 시나리오 → 구체적 에러 상황으로 재정의:
  ```
  ❌ "데이터 오류 발생 시 처리"
  ✅ "API 응답 없음 → 3회 재시도 (30초 간격) → 실패 시 Fallback 값 반환"
  ```

## 4) Success Metrics 측정 불가능성

**증상:**
- Success Metrics: "정확도 높음"
- 문제: 수치, 측정 방법, 기한이 없음
- 실제로 달성했는지 검증 불가

**확인:**
- 각 지표가 정량화되었는가? (%, 초, $, 건수)
- 측정 방법이 명시되었는가?
- 달성 기한이 명시되었는가?

**조치:**
- OKR 스킬과 연계하여 재정의:
  ```
  ❌ "정확도 높음"
  ✅ "정확도 95% 이상 (API 청구 데이터 vs 리포트 비용 오차 < 1%) by 2026-06-30"
  ```

## 5) MVP vs Full PRD 선택 기준 모호

**증상:**
- 어떤 에이전트는 1페이지 PRD, 어떤 건 7페이지 PRD
- 언제 MVP PRD를 쓸지, 언제 Full PRD를 쓸지 불명확
- 팀 내 표준화 불가

**확인:**
- 프로토타입 단계vs 구현 단계 구분이 명확한가?
- 에이전트 복잡도(도구 수, 오케스트레이션)가 기준으로 작용하는가?

**조치:**
- 명시적 기준 제시:
  - **MVP PRD (1~2페이지)**: 프로토타입 단계, 단일 에이전트, 도구 ≤3개
  - **Full PRD (7섹션)**: 본격 구현 단계, 복잡도 높음, 또는 배포 예정
- 스킬 호출 시 기준 명시 필요: "/write-agent-prd [이름] --mvp" vs "/write-agent-prd [이름] --full"
