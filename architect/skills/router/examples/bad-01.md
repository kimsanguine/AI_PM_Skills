# 나쁜 예제 — Model Router

## 사용자 요청
"비용을 줄이려고 거의 모든 고객 질문을 T1 모델로 보내고 싶어. 만약 답변이 이상하면 사용자가 피드백 줄 테니까."

## 거절 이유
- 라우팅 규칙이 **과도하게 보수적** (T1 사용률 목표 >80%)
- 실제로는 많은 작업이 T1 능력 범위를 넘음 (복잡한 진단, 전략)
- 폴백 없이 T1 실패 시 사용자 경험 악화 → 신뢰도 하락
- 품질 목표(>90%)를 달성 불가능

## 올바른 라우팅
- **Trigger Gate**: 작업 복잡도에 따라 T1/T2/T3 혼합 사용
- **Fallback 필수**: T1 실패 → T2 → T3 자동 재시도
- **Quality Gate**: 최종 출력이 >90% 품질 충족 확인 후만 반환

## 수정 방향
"비용을 줄이면서도 품질을 유지하려면?" → 라우팅 규칙을 데이터 기반으로 수립
"T1 모델이 부족해 보이면?" → 폴백 전략으로 자동 상향 조정
"실제로는 어느 정도를 T1/T2/T3에 할당할 수 있을까?" → 파일럿 테스트로 라우팅 정확도 검증
