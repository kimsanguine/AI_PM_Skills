# 나쁜 예제 — Growth Loop

## 사용자 요청
"우리 에이전트를 출시했는데, 사용자들이 쓰면 우리가 데이터를 모으고 자동으로 개선되는 구조가 생기겠지?"

## 거절 이유
- 플라이휠 설계가 **명시적이지 않음** (희망사항만 있음)
- 데이터 수집 방법이 정의되지 않음 (implicit 가정)
- 데이터 → 개선의 변환 과정이 없음 (자동 최적화 불가능)
- Anti-loop 위험 (Privacy 거부, 초기 사용 부족으로 데이터 미수집)

## 올바른 라우팅
- **Trigger Gate**: 데이터 수집 방법, 처리 파이프라인, 개선 메커니즘을 **구체적으로 설계**해야 함
- **Memory-arch와 연계**: 어떻게 메모리에 데이터를 저장하고 검색할지
- **Biz-model과 연계**: 데이터 처리 비용이 초과하지 않는지 검증

## 수정 방향
"사용자 피드백을 명시적으로 수집하고, 주 1회 자동 분석해서 프롬프트 개선하는 구체적 파이프라인을 설계하려면?" → Growth-loop 스킬 사용
"초기 사용자가 없을 때 루프를 시작하려면?" → Seed data, Manual override로 cold start 전략 수립
