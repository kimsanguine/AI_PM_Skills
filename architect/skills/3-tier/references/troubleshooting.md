# Troubleshooting — 3-tier 스킬

## 1) Worker 응답 타임아웃

**증상:**
- "Worker가 지정된 시간 내에 응답하지 않는다"
- Atlas가 계속 기다림 (전체 파이프라인 중단)
- 결과 지연

**확인:**
- 타임아웃 설정값은 적절한가? (예: 5분?)
- Worker의 평균 처리 시간 파악
- 네트워크 지연이나 외부 API 의존성 있나?

**조치:**
1. 타임아웃 설정 재검토:
   - 현재: 5분
   - Worker 평균 처리 시간: 3분
   - 재설정: 타임아웃 = 평균 × 2 = 6분 (안전 여유)
2. 자동 재시도 구현:
   - 1차 타임아웃 → 2분 대기 후 재시도
   - 2차 재시도 실패 → 대체 Worker로 전환
3. 모니터링 강화:
   - Worker 응답 시간 추적 (P95, P99)
   - 지정된 타임아웃에 가까워지면 알림
4. 병렬 처리:
   - 가능하면 여러 Worker를 병렬로 실행
   - 첫 번째 완료된 결과 사용 (경쟁 모델)

## 2) Worker 출력 형식 불일치

**증상:**
- "Worker가 예상과 다른 형식으로 결과를 반환한다"
- Atlas의 파싱이 실패한다
- 파이프라인이 중단된다

**확인:**
- Worker 출력 형식 명세 (스키마) 확인
- 실제 반환된 형식 점검
- 형식 검증 로직 존재 여부

**조치:**
1. 출력 형식 명세 강화:
   ```
   Worker Output Schema:
   {
     "status": "success|error",
     "data": { ... },
     "confidence": 0.0-1.0,
     "error_message": "optional"
   }
   ```
2. Atlas의 파싱 및 검증:
   - JSON schema 검증 추가
   - 필수 필드 확인
   - 필드 타입 검증
3. 형식 불일치 시 처리:
   - 1차: Worker에 "형식 오류, 재작업" 요청
   - 2차: 예비 Worker로 대체
   - 3차: 수동 개입 요청 (Prometheus)
4. Worker 교육:
   - 명확한 출력 형식 문서화
   - 테스트 케이스 제공

## 3) Prometheus의 모순된 목표

**증상:**
- "완료 기한은 1시간인데 완벽한 품질 보장하라고 한다"
- "비용은 $100 이하인데 최고 정확도를 원한다"
- 목표가 상충해서 어느 것을 우선해야 할지 모른다

**확인:**
- Prometheus가 명시한 제약:
  - Deadline: 1시간
  - Quality: "완벽한" (정의 불명확)
  - Budget: $100
- 이 제약들이 동시에 달성 가능한가?

**조치:**
1. Atlas가 Prometheus에 명확화 요청:
   ```
   목표 충돌 감지:
   - Deadline 1시간 + Perfect Quality는 동시 불가능
   - 우선순위 명확화 필요:
     1. Deadline 우선 (1시간 완료, Quality 80% 수용)
     2. Quality 우선 (2시간 필요, 완벽성 추구)
   ```
2. Prometheus의 재정의 대기:
   - "1시간 완료하되 Quality 80% 수용" 또는
   - "2시간 기한, Quality 95%"
3. 그 사이 이전 단계까지만 진행:
   - 충돌 해결 전까지 전체 파이프라인 중단 또는
   - 이전 단계의 부분 결과 반환
4. 향후 개선:
   - Prometheus의 목표 템플릿에 제약 명확화 항목 추가
   - "Quality/Deadline/Cost 중 우선순위 2개 선택" 규칙

## 4) 최종 결과 품질 미달

**증상:**
- "모든 Worker가 작업을 완료했는데 최종 결과가 Prometheus 기준을 충족하지 않는다"
- "무엇이 부족한가?" 불명확
- 결과를 반환할 것인가, 재작업할 것인가?

**확인:**
- 품질 게이트 기준 명확한가?
  - "완성도 80% 이상" (명확)
  - "괜찮은 수준" (불명확)
- 어느 부분이 미달했나? (컨텐츠? 형식? 정확도?)

**조치:**
1. 품질 게이트 명확화:
   ```
   Quality Gate Checklist:
   □ 모든 5개 경쟁사 데이터 포함
   □ 각 섹션 500자 이상
   □ 이미지 3개 이상
   □ 마크다운 형식 올바름
   ```
2. 재작업 사이클 구현:
   - 1차 실패 → 개선 사항 명시 + 재작업 요청
   - 2차 실패 → 다른 Worker 시도 또는 Prometheus 수동 개입
   - 최대 3회 반복 (리소스 제한)
3. 부분 결과 반환 옵션:
   ```
   "결과: 80% 완성
    미달 사항: [list]
    추천: 수동 검수 또는 추가 작업"
   ```
4. Prometheus에 정직한 보고:
   - "기한 내 완성했으나 품질 기준 미달"
   - "추가 시간 필요" vs "현재 결과 수용"

## 5) Workers 간 데이터 불일치

**증상:**
- "3개의 Worker가 동시에 작업하는데 데이터가 안 맞는다"
- "Worker A의 결과와 Worker B의 결과가 상충한다"
- "통합 결과의 일관성이 없다"

**확인:**
- Workers가 동일한 입력 데이터를 받았나?
- 각 Worker의 결과 검증 여부
- 결과 통합 로직 (단순 병합인가, 조정 로직인가?)

**조치:**
1. 입력 데이터 동일성 보장:
   - Prometheus → Atlas → 모든 Workers에게 동일 복사본 전달
   - 데이터 버전 명시 (V1.0, 2026-03-08)
2. Worker 결과 검증:
   - 각 Worker 결과에 confidence score 포함 요청
   - 상충 감지 (Worker A: "경쟁사 1위", Worker B: "경쟁사 3위")
3. 결과 통합 및 조정:
   ```
   결과 상충 시:
   1. Confidence 높은 것 우선
   2. 같은 confidence면 중앙값 선택
   3. 여전히 상충하면 Prometheus에 보고
   ```
4. Worker 교육:
   - 데이터 인용 출처 명시
   - 결과의 신뢰도 명확히 표현
