# Troubleshooting — memory-arch 스킬

## 1) 메모리 주입으로 인한 컨텍스트 오버플로우

**증상:**
- API 응답 시간이 갑자기 2초 → 5초 이상으로 증가
- 토큰 사용량 급증
- 에러: "context window exceeded" 또는 요청 timeout

**확인:**
- 메모리 예산 계산: (메모리 주입 토큰 수 / 전체 컨텍스트) × 100% = ?
  - 30% 이상이면 과도함
- 주입 메모리 구성: Episodic / Semantic / Procedural 각각 몇 토큰?
- 검색 결과 순위: 가장 관련성 낮은 기억들도 주입되고 있는가?

**조치:**
1. **메모리 예산 축소**: 30% → 20%로 조정 (또는 15% 극단적 축소)
2. **우선순위 재조정**: Episodic > Semantic > Procedural 순으로 감소시키기
3. **덜 중요한 기억 제거**: 최근 30일 미사용 메모리는 아카이브로 이동
4. **메모리 요약 추가**: 여러 에피소드를 "지난 분기에는 고객 X로부터 Y의 요청이 많았다" 형태로 요약
5. **검색 정확도 향상**: 임계값 상향 (0.7 → 0.8)으로 덜 관련성 높은 메모리만 주입
6. **LLM 프롬프트 단순화**: 시스템 프롬프트에서 불필요한 설명 제거

## 2) 메모리 검색 결과가 관련성 없음

**증상:**
- 사용자: "CEO가 뭐라고 했는데?"
- 에이전트: "아, 3개월 전 인턴십 프로그램 얘기요!" (전혀 다른 내용)
- 시간순 검색 필요한데 유사도 검색만 작동

**확인:**
- 현재 검색 메커니즘: 무엇만 사용 중인가? (Semantic similarity only? / Recency?)
- 메타데이터: 메모리에 timestamp, 우선도, 태그가 제대로 붙어 있는가?
- 쿼리 파싱: "CEO가 뭐라고"에서 keyword 추출이 제대로 되는가? ("CEO", "recent")

**조치:**
1. **Recency weighting 추가**: 유사도만 보지 말고, "최근 것일수록 점수 +10%" 가중치 추가
2. **명시적 키워드 필터링**: "지난주", "CEO", "회의" 같은 키워드로 먼저 필터, 그 안에서 유사도
3. **메타데이터 강화**: 각 메모리에 (timestamp, category, priority, keyword tags) 모두 포함
4. **쿼리 재작성**: 사용자 입력을 자동으로 구조화 ("CEO"라는 entity + "recent"라는 시간 제약)
5. **Fallback 검색**: 정확한 결과 0개면 → 덜 엄격한 조건으로 재검색

## 3) 메모리 모순: 새 정보가 기존 정보와 충돌

**증상:**
- 사용자: "이전엔 이렇게 말했잖아?"
- 에이전트가 서로 다른 정보를 주면 신뢰도 하락
- 후속: "역시 AI는 일관성이 없네"

**확인:**
- 메모리 버전 관리: 동일 entity (예: "고객 X의 선호도")에 대해 여러 버전이 있는가?
- 모순 감지 로직: 새 정보가 기존 정보와 <90% 일치하면 경고하는가?
- 갱신 정책: 새 정보가 들어오면 즉시 덮어쓰는가, 아니면 검토 후인가?

**조치:**
1. **메모리 버전 관리 시스템**: 각 메모리에 (version, updated_at, created_by) 필드 추가
   - 예: "User preference v2 (updated 2026-03-01 by system)" vs "v1 (2026-02-01)"
2. **모순 감지 로직**: 새 정보 주입 시 기존 메모리와 유사도 비교
   - 유사도 <0.9이고 같은 entity면 → 경고 또는 merge review 필요
3. **사용자에게 확인 요청**: 모순 감지 시 "이전엔 A라고 했는데, 이제는 B인가요?" 물어보기
4. **메모리 Deprecated 마킹**: 기존 버전을 삭제 대신 "deprecated" 플래그 → 필요시 추적 가능
5. **버전 선택 규칙**: "가장 최신" vs "사용자가 명시적으로 정정한 것"을 우선할지 결정

## 4) 개인정보 누수: 공유 메모리에 민감한 데이터 저장

**증상:**
- User A의 개인 정보가 User B에게 노출됨
- 또는 공개 환경에서 민감한 고객 데이터가 노출됨
- "심각한 보안 사고"

**확인:**
- 메모리 저장소 구조: 사용자 격리가 제대로 되어 있는가?
  - 예: `/memory/user_123/episodic.json` (격리됨) vs `/memory/all_users.json` (노출)
- 접근 제어: Vector DB 또는 PostgreSQL에 user_id 필터가 있는가?
- 기존 메모리 감사: 이미 저장된 데이터 중 민감한 정보가 있는가?

**조치:**
1. **즉시 격리 로직 추가**: 모든 메모리 쿼리에 `WHERE user_id = $current_user` 필터 추가
2. **기존 데이터 감사**: 지난 3개월 메모리 중 민감 정보(전화번호, 계좌번호, 주민번호) 자동 검사
3. **민감 정보 마스킹**: 저장 시점에 PII(Personally Identifiable Information) 마스킹
   - 예: "010-1234-5678" → "010-****-****"
4. **접근 권한 재검토**: 기존 메모리에 대한 접근 권한 audit log 확인
5. **사용자 교육**: "에이전트는 민감한 정보를 저장하지 않습니다" 명시
6. **투명성 제공**: 사용자가 "내 메모리 보기" 기능으로 무엇이 저장되었는지 확인 가능하게

## 5) 메모리 라이프사이클 관리 실패: Cold storage 메모리가 계속 회수됨

**증상:**
- 90일 이상 미사용 메모리가 여전히 매번 쿼리됨
- 또는 반대로, 아직 유용한 메모리가 삭제됨
- 메모리 효율이 개선되지 않음

**확인:**
- 현재 라이프사이클: "New → Active → Archive → Cold"의 기간이 정의되어 있는가?
- 자동화: 언제 Archive로 이동하는지 자동으로 판단되는가, 아니면 수동인가?
- 우선도 재조정: Archive 메모리를 쿼리할 때 점수 패널티가 있는가?

**조치:**
1. **라이프사이클 정책 명시화**:
   - New: 0~7일, 검색 점수 × 1.0 (최우선)
   - Active: 8~30일, 검색 점수 × 0.9
   - Archive: 31~90일, 검색 점수 × 0.5
   - Cold: 90일+, manual access only (자동 쿼리 제외)
2. **자동 마이그레이션**: 일일 배치 작업으로 미사용 메모리 자동 아카이브
3. **수동 복구 기능**: 사용자가 "예, 이건 자주 쓸 것 같은데?"하면 Active로 재활성화
4. **용량 관리**: Cold storage 데이터가 일정 크기 이상이면 자동 정리 (최대 1년까지만 보관)
5. **사용 패턴 분석**: 실제로 Archive/Cold 메모리가 얼마나 회수되는지 측정, 라이프사이클 기간 조정
