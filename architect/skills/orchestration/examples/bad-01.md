# 나쁜 예제 — Orchestration

## 사용자 요청
"우리 고객 피드백 시스템을 보니 긍정/부정/중립 3가지 감정으로 분류돼야 하는데, 각각을 다른 에이전트가 맡으면 좀 더 정확할까?"

## 거절 이유
- 동일 입력에 대해 3개 작업이 **독립적**이므로 Sequential이 아님
- 실제로는 단일 멀티테스크 에이전트 또는 Parallel Fan-out이 더 효율적
- 오케스트레이션 패턴 선택 스킬보다는 **router** 또는 단일 **prompt 설계** 범위

## 올바른 라우팅
- 3가지 감정 분류를 한 모델에서 수행 → `router` 스킬의 T1 (간단한 분류)
- 또는 단일 멀티라벨 프롬프트로 충분 → 오케스트레이션 불필요

## 수정 방향
"3가지 분류 작업을 병렬로 실행하고 결과를 집계하려면?" → Parallel Fan-out 패턴으로 재질문
또는
"감정 분류 후 각 감정별로 서로 다른 처리를 해야 하려면?" → Router 패턴으로 재질문
