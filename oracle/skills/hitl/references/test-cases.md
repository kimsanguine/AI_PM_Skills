# Test Cases — hitl 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "고객 이메일 자동 응답 에이전트를 만드는데, 어디에 인간이 개입해야 할지 설계해 줄 수 있어?"
   - 이유: 에이전트 워크플로우(이메일 응답) 명확, HITL 설계 요청 명시 → Step 1 작업 목록부터 시작

2. "우리 결제 처리 에이전트의 HITL을 설계해 줄 수 있어요? 오류가 발생하면 인간이 개입해야 합니다."
   - 이유: 금전 관련 작업 + 오류 처리 → 2축 매트릭스 기반 자동화 레벨 결정 필수

3. "How do I design a human-in-the-loop strategy for our data deletion agent? This is sensitive."
   - 이유: 센시티브한 작업(데이터 삭제) → Level 2 (Suggest) 또는 Approval Gate 패턴 설계 진입

4. "초기 배포 직후 에이전트 품질 검증을 위한 Shadow Mode를 어떻게 설계할까요?"
   - 이유: 배포 전 검증 전략 → Step 5 Shadow Mode 계획 설계

5. "에이전트가 점점 더 자율적으로 작동하게 하려고 합니다. Level 3에서 Level 4로 어떻게 전환할까요?"
   - 이유: 자동화 레벨 전환 전략 → 신뢰도 기반 단계적 상향 설계

### Should NOT Trigger (5)

1. "우리 에이전트 아키텍처를 전체 검토해 줄 수 있어요?"
   - 올바른 라우팅: 전체 설계 검증 → `agent-plan-review`

2. "에이전트가 너무 많은 오류를 내요. 어떻게 정확도를 높일까요?"
   - 올바른 라우팅: 정확도 개선 → `agent-plan-review` 또는 기술 최적화 스킬

3. "우리 모니터링 대시보드를 만들어 줄 수 있어요?"
   - 올바른 라우팅: 모니터링 도구 구축 → 기술 구현 스킬 (HITL 설계 아님)

4. "에이전트의 신뢰도를 A/B 테스트로 측정하고 싶어요."
   - 올바른 라우팅: 신뢰도 측정/검증 → `argus/reliability` 또는 `agent-ab-test`

5. "이 에이전트를 완전 자율(Level 5)로 만들고 싶어요."
   - 올바른 라우팅: HITL 설계 필요 (Level 5는 극히 제한적) → 2축 매트릭스로 정당성 검증 필수

## 2) Edge Cases

### 경계 사례 (5)

1. **모든 작업이 Level 5(완전 자동)로 설계된 경우**
   - 입력: "우리 에이전트는 모든 작업을 자동으로 처리하니까 HITL이 필요 없어요."
   - 예상 행동: Step 2 매트릭스 재검토 → 대부분 작업이 높은 오류 영향도를 가졌을 가능성 → Level 2~3 강권
   - 근거: Level 5는 오류 영향도가 극히 낮은 경우만 정당화 (내부 로그 기록 등)

2. **Approval Gate를 설치했는데 승인자가 매번 자동 승인만 하는 경우**
   - 입력: "승인 게이트를 만들었는데 담당자가 매번 승인만 해요. 의미가 있을까요?"
   - 예상 행동: Approval Gate 재설계 → 임계값 기반 필터링 추가(신뢰도 80% 이상만 전송) 또는 Periodic Audit으로 전환
   - 근거: 자동 승인 = HITL 부재 → 필터링으로 실질적 개입 필요

3. **Escalation Chain이 너무 깊어서 응답 지연(SLA 미달)**
   - 입력: "팀원 → 매니저 → 임원까지 에스컬레이션하면 3일 걸려요."
   - 예상 행동: 체인 단순화 (2단계 이하) 또는 신뢰도 임계값으로 에스컬레이션 빈도 감소
   - 근거: SLA vs HITL 깊이 트레이드오프 → 응답 시간 제약에 맞춰 설계

4. **Shadow Mode 기간을 정하기 어려울 때 (언제까지?)**
   - 입력: "Shadow Mode로 몇 주 동안 감시해야 해요? 기준이 뭐죠?"
   - 예상 행동: 명확한 전환 기준 정의 → "2주 또는 1000개 이메일 처리 후, 90% 이상 일치도 달성 시"
   - 근거: Shadow Mode 종료 기준 없음 = 배포 지연 → 정량적 기준 필수

5. **가역성/오류 영향도 판단이 모호할 때**
   - 입력: "파일 수정이 되돌릴 수 있는가? 영향도가 높은가? 기준이 뭐죠?"
   - 예상 행동: 매트릭스의 구체적 정의 제시 → 가역성("git으로 버전 복구 가능" = 높음), 영향도("담당자 1명만 영향" = 낮음)
   - 근거: 주관적 판단 지양 → 정의된 기준으로 객관화
