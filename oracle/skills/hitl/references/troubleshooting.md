# Troubleshooting — hitl 스킬

## 1) HITL 설계 후에도 예상치 못한 오류 발생

**증상:**
- "HITL을 설계했는데 예상 못한 에러가 배포 후 나타났어요."
- "우리가 설정한 트리거로 감지 못 한 케이스가 있었어요."
- "어떻게 해야 다음에 놓치지 않을까요?"

**확인:**
- 발생한 오류의 특성: 예상했던 오류 타입인가? 새로운 시나리오인가?
- 기존 HITL 설계의 트리거 조건: 이 오류를 감지했을까?
- Shadow Mode 기간: 충분히 길었는가? (2주 vs 1주)

**조치:**
1. 즉시: Shadow Mode 기간 연장 (배포 일시 중지)
2. 새로운 오류 시나리오를 HITL 매트릭스에 추가:
   ```
   기존: 신뢰도 < 80% → 인간 개입
   신규: [새로운 오류 유형] 감지 → Escalation Chain 추가
   ```
3. 자동화 레벨 하향 검토: Level 3 → Level 2로 재설정
4. 다음 배포: 확장된 HITL 매트릭스로 재검증

## 2) Approval Gate를 설치했는데 승인자가 형식적으로만 승인

**증상:**
- "매번 승인 버튼만 누르네요. 실제로 검토 안 하는 것 같아요."
- "Approval Gate가 의미 있는 걸림돌이 아닙니다."
- "실질적인 인간 개입을 어떻게 하죠?"

**확인:**
- 승인자의 검토 시간: 평균 몇 초? (1초 = 형식적)
- 거부율: 전체 승인 요청 중 몇 %를 거부했는가?
- 승인 게이트의 내용: 승인자가 실제로 검토할 만한 정보가 있는가?

**조치:**
1. Approval Gate 재설계: 신뢰도 필터링 추가
   ```
   // Before: 모든 초안 → 승인 대기
   all_drafts → approval_gate

   // After: 신뢰도 기반 필터링 → 일부만 승인 필수
   high_confidence (>80%) → auto_approve
   low_confidence (<60%) → approval_gate → manual_review
   ```
2. 또는 Periodic Audit으로 전환:
   - 매일 100개 초안 생성 → 주 1회 50개 샘플 검토
   - 보다 효율적인 검증 방식
3. 승인자에게 기준 명확화: "어떤 경우 거부할지" 정의서 공유

## 3) Escalation Chain이 너무 깊어서 응답 지연

**증상:**
- "에스컬레이션이 팀원 → 매니저 → 임원까지 가면 3일이 걸려요."
- "이렇게 느리면 비즈니스에 지장이 있어요."
- "어떻게 단축할까요?"

**확인:**
- 현재 체인: 몇 단계? 각 단계의 응답 시간은?
- SLA: 몇 시간 이내 응답해야 하는가?
- 임원까지 가는 경우: 얼마나 자주 발생하는가?

**조치:**
1. 체인 단순화: 2단계 이하로 제한
   ```
   Before: 팀원(2h) → 매니저(4h) → 임원(1d) = 1.25일
   After: 팀원(2h) → 매니저(2h) = 4시간
   ```
2. 신뢰도 임계값으로 에스컬레이션 빈도 감소:
   - 신뢰도 > 70% → 자동 처리 (에스컬레이션 0)
   - 신뢰도 40~70% → 매니저만 (1단계)
   - 신뢰도 < 40% → 매니저 + 임원 (2단계, 드물게)
3. 타임아웃 규칙 추가:
   ```
   if response_time > 2_hours:
       escalate_to_next_level()
   ```

## 4) Shadow Mode 기간이 너무 길어서 배포 미루어짐

**증상:**
- "Shadow Mode를 2주 했는데 아직도 충분히 확실하지 않아요."
- "언제쯤 프로덕션으로 전환할지 알 수 없어요."
- "3개월이 지났는데도 Shadow Mode에 있습니다."

**확인:**
- 현재 일치도: 몇 %인가? (90% 달성했는가?)
- 정의된 전환 기준: 있는가?
- 팀의 불안감: 어디서 비롯되었는가?

**조치:**
1. 명확한 전환 기준 정의 (지금 당장):
   ```
   Green Light (배포 진행):
   - 일치도 90% 이상
   - 고객 재문의율 < 5%
   - 에러 2건 이상 시에만 재시작 (아니면 배포)

   또는
   - 2주 또는 1000개 처리 후 자동 전환 (리스크 재평가 필수)
   ```
2. 리스크 재평가:
   - Shadow Mode의 목적: 완벽함? (불가능) vs. 충분히 안전함? (달성 가능)
   - 충분히 안전한 기준 정의 후 배포
3. 팀과 커뮤니케이션: 기준 공유 + 배포 일정 확정

## 5) 모든 작업이 자동화 상태로 설정되어 있을 때

**증상:**
- "우리 에이전트는 모든 작업을 자동으로 처리하니까 HITL이 필요 없어요."
- "Level 5로 설정했는데 문제가 있을까요?"
- "완전 자동이 가장 효율적이지 않을까요?"

**확인:**
- 각 작업의 가역성과 오류 영향도를 실제로 평가했는가?
- 2축 매트릭스를 적용했을 때 Level 5로 정당화되는 작업이 몇 개인가?
- 실제 배포 후 오류 발생률은?

**조치:**
1. 2축 매트릭스 재검토:
   ```
   작업 1: 파일 수정 (가역성: 높음, 영향도: 낮음) → Level 4 (자동+감지)
   작업 2: 이메일 발송 (가역성: 낮음, 영향도: 높음) → Level 2 (승인)
   작업 3: 고객 대면 응답 (가역성: 낮음, 영향도: 높음) → Level 2 (승인)
   ```
2. Level 5는 극히 제한적:
   - 내부 로그 기록만 (고객 영향 없음)
   - 테스트/개발 환경 (프로덕션 아님)
   - 이외 거의 불가능
3. 대부분 Level 2~4가 적절 → HITL 재설계 권장
