# Domain Context — build-or-buy

## 1) Domain Scope

Agent Build/Buy Decision Framework — 개별 에이전트에 대해 자체 구축 vs 외부 구매 vs 노코드 선택지를 평가하는 도메인

**적용 범위:**
- 특정 에이전트 또는 에이전트 모듈에 대한 단계별 의사결정
- 구현 리소스 제약이 있는 상황 (시간, 예산, 엔지니어)
- 외부 솔루션이 존재하는 경우 vs 없는 경우 모두 포함
- 하이브리드 전략 평가 (일부는 Build, 일부는 Buy)

**제외 범위:**
- 기회 발굴 단계 (opp-tree 스킬)
- 구현 후 기술 선택 (architecture 스킬)
- 가격/GTM 모델 결정만 필요한 경우 (agent-gtm 스킬)

---

## 2) Primary Users

- **PM / Product Manager**: 개발 리소스 배분 결정 (여러 에이전트 간 우선순위)
- **Tech Lead / CTO**: 기술 스택 선택의 비용-편익 분석
- **Finance / Budget Owner**: 개발 비용 vs 구독료 비교
- **Startup Founder**: 리소스 부족 상황에서 빠른 시장 진입 필요

---

## 3) Required Inputs

- **에이전트 아이디어**: "고객 문의 자동 분류", "계약서 검토" 등
- **핵심 기능 3가지**: 에이전트가 꼭 해야 할 일
- **사용자/빈도**: 일간/월간 사용자 수 또는 호출 건 수
- **개발 리소스**: 현재 팀 규모, 엔지니어 수, 가용 시간
- **시간 제약**: 런칭 데드라인 (있으면)
- **예산**: 구독료 또는 개발 비용 상한 (있으면)

---

## 4) Output Contract

**Step-by-Step Deliverables:**

1. **요구사항 명확화**
   - 에이전트 목적 (1문장)
   - 핵심 기능 3가지
   - 타겟 사용자 + 사용 빈도

2. **차별화 판단**
   - 이 에이전트가 제품의 핵심 차별화인가? (Yes/No)
   - 도메인 특화 데이터/로직 필수인가? (Yes/No)

3. **외부 솔루션 조사**
   - 2~3개 기존 솔루션 비교표
   - 각 솔루션의 비용, 제약, 커스터마이징 한계

4. **평가 매트릭스**
   - 6개 항목 (차별화, 속도, 비용, 커스터마이징, 유지보수, 도메인)
   - Build/Buy/No-Code 각각 점수 (1~5)
   - 프로젝트 맥락별 가중치 적용
   - 가중 합계 비교

5. **하이브리드 전략 검토**
   - 레이어별 다른 선택지 조합 가능한가?
   - (예: 오케스트레이션은 Build, 이메일은 Buy)

6. **최종 권장 + 근거**
   - 선택지 1개 (또는 하이브리드)
   - 근거 3가지 이상
   - 재검토 트리거 명시

7. **다음 단계**
   - Build → instruction-design 스킬로 이동
   - Buy → 통합 방법 가이드 제공
   - No-Code → 플랫폼 선택 기준

---

## 5) Guardrails

**필수 체크사항:**

- ✓ 평가 항목 6가지 모두 Build/Buy/No-Code 점수가 있는가?
  - 하나라도 빠지면 평가 불완전

- ✓ 가중치가 프로젝트 맥락을 반영하는가?
  - "모든 항목 동일 가중치" (X)
  - "속도가 최우선이므로 2배", "차별화는 3배" (O)

- ✓ 외부 솔루션이 최소 2~3개 조사되었는가?
  - "우리가 만드는 게 최고다" (주관적) → 거절
  - "Intercom, Zendesk, Freshdesk 3개 비교" (객관적) → 승인

- ✓ 재검토 트리거가 명시되었는가?
  - "6개월 후", "유지보수 > 월 20시간", "비용 > 월 $500" 등

## 6) Working Facts

**Build-or-Buy 의사결정의 일반적 특성:**

| 항목 | Build 우호 | Buy 우호 | No-Code 우호 |
|------|----------|---------|-------------|
| **차별화** | 높음 | 낮음 | 낮음 |
| **구현 속도** | 느림 | 빠름 | 매우 빠름 |
| **장기 비용** | 낮음 (투자 후) | 높음 | 중간 |
| **커스터마이징** | 완전 자유 | 제한적 | 약간 자유 |
| **유지보수** | 팀이 담당 | 벤더 담당 | 플랫폼 담당 |
| **도메인 특화** | 매우 가능 | 제한적 | 불가능 |

**일반적 선택:**
- **Pain Intensity 높음 + 차별화 높음** → Build
- **Pain Intensity 높음 + 차별화 낮음** → Buy (빠르게)
- **시간 제약 + 리소스 부족** → No-Code 또는 Buy (예시값, 조직별 구독료 조정 가능)
- **도메인 특화 필수** → Build (또는 하이브리드)

**TO BE UPDATED by reviewer:**
- 우리 조직의 평균 개발 속도 (1주일에 몇 % 완성?)
- 외부 솔루션 구독료 평균 (도메인별)
- 하이브리드 전략의 통합 복잡도 (우리 조직 경험상)

## 7) Fill-in Checklist

구현 팀이 build-or-buy 스킬 실행 후 다음을 확인:

- [ ] 요구사항이 명확하게 (1문장 목적 + 핵심 기능 3개) 정의되었는가?
- [ ] 차별화 판단(핵심 기능인가? 도메인 특화인가?)이 완료되었는가?
- [ ] 외부 솔루션 2~3개가 조사되고 비교표로 정리되었는가?
- [ ] 6개 항목 점수와 가중치가 명시되었는가?
- [ ] 최종 권장 선택지(Build/Buy/No-Code 또는 하이브리드)와 근거 3가지가 있는가?
- [ ] 재검토 트리거(언제, 어떤 조건에서 다시 평가할 것인가?)가 정의되었는가?
- [ ] 다음 단계 연결이 명확한가?

## 8) 참고 사례: Agent 프레임워크 선택

> 아래는 특정 프로덕션 환경에서의 사례입니다. 조직과 도메인에 따라 다르게 설계할 수 있습니다.

### Build 의사결정 후 참고할 수 있는 도구 선택

Build 결정 후 구현 시 프레임워크를 선택해야 합니다. 이 섹션은 구현 단계에서만 참고하는 수준이며, **정확한 기술 선택은 별도 architecture 스킬을 참조하세요.** 이 도메인은 Build vs Buy 의사결정 자체에 집중합니다.

#### 프레임워크 선택의 일반적 트레이드오프

| 특성 | LangGraph | CrewAI | MCP |
|------|----------|--------|-----|
| **복잡도 제어** | 높음 | 중간 | 중간 |
| **구현 속도** | 중간 | 빠름 | 빠름 |
| **도구 통합** | 수동 | 자동화 | 표준화 |

**세부 기술 선택(아키텍처, 성능 최적화)은 별도 architecture 스킬을 참조하세요.**

---

### 프레임워크 선택 후 다음 단계

Build 선택 후 프레임워크 결정이 필요하면, 다음 스킬들을 활용하세요:
- **instruction-design 스킬**: 에이전트 구조 및 프롬프트 설계
- **architecture 스킬**: 기술 스택, 성능 최적화, 도구 통합 세부사항
