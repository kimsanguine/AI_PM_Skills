# Troubleshooting — build-or-buy

## 1) Build 선택 후 유지보수 비용이 예상 대비 3배 이상 높음

**증상:**
- 월 유지보수 시간이 예상 10시간 → 실제 30시간+
- 예: "매주 디버깅에 16시간씩 쓴다"
- 팀이 Build 선택을 후회하는 상태

**확인:**
- 실제 유지보수 시간이 월 20시간 초과하는지 확인
- 외부 솔루션 비용(Buy)이 월 $500 이하인지 재확인

**조치:**
- Build → Buy 재검토 강제
- "유지보수 비용이 월 20시간(= 약 $3,000)을 넘으면, Buy가 더 저렴할 가능성 높음"
- 외부 솔루션 재평가 후 즉시 전환 권장
- 전환 비용(마이그레이션)도 고려

---

## 2) Buy 선택 후 솔루션의 커스터마이징 한계에 부딪힘

**증상:**
- 구현 후 "우리 요구사항의 30%만 지원한다" 판명
- 원래 평가에서는 "70% 충족"이라고 했는데...

**확인:**
- 원래 요구사항 분석이 정확했는지 재검토
- 솔루션의 실제 커스터마이징 범위 재평가

**조치:**
- No-Code 통합 레이어로 임시 해결 (예: Zapier로 부족 기능 보충)
- 또는 Build 전환 검토
- 또는 다른 솔루션 탐색
- 다음 평가에서는 "요구사항 충족도 70%"를 어떻게 측정했는지 명시

---

## 3) No-Code 선택 후 확장이 필요해졌는데 플랫폼 한계 직면

**증상:**
- Zapier/Make로 초기 검증 완료
- 사용량 증가 후 "Rate Limit 또는 복잡도 한계" 발생
- "이제 더 이상 No-Code로는 불가능" 상황

**확인:**
- 현재 사용량이 플랫폼 한계를 실제로 초과했는지 확인
- Build로 전환할 필요성 재평가

**조치:**
- No-Code → Build 전환 진행
- "검증 단계는 완료했으므로, 이제 확장 가능한 Build로 전환"
- 기존 검증된 프롬프트/워크플로우는 그대로 이관 가능
- Build 비용은 초기 4주 개발 + 월 유지보수로 계산

---

## 4) 하이브리드 전략이 너무 복잡해짐

**증상:**
- 원래 계획: 오케스트레이션(Build) + 이메일(Buy) = 2개 레이어
- 실제: 오케스트레이션(Build) + 도메인 로직(Build) + 이메일(Buy) + 캘린더(No-Code) + 모니터링(Buy) = 5개 통합 지점
- "관리가 너무 어렵다" 팀 호소

**확인:**
- 통합 지점이 실제로 3개 이상인지 확인
- 각 통합 지점의 복잡도 평가

**조치:**
- 단순화: 핵심 2개 레이어만 남기고 나머지는 단일 솔루션으로 통합
  - 예: "오케스트레이션(Build) / 나머지 모두(Freshdesk Buy)"
  - 통합 지점: 1개 (webhook)
- 또는 전체를 1개 플랫폼(Buy)에 통합
- 다음 평가에서는 "하이브리드의 통합 복잡도 평가" 항목 추가
