# Troubleshooting — agent-gtm 스킬

## 1) Beachhead 선정 후 고객이 "우리 요구사항과 안 맞는다"고 판명

**증상:**
- Beachhead 점수: 23/25 (적합)
- Lighthouse 고객 오보딩 1주 후: "실제로는 우리와 안 맞다"는 피드백
- 또는 고객이 명시적으로 거절

**확인:**
- Beachhead 선정 기준이 실제와 다른가?
  - Pain Intensity: 실제로 5/5 수준의 절박한가?
  - Willingness to Trust: 실제로 AI 사용에 동의할 준비가 되었나?
  - Data Availability: 실제 데이터가 충분한가?
- 고객의 실제 니즈 vs 우리 가정: 간극이 있나?

**조치:**
1. **즉시 다음 비치헤드로 전환**: Beachhead 우선순위 2번(또는 점수 2~3번)으로 이동
2. **탈락 사유 기록**: "왜 적합하지 않았나?" 문서화 → 향후 Beachhead 선정 개선
3. **빠른 학습 적용**: 1주차 경험을 바탕으로 다음 Lighthouse 고객과의 대화 개선
4. **타자료 수집**: Lighthouse 고객과 대화 → "실제 Beachhead 특징 재정의"
5. **손실 최소화**: 이 고객과의 관계는 유지하되, "나중에 확장할 수 있다"는 메시지

## 2) Shadow Mode 기간이 너무 길어짐: 고객 참여도 감소

**증상:**
- Shadow Mode 설정: "2주"
- 실제 운영: "6주 초과"
- 고객 피드백: "일단 자동화해 줄래?" (지쳐함) / 참여도 급락

**확인:**
- 지연 원인:
  - 에이전트 정확도가 80% 미만인가? (KPI 미달)
  - Shadow Mode 완료 기준이 명확하지 않았나?
  - 외부 요인 (휴가, 기타 프로젝트)?
- 실제 정확도: 목표(80%)에 도달했는가?

**조치:**
1. **Stage 1 기간 단축**: Shadow Mode 2주 → 1주로 조정
   - 고객이 기다리는 심리 < 부분적 자동화 체험 가치
2. **신뢰도 임계값 재조정**: 정확도 요구사항을 80% → 70% 낮추기
   - "완벽할 때까지" 대기 → "충분히 좋을 때" 진입
3. **Co-pilot Mode로 진전**: Shadow가 길어지면 Co-pilot(사용자 승인)으로 빨리 전환
   - 사용자가 "내가 확인하면 안전하구나" 신뢰 축적
4. **투명성**: "지난주 정확도 75% → 이번주 78%"처럼 매주 진전 공유
5. **사용자 선택권**: "계속 Shadow로 할까, Co-pilot으로 넘어갈까?" 물어보기

## 3) Trust Sequence를 거쳤는데도 Wedge Phase 구매 의향 낮음

**증상:**
- Lighthouse: NPS 65, 만족도 높음 (성공!)
- Wedge Phase 진입 후: NPS < 30, 구매 의향 없음
- "Lighthouse가 특수한 경우였나?"

**확인:**
- Lighthouse vs Wedge 고객 차이:
  - 회사 규모? 예산? 기술 역량?
- GTM 메시지: Lighthouse와 다른가?
- Wedge 고객의 Pain Intensity: 실제로 높은가?
- Wedge 온보딩: Lighthouse와 동일한가?

**조치:**
1. **Lighthouse 고객 피드백 재수집**: "뭐가 가장 마음에 들었어?" "다른 팀에 추천할까?"
2. **GTM 메시지 재평가**: Lighthouse 고객 사례를 기반으로 메시지 재작성
   - 기술 스펙 중심 → 비즈니스 가치 중심으로 변경
3. **가격 재검토**: Wedge 고객이 구매하지 않는 이유 = 가격?
   - 가격 모델 변경 또는 할인 검토
4. **Beachhead 재평가**: "실제 Beachhead가 우리가 선택한 것과 다른가?"
   - Lighthouse 고객과 같은 특징(규모, 산업, 기술 성숙도)의 고객부터 시작
5. **온보딩 차이점 분석**: Lighthouse 성공 프로세스를 Wedge에 정확히 재현

## 4) Lighthouse에서 성공, Wedge 고객은 구축 난이도 높음

**증상:**
- Lighthouse: 3주 온보딩, "쉽다", 즉시 value 체감
- Wedge 고객: 6주 요청, "복잡하다", 내부 저항
- "왜 이렇게 다르지?"

**확인:**
- Lighthouse 환경의 특수성:
  - 작은 팀? 단순한 데이터 구조? 기술 관리자가 있었나?
- Wedge 고객의 특징:
  - 더 큰 조직? 복잡한 legacy 시스템? 기술 지원 부족?
- 온보딩 가이드: Lighthouse 특수 사항을 일반화하지 않았나?

**조치:**
1. **Lighthouse 환경 분석**: "이 고객이 왜 쉬웠을까?" 체크리스트 작성
2. **온보딩 자동화 강화**: Lighthouse 성공의 "쉬운 부분" 자동화
   - 예: 데이터 import template, 샘플 workflow 제공 등
3. **Wedge 고객 유형별 가이드**: "대규모 조직은 이렇게", "Legacy 시스템은 이렇게"
4. **온보딩 기간 연장 수용**: Wedge 고객은 실제로 6주가 필요할 수 있음
   - KPI 재조정: "3주" → "첫 달은 셋업, 둘째 달은 운영"
5. **고객 지원 강화**: 전담 CSM 또는 구현 팀 할당

## 5) Lighthouse 정확도는 높은데 NPS가 낮음

**증상:**
- KPI: 정확도 > 80% (달성: 85%)
- 그런데 NPS: 35 (목표: 60)
- "정확도가 좋은데 왜 고객 만족도는 낮지?"

**확인:**
- 정확도 vs 신뢰: "정확하다"와 "신뢰한다"는 다른 개념
- 사용자 경험:
  - 에이전트 응답 속도는?
  - UI/UX는 직관적인가?
  - 고객이 에이전트를 "실제로" 믿고 사용하는가?
- Trust Sequence: Shadow/Co-pilot/Auto 진행이 자연스럽지 않았나?

**조치:**
1. **NPS 점수의 의미 파악**: "왜 추천하고 싶지 않아?" 직접 물어보기
2. **정확도 외 지표 추가**: 응답 속도, 사용자 만족도(별점), 재사용율
3. **Trust Sequence 재검토**: 정확도 높아도 "신뢰"가 형성되지 않으면 NPS 낮음
   - Shadow Mode: "가장 정확한 경우들만" 보여주기
   - Co-pilot Mode: "사용자가 승인했으니 안전하다"는 심리 강화
4. **고객 피드백 적용**: "고객이 원하는 개선사항 1순위"를 신속하게 반영 → NPS 회복
5. **성공 시나리오 재정의**: 정확도 뿐 아니라 "신뢰도" 지표 추가 (예: "사용자 승인율 > 90%")
