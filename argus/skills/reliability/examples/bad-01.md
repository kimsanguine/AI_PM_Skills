# Bad Example — Reliability

## 사용자 요청

"에이전트가 간헐적으로 이상한 응답을 내고 있어. 뭔가 잘못된 것 같은데 뭐가 문제인지 모르겠어. 어떻게 해야 할까?"

## 거절 이유

- "신뢰성"이 정의되지 않음 (Success Rate가 몇 %인가?)
- 기준선 데이터 없음 (현재 상태를 정량화할 수 없음)
- 실패 분류 없음 (어떤 카테고리의 에러인가?)
- Safeguard 설계 불가 (대응 방법이 불명확)
- 목표 수준 미설정 (뭐가 "좋은" 신뢰성인가?)
- 개선 로드맵 부재 (언제, 어떤 순서로 개선할 것인가?)

## 올바른 라우팅

**argus/skills/reliability** (기준선 측정 → 실패 분류 → Safeguard 설계 필요)

또는 구체적 장애 상황이라면:
- **argus/skills/incident** (실제 발생한 장애의 원인 분석과 즉시 대응)

## 수정 방향

"지난 30일 운영 데이터를 기반으로 현재 신뢰성(Success Rate, Error Rate)을 측정해줄 수 있어? 그리고 실패들을 분류해서 각 유형별로 safeguard를 설계해주면 좋겠어. 목표는 95% 이상 성공률이야."
