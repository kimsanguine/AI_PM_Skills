# Troubleshooting — agent-ab-test 스킬

## 1) 표본 크기 계산 오류

**증상:**
- "1주일 데이터로 테스트했는데 유의성이 없다고 한다"
- p-value가 지속적으로 0.05 이상 (통계적 유의 아님)
- "샘플이 부족하다"는 메시지 반복

**확인:**
- MDE(Minimum Detectable Effect) 계산 여부 확인
- 필요 표본 수 계산 완료? (α=0.05, β=0.20 기준)
- 현재 수집된 표본 수가 필요 수보다 작은가?

**조치:**
1. 기존 테스트 결과 분석: baseline 메트릭 값 확인
2. MDE 재설정 (보수적 목표로 조정, 예: 5% → 3%)
3. 필요 표본 수 재계산 (N = 2σ²(Z_α + Z_β)² / Δ²)
4. 데이터 수집 기간 연장 (일일 트래픽 기준으로 필요 일수 계산)
5. 재분석 스케줄 설정 (예: 14일 후 재분석)

## 2) 테스트 오염(시간대 편향)

**증상:**
- A와 B 사이의 성능 차이가 명확하지 않음
- 같은 B 변형을 다른 시간대에 실행하면 결과가 크게 다름
- "혹시 요일이나 시간대가 영향을 주는 걸 아닌가?"

**확인:**
- A/B 동시 실행(50/50 스플릿) 여부 확인
- 순차 실행(A 먼저, 후 B) 진행되지 않았나?
- 입력 데이터 분포가 기간별로 변했는가? (주중/주말, 시간대별)

**조치:**
1. 테스트 재설계: A/B 병렬 실행 구현
2. 트래픽 스플리팅: 50/50 또는 canary (90/10)로 설정
3. 기간 동안 입력 데이터 분포 검증 (데이터 드리프트 체크)
4. 필요시 시간대 편향 보정 (통계 모델 적용) 또는 테스트 재실행

## 3) 비결정론 효과로 인한 높은 변동성

**증상:**
- 같은 입력에 5회 반복 실행 결과가 큰 폭으로 변동
- 표준편차가 평균의 30% 이상
- "결과가 일관되지 않아서 신뢰하기 어렵다"

**확인:**
- 모델의 temperature 설정값 확인 (높으면 변동성 증가)
- 반복 실행 횟수 (5회 충분한가?)
- 프롬프트에서 결정론적 구조(예: step-by-step instructions) 포함 여부

**조치:**
1. 모델 파라미터 조정: temperature 낮추기 (예: 0.7 → 0.5)
2. 테스트 기간 연장: 변동성이 감소할 때까지 더 많은 데이터 수집
3. 프롬프트 강화: 더 구체적 지시사항으로 출력 안정성 향상
4. 대안: 변동성이 감소하지 않으면, 다른 모델 검토 또는 메트릭 재정의 (평균 대신 중앙값 사용)

## 4) Guardrail 메트릭 악화

**증상:**
- B 변형이 Primary 메트릭은 개선했으나 Guardrail(예: Hallucination Rate) 악화
- "정확도는 높아졌는데 환각이 늘었다"
- 즉시 테스트 중단 필요한 상황

**확인:**
- Guardrail 임계값 설정 여부 (예: Hallucination < 5%)
- 현재 B의 Guardrail 값이 임계값 초과했는가?
- Primary와 Guardrail 간 인과관계 분석 (우연인가, 구조적 문제인가?)

**조치:**
1. 즉시 대응: B 변형 중단, 이전 버전으로 복구
2. 원인 분석:
   - "왜 정확도는 높아졌는데 환각이 늘었나?" (프롬프트 변경의 부작용?)
   - "Guardrail 악화가 구조적인가, 일시적인가?"
3. 재설계: B'(B의 개선된 버전) 설계, Guardrail 보호 장치 추가
4. 재테스트: B' vs A 테스트 재실행 (더 긴 기간, 더 큰 표본)

## 5) 결과 해석 불일치 (트레이드오프)

**증상:**
- Primary 메트릭 개선, Secondary 메트릭 악화 (상충)
- 팀 내 의견 불일치: "B를 쓸까, 말까?"
- "이게 진짜 개선인지 확실하지 않다"

**확인:**
- Primary vs Secondary 메트릭의 가중치 명확한가?
- 개선/악화 폭이 실무 의미가 있는가? (통계 유의 vs 실무 유의)
- 신뢰 구간(Confidence Interval) 겹치는가?

**조치:**
1. 메트릭 가중치 재정의:
   - "우리가 정말 우선순위를 두는 건 Primary인가?"
   - 필요시 Primary/Secondary 순서 재조정
2. 신뢰 구간 확인: 95% CI가 겹치면 차이 없을 가능성 높음
3. 의사결정 프레임 선택:
   - **Ship B**: Primary 우선, Secondary 악화는 용인 가능 범위
   - **Keep A**: Secondary 악화가 중대 (Guardrail 수준)
   - **Iterate**: B의 방향은 맞지만 Secondary 문제 해결 필요
   - **Investigate**: 추가 분석 필요, 더 많은 데이터 수집
4. 문서화: 의사결정 근거와 트레이드오프 명시

## 6) 테스트 재실행 비용 관리

**증상:**
- "테스트를 반복하다 보니 비용이 예상보다 크다"
- 실험 예산 초과 위험
- "몇 번 더 테스트할 여유가 없다"

**확인:**
- 테스트당 예상 비용 계산 (표본 수 × execution당 비용)
- 재테스트 필요한 경우의 수 확인
- 예산 남은 액 계산

**조치:**
1. 재테스트 우선순위 재평가:
   - "정말 이 문제를 해결해야 하나?"
   - "더 효율적인 테스트 설계는 없나?" (Bayesian 테스트 등)
2. 표본 크기 최적화: 더 작은 MDE로 조정 (시간 대신 파워 조정)
3. 예산 재협상: 3개월 테스트 예산 중 이번 달 배분 재조정
4. 대안 검토: A/B 테스트 대신 코호트 분석이나 정성 평가로 의사결정
