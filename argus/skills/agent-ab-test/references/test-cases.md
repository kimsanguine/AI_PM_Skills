# Test Cases — agent-ab-test 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "프롬프트 버전 A와 B의 성능 차이를 통계적으로 비교하고 싶은데 샘플 크기는 얼마나 필요할까?"
   - 이유: A/B 테스트 설계 및 샘플 크기 계산은 핵심 트리거 시나리오

2. "Haiku 모델로 다운그레이드했을 때 정확도 영향을 측정해야 한다"
   - 이유: 모델 변경 효과 검증 → 명확한 agent-ab-test 사용

3. "새로운 TK 세트를 추가한 후 에이전트가 개선되었는지 검증하려면?"
   - 이유: TK 주입 효과 측정은 A/B 테스트 트리거 조건

4. "지난주 프롬프트 업데이트로 응답 시간이 2배 늘었는데 정말 의미 있는 차이인가?"
   - 이유: 성능 변화의 통계적 유의성 검증 필요

5. "기존 단일 에이전트를 멀티 에이전트 아키텍처로 변경하면 성능이 나아질까?"
   - 이유: 아키텍처 변경 영향 정량화

### Should NOT Trigger (5)

1. "북스타 지표가 이전 달과 비교해서 5% 떨어졌다. 분석해 줄 수 있나?"
   - 올바른 라우팅: `north-star` (지표 추세 분석)

2. "우리 에이전트가 이번 달 비용이 2배가 되었어. 왜 그럴까?"
   - 올바른 라우팅: `burn-rate` (토큰 비용 추적 및 최적화)

3. "v1.0과 v1.1 에이전트의 사용자 재방문율이 차이나는지 알고 싶다"
   - 올바른 라우팅: `cohort` (버전별 장기 성능 추적)

4. "어제 정확도가 급락했는데 뭔가 문제가 있나?"
   - 올바른 라우팅: `incident` (장애 대응)

5. "매주 정확도, 응답시간, 비용을 모니터링하려면 어떻게 설정할까?"
   - 올바른 라우팅: `kpi` (KPI 정의 및 대시보드)

## 2) Edge Cases

### 경계 사례 (5)

1. **표본 크기 미달**
   - 입력: "100건 데이터로 A/B 테스트를 했는데 p-value가 0.12가 나왔다"
   - 예상 행동: MDE 계산 후 필요 표본 수 재산정, 추가 데이터 수집 권고
   - 근거: 표본이 부족하면 통계적 유의성 검증 불가 → 재테스트 필수

2. **비결정론 효과 과도**
   - 입력: "같은 입력을 5회 반복했는데 성공률이 100%, 80%, 75%, 85%, 90%로 들쑥날쑥이다"
   - 예상 행동: 표준편차 > 평균의 30% 감지, 더 긴 테스트 기간 또는 모델 온도 조정 권고
   - 근거: LLM 비결정론이 과도하면 노이즈가 신호를 압도

3. **순차 실행 편향**
   - 입력: "A를 월요일~수요일, B를 목요일~금요일에 테스트했다"
   - 예상 행동: 시간대 편향 가능성 지적, 주중/주말 차이 보정 또는 테스트 재설계 권고
   - 근거: 시간대별 입력 분포 변화는 통제 불가능한 변수 → 동시 실행 권고

4. **Guardrail 메트릭 악화**
   - 입력: "B가 정확도는 +3% 개선했는데 환각율이 2%에서 8%로 악화됐다"
   - 예상 행동: 즉시 테스트 중단, guardrail 메트릭 우선 → 트레이드오프 재검토
   - 근거: 안전 지표 악화는 Ship 결정을 무효화

5. **결과 해석 불일치**
   - 입력: "Primary 메트릭은 +5% 개선, Secondary는 -8% 악화, 뭐가 맞는 답?"
   - 예상 행동: 트레이드오프 분석 후 Ship/Iterate/Investigate 판단 재검토, Primary 가중치 명확화
   - 근거: 단일 지표 개선은 무의미, 실무 유의성(Practical Significance) 평가 필수
