# Troubleshooting — burn-rate 스킬

## 1) 비용 기준선 계산 오류

**증상:**
- "현재 비용이 정확히 얼마인지 모르겠다"
- 입출력 토큰 비율이 부정확하거나 모델 가격 정보가 오래됨
- 월간 비용 예측과 실제 청구액 큰 차이

**확인:**
- 입력/출력 토큰 평균 계산 (최소 100건 샘플 필요)
- 현재 사용 모델과 가격 정보 최신인가? (API 문서 확인)
- 일관된 기간(예: 지난 30일)으로 평균 계산했는가?

**조치:**
1. 비용 기준선 재계산:
   - 최근 30일 데이터 수집 (완전한 달 기준)
   - 입출력 토큰 분리 계산 (가격이 다름)
   - 실행 건수 × (평균 입력토큰 × 입력가격 + 평균 출력토큰 × 출력가격)
2. 가격 정보 최신 확인 (Anthropic API 문서)
3. 이상 거래 확인 (특이한 사용량 스파이크)

---

## 2) 비용 이상 폭증 감지

**증상:**
- 일일 비용이 예상의 2배 이상
- "어제는 정상인데 오늘 비용이 10배 뛰었다"
- 월말 누적 비용이 예산을 크게 초과할 예상

**확인:**
- 일일 실행 건수가 평상시와 다른가?
- 특정 기능이나 사용자 그룹에서 비용 급증한가?
- API 호출에 실패했거나 재시도 루프가 생겼나?
- 모델이 갑자기 더 큰 모델로 변경되지 않았나?

**조치:**
1. 즉시 조치:
   - API rate limit 설정 (1시간당 최대 호출 수 제한)
   - 의심 기능 즉시 비활성화 또는 캐시 활용
2. 원인 분석:
   - 로그 검토: 특정 유저/기능의 비용 급증?
   - 무한 루프 여부 확인 (재시도 로직)
   - 모델 라우팅 설정 확인
3. 비용 캡 설정: 다음 단계 실행 전 일일/주간 비용 한계선 설정
4. 영향 범위 산정: 폭증 기간 동안의 정확한 추가 비용 계산

## 3) 최적화 후 품질 저하

**증상:**
- "비용은 절감했는데 정확도가 내려갔다"
- 프롬프트 압축, 캐싱, 모델 다운그레이드 후 성능 악화
- 고객 피드백 부정적 증가

**확인:**
- 최적화 전후 Accuracy, Hallucination Rate 비교
- 품질 저하가 일시적인가, 지속적인가?
- Guardrail 메트릭(예: Accuracy > 90%) 위반했는가?

**조치:**
1. 최적화 즉시 롤백 (원래대로 복구)
2. 보수적 접근으로 재계획:
   - 프롬프트 압축: 30% → 10%로 축소
   - 모델 다운그레이드: Sonnet → Haiku 대신 Sonnet + 캐싱
   - 캐싱: 전체 → 자주 사용되는 부분만
3. A/B 테스트로 검증: 최적화 버전 vs 원본 테스트
4. 점진적 롤아웃: 100% 적용 전 10% 사용자부터 시작

## 4) 추적 데이터 누락

**증상:**
- "API 로그에서 일부 기간의 데이터가 없다"
- 추적 시스템이 다운되었거나 데이터 손실
- 월간 비용 계산에서 빈 구간 발생

**확인:**
- 로깅 시스템 상태 확인 (정상 작동 여부)
- 누락 기간 확인 (예: 3월 5~7일 데이터 없음)
- 그 기간 사용량 추정 가능한가? (유사 기간 참고)

**조치:**
1. 누락 기간 비용 추정:
   - 누락 전후 유사 기간 평균 비용 계산
   - 보수적 추정(높게) 적용
2. 향후 로깅 강화:
   - 듀얼 로깅 시스템 구축 (주 로거 + 백업)
   - 로그 검증 자동화 (매일 데이터 완성도 체크)
3. 예산 재조정: 추정 비용을 월간 예산에 반영

## 5) 모델 가격 정책 변경

**증상:**
- API 제공자 가격 인상 공지
- "가격이 올랐으니 월간 비용이 X% 증가할 것 같다"
- 기존 계획된 예산이 맞지 않음

**확인:**
- 현재 사용 모델들의 가격 변화율 정확히 확인
- 현재 월간 사용 토큰 규모 파악
- 영향받을 예상 비용 계산 (기존 사용량 × 새 가격)

**조치:**
1. 영향도 계산:
   - 월간 토큰 사용량 × 가격 인상률 = 추가 비용
   - 연간 영향도도 계산
2. 대응 전략 검토:
   - **수용**: 예산 증액 요청
   - **회피**: 모델 라우팅 (다른 제공자 또는 더 저가 모델)
   - **개선**: 캐싱, 배치 처리 등으로 토큰 사용 줄이기
3. 예산 재책정: 새 가격 반영하여 월간/분기 예산 업데이트
4. 대체 제공자 검토: 필요시 여러 모델/제공자 비교

---

## 6) 예산 초과로 인한 재정 압박

**증상:**
- "월말에 보니 예산을 50% 초과했다"
- "이 추세면 분기 예산을 200% 초과할 것 같다"
- 재정팀의 비용 통제 요청

**확인:**
- 예산 초과 원인 분석:
  - 예측 오류(사용량 예상 오류)?
  - 트래픽 급증?
  - 새 기능 추가로 인한 비용 증가?
  - 모델 가격 인상?
- 초과된 월이 특이점인가, 새로운 추세인가?

**조치:**
1. 원인별 대응:
   - **예측 오류**: 다음 분기 예산 현실적으로 재설정
   - **트래픽 급증**: 최적화 또는 기능 제한 (rate limiting)
   - **새 기능**: ROI 평가, 필요시 기능 스코프 축소
   - **가격 인상**: 모델 라우팅 또는 다른 제공자 검토
2. 긴급 최적화 실행:
   - 가장 비용 효율 낮은 기능부터 최적화
   - 불필요한 API 호출 제거
3. 예산 재협상: 경영진과 현실적 예산선 재합의
4. 분기별 리뷰 회의: 월간 비용 추이 정기 리뷰
