# Test Cases — burn-rate 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "우리 에이전트의 월간 토큰 비용을 추적하고 최적화하려면?"
   - 이유: 비용 기준선 설정과 추적은 핵심 트리거

2. "Claude Sonnet에서 Haiku로 다운그레이드하면 비용이 얼마나 절감될까?"
   - 이유: 모델 선택의 경제학적 영향 계산

3. "프롬프트 압축과 캐싱으로 토큰 비용을 30% 줄일 수 있을까?"
   - 이유: 최적화 전략의 비용 효과 분석

4. "이번 달 API 비용이 $500에서 $1,200으로 폭등했어. 왜?"
   - 이유: 비용 이상 감지 및 원인 분석

5. "월간 예산을 정하려면 평균 비용 기준선부터 파악해야 한다"
   - 이유: 예산 책정을 위한 비용 가시화

### Should NOT Trigger (5)

1. "A/B 테스트에서 두 프롬프트 버전 중 어느 게 정확도가 더 높을까?"
   - 올바른 라우팅: `agent-ab-test` (성능 비교)

2. "지난 3개월간 정확도 추이를 보면 점점 떨어진다"
   - 올바른 라우팅: `cohort` (버전별 시간 경과에 따른 성능)

3. "KPI 대시보드에 Cost Per Execution 메트릭을 추가하고 싶다"
   - 올바른 라우팅: `kpi` (KPI 정의)

4. "어제 API 호출이 갑자기 10배 증가했는데 뭔가 잘못된 건가?"
   - 올바른 라우팅: `incident` (장애 대응)

5. "비용 효율을 North Star 지표로 포함해야 할까?"
   - 올바른 라우팅: `north-star` (핵심 지표 정의)

## 2) Edge Cases

### 경계 사례 (5)

1. **추적 데이터 누락**
   - 입력: "API 로그에 일부 호출이 기록되지 않았어. 3월 1~5일 데이터가 없다"
   - 예상 행동: 누락 기간 비용 추정치 계산, 향후 로깅 강화 권고
   - 근거: 부정확한 비용 추적은 예산 책정 오류로 이어짐

2. **최적화로 인한 품질 저하**
   - 입력: "프롬프트 압축으로 비용은 30% 줄었는데 정확도가 92%에서 85%로 떨어졌다"
   - 예상 행동: 최적화 롤백, 보수적 절감 전략으로 재계획, Guardrail 메트릭 확인
   - 근거: 비용 절감이 품질 저하로 이어지면 고객 만족도 및 장기 비용 손상

3. **모델 가격 정책 변경**
   - 입력: "Anthropic이 Claude 가격을 올렸다는 공지를 받았다"
   - 예상 행동: 영향도 계산 (월간 추가 비용), 모델 라우팅 검토, 예산 재책정
   - 근거: 외부 가격 변화는 비용 구조 전반을 재검토 필요

4. **예산 초과 누적**
   - 입력: "월말에 보니 월간 예산을 200% 초과했다"
   - 예상 행동: 원인 분석 (예측 오류? 트래픽 급증? 새 기능 추가?), 긴급 최적화, 다음 분기 예산 재협상
   - 근거: 예산 초과는 회사 재정 계획에 영향

5. **비용 최적화의 딜레마**
   - 입력: "배치 처리로 처리량은 10배 늘었는데 latency가 30분에서 2시간으로 늘었다"
   - 예상 행동: 비용 절감 vs 사용자 경험 트레이드오프 분석, 하이브리드 전략(실시간/배치 혼합) 검토
   - 근거: 단순 비용 절감만으로는 제품 가치 훼손 가능
