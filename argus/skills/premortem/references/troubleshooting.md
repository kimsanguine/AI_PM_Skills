# Troubleshooting — Premortem

## 1) Pre-mortem 도출이 빈약함 (5개 미만)

### 증상
- Pre-mortem exercise에서 "3개월 후 실패했다면?" 질문에 대해 2-3가지만 도출됨
- 팀이 "다 잘될 것 같은데..."라는 반응 보임

### 확인
- 질문 방식이 구체적이지 않은지 (개방형 질문 충분한가?)
- 팀의 심리 심리적 안전감이 부족한지 (부정적 답변을 꺼리는가?)
- 도메인 지식이 충분한지 (에이전트 특성, 의존성을 팀이 이해하는가?)

### 조치
1. 질문 방식 변경: "이 에이전트가 완전히 실패했다. 왜 실패했을까?" (강압적 표현)
2. 다양한 관점 포함: 기술자(모델 오류), 운영자(장애 감지), 사용자(가치 미전달) 등 역할별 분석
3. 도메인 워크숍: 에이전트 아키텍처, 외부 의존성, 비즈니스 목표 재검토 후 재진행

## 2) RPN 계산 표준화 부재 (평가자 간 편차 큼)

### 증상
- 같은 실패 모드의 RPN을 평가자마다 크게 다르게 책정
- 예: "환각 오정보" — Alice는 RPN 200, Bob은 RPN 80

### 확인
- Severity 정의가 팀마다 다른지 (1~10 척도의 기준 불명확)
- Probability 정의가 불명확한지 (과거 데이터 없이 추정인가?)
- Detection 척도를 제대로 이해하는지 (낮은 값 = 쉽게 감지)

### 조치
1. RPN 워크숍 진행
   ```
   Severity 척도 표준화:
   - 1-3: 소수 사용자 영향, 쉽게 우회 가능
   - 4-6: 10-25% 사용자 영향, 주요 기능 불편
   - 7-9: 25%+ 사용자 영향, 서비스 중단 수준
   - 10: 전체 서비스 중단, PII 유출 등 규제 이슈
   ```

2. 과거 데이터 검토: 이전 장애의 Probability 재평가로 경험 데이터 축적
3. 합의 프로세스: 평가자 간 편차 > 2 수준인 항목은 토론 후 합의값 도출

## 3) Prevention 액션이 미실행됨

### 증상
- FMEA 테이블은 완성했으나 Prevention 컬럼만 정성적 설명으로 끝남
- "Circuit breaker 추가", "모니터링 강화" 등 설명만 있고 실제 구현 계획 미흡

### 확인
- Owner/Deadline이 명시되었는지
- 액션 아이템의 구체성 (기술적 구현 방안이 명확한가?)
- 추적 체계 (누가, 언제, 어떻게 확인할 것인가?)

### 조치
1. 액션 아이템 구체화
   ```
   Before: "Circuit breaker 구현"
   After:
     - Owner: Alice (엔지니어)
     - Deadline: 2026-03-15
     - 구현: timeout 3초 설정 + 5회 연속 실패 시 break
     - 검증: 통합 테스트 + A/B 검증 (1주)
   ```

2. 월간 추적 미팅 설정 (고정 일정)
3. 미완료 액션 에스컬레이션 경로 수립

## 4) 새로운 위험 도입 미감지

### 증상
- Prevention 구현 중 새로운 실패 모드 발생
- 예: Circuit breaker 추가 → 복잡성 증가 → 신규 버그 발생
- 이를 감지하지 못하고 배포함

### 확인
- Prevention 구현 후 재premortem 일정이 있는가?
- 구현 과정에서 발견한 아키텍처 변화가 반영되었는가?
- 새로운 의존성이 생기지는 않았는가?

### 조치
1. Incremental premortem 프로세스
   ```
   배포 전 최종 premortem:
     - Prevention 구현된 상태에서 다시 "3개월 후 실패?"
     - 새로운 실패 모드 발생 여부 확인
   ```

2. 구현 리뷰 체크리스트에 "신규 위험" 항목 추가
3. 배포 후 2주 사후 모니터링으로 신규 실패 감지

## 5) Detection score 저평가 (감지 어려움을 과소평가)

### 증상
- Detection score = 3으로 설정 ("쉽게 감지")
- 배포 후 실제로는 조용한 실패(Silent Failure) 발생
- "왜 이걸 못 잡았나?" → Detection 점수가 잘못 평가됨

### 확인
- 모니터링 인프라가 실제로 이 실패를 감지할 수 있는가?
- 실패가 로그에 남는가? (아니면 조용히 잘못된 답을 주는가?)
- Alert 임계값이 현실적인가?

### 조치
1. Detection score 재평가
   ```
   Before: "환각" — Detection 3 (에러 로그로 감지)
   After: "환각" — Detection 8 (Human evaluation 필요, 자동 감지 불가)
   ```

2. 모니터링 인프라 강화
   - Automated evaluation (샘플 기반)
   - User feedback 수집 (Thumbs up/down)
   - 정기적 수동 감시

3. Alert 임계값 현실화
   - "Hallucination rate > 2%" 같이 측정 가능한 메트릭 기반

## 6) 위험 평가 노후화 (분기별 재검토 누락)

### 증상
- 3개월 전 premortem 결과를 그대로 신뢰
- 조직/기술 환경이 변해도 위험도 업데이트 안 함
- 예: 모델 업그레이드 후 hallucination 특성이 변했는데 RPN은 그대로

### 확인
- 마지막 premortem이 언제인가?
- 그 후 주요 변화가 있었는가? (모델 변경, 사용자 규모 2배, 새로운 API 연동)
- 현재 위험도 평가의 기준선이 유효한가?

### 조치
1. 재premortem 일정 엄격화
   ```
   기본: 분기별 (3개월마다)
   즉시: 다음 경우에 즉시 실행
     - 모델 업그레이드
     - 사용자 규모 2배 이상 증가
     - 새로운 의존성 추가
     - 이전 고위험 모드와 유사한 장애 발생
   ```

2. Premortem 결과 버전 관리 (이력 추적)
3. 변화 기록 → 위험도 영향 평가 (간단한 체크리스트)

## 7) 팀 정렬 부족 (FMEA 결과를 개선 계획으로 연결 안 함)

### 증상
- Premortem 완료 후 결과 보고서만 있고 실행 계획 없음
- "위험 RPN XXX입니다"로 끝남
- 팀이 어떤 액션을 해야 할지 불명확

### 확인
- Premortem 결과가 팀과 공유되었는가?
- 액션 아이템이 개인 목표(OKR)나 스프린트 계획에 반영되었는가?
- 리더가 "이게 정말 우선순위인가?" 확인했는가?

### 조치
1. Premortem 결과 공개 발표 + Q&A
2. 액션 아이템을 스프린트 백로그에 포함 (Deadline과 함께)
3. 주간 스탠업에서 진행 상황 확인
4. Owner별 체크인 (격주)
