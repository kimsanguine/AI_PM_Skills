# Test Cases — cohort 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "v1.1과 v2.0 에이전트를 배포한 후 시간이 지나면서 성능이 어떻게 변하는지 추적하고 싶다"
   - 이유: 버전별 장기 성능 추적은 cohort 분석의 핵심

2. "정확도가 주당 2%씩 떨어지고 있는데 언제부터 문제가 시작된 걸까?"
   - 이유: 성능 저하 패턴 조기 발견 및 시계열 추적

3. "내부 사용자와 외부 API 사용자의 에이전트 성능이 다르다"
   - 이유: 세그먼트 기반 코호트 분석

4. "TK를 점진적으로 추가하면서 에이전트가 개선되는지 추적해 줄 수 있나?"
   - 이유: TK 축적 수준별 코호트 분석

5. "사용자 재방문율이 버전마다 어떻게 다른지 알고 싶다"
   - 이유: 리텐션 곡선 비교

### Should NOT Trigger (5)

1. "지난 주 정확도는 92%이고 이번 주는 94%인데 통계적으로 의미 있는 차이인가?"
   - 올바른 라우팅: `agent-ab-test` (주 단위 변동은 통계 유의성 검증 필요)

2. "현재 에이전트의 정확도, 응답시간, 비용을 모니터링하려면?"
   - 올바른 라우팅: `kpi` (KPI 정의 및 대시보드)

3. "어제 정확도가 95%에서 45%로 급락했어, 뭔가 잘못됐나?"
   - 올바른 라우팅: `incident` (급락은 장애 대응)

4. "여러 버전의 코호트 성능을 하나의 지표로 통합해서 추적하고 싶다"
   - 올바른 라우팅: `north-star` (단일 핵심 지표 정의)

5. "코호트별로 비용이 다르게 나오는데 최적화할 방법이 있을까?"
   - 올바른 라우팅: `burn-rate` (비용 최적화)

## 2) Edge Cases

### 경계 사례 (4)

1. **코호트 샘플 부족**
   - 입력: "주간 v1.2 코호트가 50건만 있어서 신뢰도가 떨어진다"
   - 예상 행동: 추적 기간 연장 또는 월간 코호트로 재정의, 표본 통계 신뢰도 명시
   - 근거: 주간 100건 미만이면 통계적 신뢰도 낮음

2. **외부 변수 혼동**
   - 입력: "v2.0 정확도가 내려갔는데 그 기간 입력 데이터가 어려워졌을 수도 있다"
   - 예상 행동: 입력 데이터 분포 분석, 필요시 동일 입력으로 A/B 재테스트
   - 근거: 성능 변화 원인이 에이전트인지 데이터인지 분리 필수

3. **계절성 무시**
   - 입력: "월초는 사용량이 많고 월말은 적은데 성능도 다르다"
   - 예상 행동: 주기적 패턴 학습, 정상 범위(밴드) 설정, 범위 이탈 시만 알림
   - 근거: 정상 변동을 저하로 착각하면 불필요한 대응 발생

4. **리텐션 곡선 해석 오류**
   - 입력: "v1.1의 Week 1 리텐션이 35%인데 이게 좋은 건가 나쁜 건가?"
   - 예상 행동: 비교값(이전 버전, 경쟁사) 제시, 업계 표준 범위 명시, 상대 평가
   - 근거: 절대값 해석은 의미 없음, 비교 기준 필수
