# Test Cases — north-star 스킬

## 1) Trigger Tests

### Should Trigger (5)

1. "새로운 AI 에이전트 제품을 론칭하는데 팀 전체가 추구할 하나의 핵심 지표가 필요하다"
   - 이유: 팀 정렬을 위한 단일 North Star 정의가 필수

2. "우리 팀이 Accuracy만 높이려고 하고, 다른 팀은 Cost를 줄이려고 한다. 뭘 우선할까?"
   - 이유: 상충하는 KPI 중 우선순위 정렬

3. "정확도도 중요하고 비용도 중요한데 이 둘을 합쳐서 하나의 지표로 나타낼 수 없을까?"
   - 이유: 복합 지표 설계 필요

4. "분기 OKR을 에이전트 목표와 어떻게 연결할까?"
   - 이유: North Star를 통한 OKR 연결

5. "우리 에이전트의 성공을 측정하는 가장 좋은 단일 지표는 뭘까?"
   - 이유: North Star 후보 평가

### Should NOT Trigger (5)

1. "Accuracy, Latency, Cost 등 여러 지표를 모니터링하려면?"
   - 올바른 라우팅: `kpi` (KPI 대시보드 설계)

2. "지난 분기 정확도가 점점 내려갔다"
   - 올바른 라우팅: `cohort` (지표 추이 분석)

3. "이번 달 정확도가 예상과 다르게 나왔다. 왜일까?"
   - 올바른 라우팅: `incident` (급락) 또는 `cohort` (장기 추세)

4. "KPI 대시보드에 North Star를 최상위로 놓고 싶다"
   - 올바른 라우팅: `kpi` (대시보드 구조)

5. "A/B 테스트에서 주 지표로 뭘 써야 할까?"
   - 올바른 라우팅: `agent-ab-test` (테스트 설계)

## 2) Edge Cases

### 경계 사례 (4)

1. **North Star 달성 불가**
   - 입력: "3개월 연속 목표 미달. 100에서 65로 떨어졌다"
   - 예상 행동: 목표 현실성 재검토 (보수적으로 재설정), 분해 트리 분석 (막힌 드라이버 파악), 막힌 부분 해결
   - 근거: 달성 불가능한 목표는 팀 사기 저하

2. **Anti-metric 악화**
   - 입력: "Executions per week 늘렸는데 Accuracy 85%로 추락했다"
   - 예상 행동: North Star 재정의 (품질 가중치 상향) 또는 anti-metric threshold 강화
   - 근거: 안전 지표 악화는 North Star 개선을 무효화

3. **분해 트리 미작동**
   - 입력: "레버를 이렇게 당기면 North Star가 올라가야 하는데 안 올라간다"
   - 예상 행동: 드라이버-레버 연결 검증, 누락된 레버 추가, 인과 관계 재확인
   - 근거: 레버와 North Star 간 연결이 명확하지 않으면 팀 혼동

4. **팀 정렬 부족**
   - 입력: "일부 팀이 자신의 KPI만 최적화하고 North Star는 무시한다"
   - 예상 행동: 월간 North Star 리뷰 미팅 강제화, OKR 명시적 연결, 보상 체계 North Star 연동
   - 근거: 인센티브가 North Star와 맞지 않으면 정렬 불가능
