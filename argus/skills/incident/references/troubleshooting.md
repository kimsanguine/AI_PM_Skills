# Troubleshooting — incident 스킬

## 1) 심각도 분류 오류

**증상:**
- "SEV-3로 분류했는데 나중에 실제 영향이 훨씬 크다는 걸 알았다"
- 대응 인력이 부족했거나 너무 많았다
- "처음부터 심각도를 제대로 판단했으면 더 빨리 해결했을 텐데"

**확인:**
- 심각도 분류 시점의 정보 대비 실제 영향 확인
- 영향받은 사용자 수, 기간, 데이터 손상 여부 정량화
- 초기 분류의 근거 검토 (충분했나?)

**조치:**
1. 즉시 심각도 상향 조정:
   - 발견 후 정보가 증가하면 재분류 (SEV-3 → SEV-2)
   - 상향 조정 시 대응 인력 증원
2. 대응 강화:
   - SEV-2: 1시간 이내 1차 대응 (원래 4시간)
   - 추가 인력 동원, 우선순위 상향
3. 향후 개선:
   - 초기 분류가 보수적이도록 기준 수정 (모르면 상향)
   - "영향 추정 불명확 시 SEV-2로 분류" 규칙 추가

## 2) 원인 분석 불완전

**증상:**
- "API timeout이 원인이라는 걸 알았는데, 왜 타임아웃이 났는지 모른다"
- "다음 주에 같은 문제가 또 생길 것 같다"
- 5 Whys 분석이 1-2단계에서 멈춤

**확인:**
- 현재까지 파악된 원인이 직접 원인인가? (예: API timeout)
- 근본 원인은 무엇인가? (왜 timeout이 났나?)
- 더 깊은 원인 단계가 있나? (왜 그 설정이 되어 있었나?)

**조치:**
1. 5 Whys 재진행 (최소 3단계):
   ```
   Why 1: API timeout이 발생 → timeout 설정이 너무 짧음 (5초)
   Why 2: 왜 5초로 설정? → 기존 프롬프트 최적화 구버전 기준
   Why 3: 왜 업데이트 안 함? → 설정 변경 프로세스 없음
   Why 4: 왜 프로세스가 없나? → 설정 변경 시 자동 테스트 없음
   ```
2. 근본 원인 식별:
   - Level 3-4: 시스템/프로세스 레벨 원인
3. 재발 방지책 수립:
   - 자동화된 성능 테스트 추가
   - 타임아웃 설정 자동 조정 로직
4. TK 추출:
   - "TK-NNN: 성능 저하 후 구성 설정 자동 갱신 필수"

## 3) 블래스트 반경 과소 평가

**증상:**
- "100명 영향받았다고 보고" → 나중에 1,000명 발견
- "24시간 기간이라고 했는데" → 실제 48시간
- 영향도 재산정으로 보상 규모 변경 필요

**확인:**
- 영향받은 사용자 범위를 정확히 파악했나?
  - 로그 검증: 실제 영향받은 계정 수
  - 데이터 손상: 읽기만인가, 쓰기/삭제인가?
- 영향 기간 정확한가? (첫 발생 ~ 복구까지)
- 중복 계산이나 누락이 있나?

**조치:**
1. 영향도 재산정:
   - 로그 기반 정확한 사용자 수 계산
   - 영향받은 작업/데이터 정량화
   - 금전적 영향 재계산
2. 추가 공지:
   - "초기 보고보다 영향이 크다는 것을 발견했습니다"
   - 영향받은 추가 사용자/고객 특정해서 개별 공지
3. 보상 재계산:
   - 확대된 영향도 반영
   - 필요시 추가 보상 제시
4. TK 추출:
   - "블래스트 반경 산정은 로그 기반 정량, 추정치는 보수적으로"

## 4) 롤백 불가능 또는 복구 경로 불명확

**증상:**
- "이전 버전이 없다" 또는 "어떤 버전으로 돌려야 하는지 모른다"
- "복구 스크립트가 없어서 수동으로 해야 한다"
- 복구 시간 지연으로 영향 확산

**확인:**
- 버전 관리 시스템 상태 (git tag, release notes)
- 이전 버전 배포 가능 여부 (artifact 존재?)
- 자동 롤백 스크립트 존재 여부
- 데이터베이스 마이그레이션 (롤백 시 되돌릴 수 있나?)

**조치:**
1. 즉시 복구:
   - 롤백 불가능하면 hotfix 대안 검토
   - 임시 우회(workaround) 시행 (예: 기능 비활성화)
2. 복구 경로 재구성:
   - 이전 버전 배포 (tag 기반)
   - 데이터 상태 확인 (필요시 백업에서 복구)
   - 배포 후 smoke test 실행
3. 장기 개선:
   - 모든 배포 후 이전 버전 artifact 보존 (최소 1주)
   - 자동 롤백 스크립트 구현
   - 배포 전 rollback plan 검토 (QA)
4. 프로세스 강화:
   - "배포 시 이전 버전 롤백 경로 확인" 체크리스트 추가

## 5) 커뮤니케이션 지연

**증상:**
- "발견 후 12시간 만에 고객에게 공지했다"
- 고객이 SNS에 부정적 언급 → 신뢰도 급락
- "우리가 얼마나 빨리 대응하는지 몰랐다"

**확인:**
- 발견 시각과 첫 공지 시각 간 시간 계산
- 내부 공지 대비 외부 공지 지연
- 사전에 공지 텍스트가 준비되어 있었나?

**조치:**
1. 긴급 1차 공지 (1시간 이내):
   ```
   "현재 [서비스/기능]에 문제가 있음을 감지했습니다.
    즉시 대응 중이며 30분마다 업데이트하겠습니다."
   ```
   - 상황 인식(Awareness) 전달
   - "우리가 알고 있다" 신호
2. 정기 업데이트 (4시간 주기):
   - 진행 상황, 영향 범위, 예상 복구 시간
3. 최종 결과 보고:
   - 원인, 복구 방법, 향후 대책
4. 신뢰도 재확보:
   - 공식 사과문 + 투명한 postmortem 공개
   - 재발 방지 계획 공유
5. 프로세스 개선:
   - 사전 공지 템플릿 준비 (서비스 시작 때)
   - 공지 담당자/채널 사전 지정
   - 자동 알림 시스템 (발견 시 Slack/Email 자동 전송)
