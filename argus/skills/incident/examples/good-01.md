# Good Example — Incident

## 사용자 요청

"우리 고객 지원 에이전트가 지난 2시간 동안 정확도가 48%로 급락했어. 어떤 고객들이 잘못된 정보를 받았는지, 원인이 뭔지 빨리 파악해줄 수 있어? 그리고 대응 절차를 이끌어줘."

## 승인 이유

- 급락 감지 (정확도 48% — SEV-2 수준)
- 영향도 파악 필요 (몇 명의 고객?)
- 즉시 대응 프로세스 요구
- 원인 분석 및 복구 전략 필요

## 예상 처리

1. 심각도 분류: SEV-2 (핵심 기능 장애, 다수 유저 영향)
2. Immediate response: 에이전트 기능 비활성화, 영향도 파악
3. 사용자 공지: 1차 (30분 내), 이후 4시간 주기 업데이트
4. 원인 분석: 5 Whys (프롬프트? 모델? 데이터?)
5. 복구: Rollback 또는 Hotfix
6. Postmortem: Timeline, 근본 원인, 액션 아이템, TK 추출

## Preflight 기반 인시던트 예방 Good Example

### 배경: Rate Limit Preflight의 실전 가치

이 사례는 **T08 Rate Limit 초과**를 Preflight 체계로 예방한 실제 시나리오입니다.

### 타임라인

**14:32 — Rate Limit 80% 경고 (Automatic Preflight)**
```
Alert: Customer Support Agent API usage at 80% of daily quota
Event: 사용자 요청 20,000개/일 중 16,000개 소비 (평소 8,000/일)
Action: 자동 로그 기록 + 대시보드 알림 (운영팀 실시간 가시성)
```

**14:45 — 운영팀 인지 및 초기 분석**
```
On-call Engineer: "어? 오늘 트래픽이 갑자기 2배네?"
분석: 신규 고객 온보딩 캠페인으로 예상치 못한 트래픽 급증
```

**15:00 — Rate Limit 95% 도달 (자동 큐잉 시작)**
```
Preflight Action: quota_budget_valid 트리거
메커니즘:
  - 기존 요청: 즉시 처리 (유지)
  - 신규 요청: 큐에 삽입 (FIFO 순서)
  - 예상 처리 시간: 30분 내 모두 처리 가능 (탄성적 부하 흡수)

고객 영향: 30분 지연, 하지만 에러나 실패 없음 (완전 차단 회피)
```

**15:30 — 자동 복구 및 에스컬레이션**
```
System Action:
  1. 큐에 남은 요청 모두 처리 완료
  2. 사용량 다시 60%로 감소
  3. 운영팀에 "자동 복구 완료" 보고

운영팀 Action:
  - "우리 Rate Limit이 좋은 타이밍에 고객을 보호했네"
  - 향후 대비: 신규 고객 온보딩 전에 미리 limit 상향 신청
  - 예방: "고객별 quota 예측 대시보드" 추가 구축 결정
```

**16:00 — Postmortem 기록**

| 항목 | 내용 |
|---|---|
| **인시던트** | Rate Limit 자동 유발 (하지만 완전한 중단 없음) |
| **심각도** | SEV-3 (부분 저하, 고객 영향 최소화) |
| **근본 원인** | 예상치 못한 신규 고객 온보딩 트래픽 증가 |
| **대응** | Preflight 80%/95% gate 자동 작동 → 큐잉 → 30분 지연 (장애 아님) |
| **결과** | 서비스 무중단, 고객 신뢰도 유지 |
| **배운 점 (TK 추출)** | "Rate Limit Preflight이 없었다면, 이 상황에서 500 에러 대량 발생 → SEV-1 사건으로 에스컬레이션" |
| **액션** | 1. 신규 고객 온보딩 전 rate limit 미리 검토 프로세스 추가 2. 고객별 quota 예측 대시보드 구축 |

### 핵심 교훈

**이 사례가 Good Example인 이유:**

1. **Preflight이 있었다면 → 자동 차단 + 안정성 유지**
   - Rate Limit 80% 경고 → 운영팀이 미리 알림
   - 95% 자동 큐잉 → 고객에게 30분 지연(나쁨)이지, 500 에러(최악)는 아님

2. **만약 Preflight이 없었다면 → SEV-2 사건으로 악화**
   - 16,000개 요청이 모두 API 한계 도달 → 대량 실패
   - "우리 에이전트가 갑자기 안 됨" → 소셜미디어 불만 폭주
   - 야간 호출(oncall) 활성화 → 엔지니어 수동 개입
   - 비용 급증 + 신뢰도 급락

3. **사전 차단의 가치 입증**
   - 예방: Rate Limit Preflight (구현 비용 낮음)
   - vs. 사후: 대량 에러 대응 (비용/시간 매우 높음)
   - **결론**: Preflight은 "보험료"이자 "무한 배당"
