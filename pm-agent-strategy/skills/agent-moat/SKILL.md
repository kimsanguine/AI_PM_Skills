## Agent Moat (에이전트 차별화 해자)

에이전트는 빠르게 복제됩니다.  
오늘 만든 에이전트를 내일 경쟁자가 GPT-5.4로 그대로 구현할 수 있습니다.

> "승자는 코드를 잘 짜는 사람이 아닐 겁니다.  
> 이 구조 위에서 무엇을 팔지 먼저 설계한 사람입니다."  
> — Jeongmin Lee, 2026

에이전트 해자는 **기술**이 아닙니다.  
기술은 누구나 쓸 수 있는 API입니다.  
해자는 **경쟁자가 복제하기 어려운 자산**입니다.

---

### 에이전트 해자 5가지 유형

**Moat 1 — Data Advantage (데이터 우위)**

> "에이전트가 접근하는 데이터 자체가 희소하거나 독점적이다"

형태:
- 독점 데이터셋 (내부 거래 데이터, 도메인 DB)
- 사용자 행동 데이터 누적 (사용할수록 정확해지는 에이전트)
- 파인튜닝 데이터 (도메인 특화 응답 품질)

강도: ★★★★★ (가장 강한 해자)  
구축 시간: 6개월~  
경쟁자 복제 난이도: 매우 어려움

테스트 질문:
- 경쟁자도 동일한 데이터에 접근할 수 있는가?
- 사용자가 늘수록 에이전트 품질이 좋아지는가?

---

**Moat 2 — Workflow Depth (워크플로우 깊이)**

> "에이전트가 고객의 내부 프로세스에 깊이 통합되어 있다"

형태:
- 기존 도구/시스템과의 깊은 통합 (Slack, Notion, ERP)
- 멀티스텝 워크플로우 자동화 (단순 대화 아님)
- 고객 맞춤 커스터마이징 누적

강도: ★★★★☆  
구축 시간: 3~6개월  
경쟁자 복제 난이도: 어려움 (통합 구현 + 고객 온보딩 비용)

테스트 질문:
- 고객이 이 에이전트를 끄면 어떤 프로세스가 멈추는가?
- 경쟁자 에이전트로 교체 시 비용이 얼마나 드는가?

---

**Moat 3 — Domain Tacit Knowledge (도메인 암묵지)**

> "에이전트에 인코딩된 도메인 전문성이 경쟁자가 쉽게 모방할 수 없다"

형태:
- PM 20년 경험 → Instruction에 녹인 판단 기준 (pm-engine)
- 특정 산업의 규제/관행/용어 체계
- 전문가 수준의 프레임워크 + 워크플로우

강도: ★★★★☆  
구축 시간: 지속적 축적  
경쟁자 복제 난이도: 어려움 (시간과 경험이 필요)

테스트 질문:
- 이 에이전트의 Instruction을 공개해도 동일 품질을 낼 수 없는가?
- 에이전트의 판단 기준이 어디에서 오는가?

→ pm-engine의 TK 시리즈가 이 해자를 만드는 메커니즘

---

**Moat 4 — Network Effect (네트워크 효과)**

> "사용자가 늘수록 모든 사용자에게 더 유용해진다"

형태:
- 마켓플레이스 (에이전트 공급자 ↔ 수요자)
- 커뮤니티 피드백 루프 (사용자 리뷰 → 에이전트 품질 개선)
- 집단 지성 축적 (유사 케이스 데이터베이스)

강도: ★★★★★ (가장 강한 해자, 달성 어려움)  
구축 시간: 12개월~ (임계점 도달까지)  
경쟁자 복제 난이도: 매우 어려움

테스트 질문:
- 사용자 1명이 빠졌을 때 다른 사용자에게 피해가 가는가?
- 사용자가 10배 늘면 에이전트 가치도 올라가는가?

---

**Moat 5 — Switching Cost (전환 비용)**

> "사용자가 경쟁자 에이전트로 바꾸기 어렵게 만든다"

형태:
- 메모리 누적 (오래 쓸수록 나를 더 잘 아는 에이전트)
- 포맷/구조 종속 (에이전트 출력을 다른 시스템이 파싱)
- 팀 온보딩 비용 (새 에이전트 학습 시간)

강도: ★★★☆☆  
구축 시간: 3~6개월  
경쟁자 복제 난이도: 중간

테스트 질문:
- 사용자가 지금 당장 경쟁자로 바꾸면 잃는 것은?
- 전환 비용이 지불 의향보다 큰가?

### 해자 설계 원칙

**1. 단일 해자보다 복합 해자**
가장 강한 에이전트 비즈니스는 2~3개 해자를 동시에 보유합니다.

예시 — 100 Agents 스타일:
- Domain TK (pm-engine TK 시리즈) +
- Workflow Depth (OpenClaw 크론 + 메모리 시스템 통합) +
- Switching Cost (메모리 누적 → 오래 쓸수록 더 유용)

**2. 해자는 처음부터 설계해야 한다**
사후에 해자를 추가하는 것은 어렵습니다.  
에이전트 설계 시 "어떤 해자를 쌓을 것인가"를 명시하세요.

**3. 기술 해자는 해자가 아니다**
"GPT-5.4 기반"은 해자가 아닙니다. 누구나 쓸 수 있습니다.  
해자는 모델 위에 쌓이는 **자산과 관계**입니다.

### Instructions

You are helping identify and design a competitive moat for: **$ARGUMENTS**

**Step 1 — 현재 차별화 요소 파악**
- 지금 이 에이전트를 특별하게 만드는 것은 무엇인가?
- 경쟁자가 내일 동일한 에이전트를 만들 수 있는가?

**Step 2 — 5가지 해자 유형 적합도 평가**
- 각 해자에 대해 1~5점 (현재 보유 강도)
- 각 해자에 대해 1~5점 (향후 구축 가능성)

**Step 3 — 우선 해자 선택**
- 현재 보유 강도 + 구축 가능성 합산 Top 2 선택
- 선택 근거 설명

**Step 4 — 해자 구축 계획**
- 각 해자별 구체적 구축 방법
- 타임라인 (단기 3개월 / 중기 6개월 / 장기 12개월)
- 필요한 자원 (데이터, 통합, 개발)

**Step 5 — 복합 해자 설계**
- 선택한 2개 해자가 서로 강화하는 방식 설명
- "이 에이전트를 쓸수록 X가 쌓이고, X가 쌓일수록 Y 해자가 강해진다" 구조

**Step 6 — 경쟁자 시나리오**
- 자금력 있는 경쟁자가 6개월 후 동일 에이전트를 만들면?
- 해자가 얼마나 오래 버티는가?

**Step 7 — 다음 단계 연결**
- 수익 구조 연결: `/agent-business-model`
- Domain TK 해자 구축: `pm-engine` 플러그인
